Shared Secret

Alright, let’s break shared secrets down properly—from intuition → math → real-world systems—so it really sticks.

1. What a “shared secret” actually is

A shared secret is a piece of information known only to two (or more) parties, used to prove identity or protect data.

Think of it like:

  • A password both sides know
  • A private key both sides derive
  • A token agreed on earlier

The key idea:

If you know the secret, you must be legitimate.

2. The core problem shared secrets solve

When two systems talk over the internet:

  • The network is public
  • Anyone can listen
  • Anyone can replay messages

So the challenge is:

“How do I prove I’m really me, without sending my secret?”

That’s where shared secrets + cryptography come in.

3. Naive approach (what NOT to do ❌)

❌ Sending the secret directly


Client → Server: password=supersecret123

Problems:

  • Anyone sniffing traffic steals it
  • Replay attacks
  • Permanent compromise

4. Correct approach: Proving knowledge of the secret

Instead of sending the secret:

You send a cryptographic proof that you know it

This is usually done using:

  • Hash functions
  • HMAC
  • Signatures
  • Key derivation

5. Shared secret with hashing (basic concept)

Setup

Both sides know:


shared_secret = "abc123"

Authentication

  1. Server sends a challenge (random value)
  2. Client hashes: 
    
hash(shared_secret + challenge)
  3. Server computes the same hash
  4. If they match → authenticated ✅

Why this works:

  • The secret is never sent
  • Challenge prevents replay attacks

6. HMAC (real-world standard)

Most real systems use HMAC (Hash-based Message Authentication Code).

Formula


HMAC(secret, message)

Example


secret = "abc123"
message = "timestamp=1700000000"
signature = HMAC(secret, message)

Client sends:


{
    "message": "timestamp=1700000000",
    "signature": "a94f..."
}

Server:

  • Recomputes HMAC using the same secret
  • Compares signatures
  • If equal → request is authentic

7. Why HMAC is secure

HMAC gives you:

  • Authentication – proves who sent it
  • Integrity – message wasn’t modified
  • Replay protection – when combined with timestamps / nonces

Attackers:

  • Can see the message
  • Can see the signature
  • Cannot forge a new valid signature without the secret

8. Shared secret lifecycle (VERY important)

1️⃣ Generation

  • Random
  • High entropy (never human passwords)

Example:


f7A9!xQ29@LkZp1M...

2️⃣ Distribution

The hardest part.

Common methods:

  • Stored securely
  • Never logged
  • Often encrypted at rest

4️⃣ Rotation

  • Secrets expire
  • New secrets issued regularly

9. Diffie–Hellman: shared secret without sharing it 🤯

This is the magic part.

Goal

Create a shared secret:

  • Over a public network
  • Without ever transmitting it

High-level idea

  1. Both agree on public math parameters
  2. Each side picks a private number
  3. They exchange derived public values
  4. Both independently compute the same secret

Even if an attacker sees everything:

  • They cannot compute the secret

This is used in:

  • TLS
  • HTTPS
  • Secure messaging

10. Shared secret vs public/private keys

Feature Shared Secret Public/Private Key
Keys per pair One Two
Distribution Hard Easy
Scalability Poor Excellent
Performance Fast Slower
Used in HMAC, JWT HS256 RSA, ECDSA, JWT RS256

In practice:

🔥 Public-key crypto establishes trust

🔥 Shared secrets are used afterward for speed

11. Real-world examples you’ve probably used

🔹 JWT HS256

  • Same secret signs and verifies token
  • Both issuer and verifier must know secret

🔹 API keys + HMAC

  • AWS, Stripe-style authentication

🔹 Webhooks

  • Provider signs payload with shared secret
  • Receiver verifies signature

🔹 TOTP (Google Authenticator)

  • Shared secret + time → one-time code

12. Common mistakes (please don’t 😅)

❌ Reusing secrets across environments

❌ Storing secrets in plaintext

❌ Long-lived secrets with no rotation

❌ Using human passwords as secrets

❌ Logging secrets accidentally

13. When to use shared secrets (and when not to)

✅ Use when:

  • Two systems already trust each other
  • Server-to-server communication
  • Performance matters

❌ Avoid when:

  • Many clients
  • Mobile apps (secrets can be extracted)
  • Zero-trust environments

14. Mental model to remember

Think of a shared secret like:

A secret handshake where both sides know the moves, but never show the instructions.
