What Are Public Keys?

A public key is one half of a public-key (asymmetric) cryptography pair:

  • 🔐 Private Key – kept secret by the owner
  • 🌍 Public Key – shared openly with anyone

They are mathematically linked.

What one key encrypts, only the other can decrypt.

What one key signs, the other can verify.

There are two major uses:

1️⃣ Encryption

  • Public key encrypts data
  • Private key decrypts data

2️⃣ Digital Signatures (what Google & Apple mainly use for auth)

  • Private key signs data
  • Public key verifies the signature

Authentication systems mostly use digital signatures, not encryption.

How Google Uses Public Keys (Detailed Flow)

When using Google Sign-In with Firebase or OAuth, Google uses RSA-based digital signatures with JWT tokens.

Relevant systems:

  • Google
  • Firebase
  • Google Identity Platform

Step-by-Step: Google Sign-In Flow

🔹 Step 1: User Authenticates

User logs in with Google in your app.

Google authenticates the user and generates:

  • ID Token (JWT)
  • Access Token

We focus on the ID Token.

🔹 Step 2: Google Signs the ID Token

The ID token is a JWT (JSON Web Token).

Structure:


HEADER.PAYLOAD.SIGNATURE

Example header:


{
    "alg": "RS256",
    "kid": "abc123",
    "typ": "JWT"
}
  • RS256 = RSA + SHA256
  • kid = Key ID (which public key to use)

Google signs the token using its private key.

Only Google has that private key.

What Are Public Keys?

A public key is one half of a public–private key pair used in asymmetric cryptography.

  • 🔐 Private Key → kept secret, used to sign or decrypt
  • 🔓 Public Key → shared publicly, used to verify signatures or encrypt

They are mathematically related. Data signed with the private key can only be verified with the corresponding public key.

Core Concept (Simple Example)

  1. Google signs a token using its private key.
  2. Your backend verifies that token using Google’s public key.
  3. If verification succeeds → you know:
    • The token was really issued by Google.
    • The token was not tampered with.

This is how trust works in modern authentication systems.

How Google Uses Public Keys (Google Auth)

When using Google Sign-In or Firebase Authentication, the flow relies on JWT (JSON Web Tokens) signed with asymmetric cryptography (usually RS256).

Step-by-Step Flow

1️⃣ User Signs In

User logs in via Google Sign-In in your React Native app.

Google issues:

  • id_token (JWT)
  • access_token

2️⃣ The ID Token Is Signed

The ID Token:

  • Is a JWT
  • Is signed by Google using Google's private key
  • Contains:
    • sub (user ID)
    • email
    • aud (your app's client ID)
    • iss (issuer)
    • exp (expiration)

Example header:


{
    "alg": "RS256",
    "kid": "abc123"
}
  • RS256 → RSA with SHA-256
  • kid → key ID (tells you which public key to use)

3️⃣ Your Backend Verifies It

Your AWS Lambda backend:

  1. Reads the token header
  2. Extracts kid
  3. Fetches Google’s public keys from: 
    
https://www.googleapis.com/oauth2/v3/certs
  4. Finds matching public key
  5. Verifies signature

If signature matches → token is authentic.

Why This Is Secure

  1. Only Google has the private key
  2. Anyone can verify using the public key
  3. Even if someone copies the token, they can’t forge a new valid one
