Refresh Token

A refresh token is part of an authentication system (usually OAuth 2.0 / JWT-based) that lets a client obtain a new access token without forcing the user to log in again.

🔑 Concept (simple view)

  1. Access Token
    • Short-lived (e.g., 15 minutes)
    • Used to access APIs
    • Sent on every request
  2. Refresh Token
    • Long-lived (e.g., days/weeks)
    • Used only to get a new access token
    • Stored securely (NOT sent on every request)

🔁 Flow

  1. User logs in
    • Access token (short-lived)
    • Refresh token (long-lived)
  3. Access token expires
  4. Client sends refresh token to backend
  5. Backend verifies it and issues a new access token

🧠 Why use refresh tokens?

  • Improves security (short-lived access tokens)
  • Improves UX (no frequent logins)
  • Enables session control (you can revoke refresh tokens)

⚙️ Implementing in AWS Serverless (Node.js)

Typical stack:

  • Amazon Web Services (AWS)
  • AWS Lambda
  • Amazon API Gateway
  • Amazon DynamoDB
  • Amazon Cognito (optional but recommended)

🧩 Option 1 (Recommended): Use Cognito (Built-in Refresh Tokens)

If you use Amazon Cognito, it handles refresh tokens automatically.

Flow:

  • User logs in → Cognito returns:
    • Access token
    • ID token
    • Refresh token
  • Use /oauth2/token endpoint to refresh tokens

Node.js example:


const axios = require("axios");

async function refreshToken(refreshToken) {
  const response = await axios.post(
    "https://your-domain.auth.region.amazoncognito.com/oauth2/token",
    new URLSearchParams({
      grant_type: "refresh_token",
      client_id: process.env.CLIENT_ID,
      refresh_token: refreshToken,
    }),
    {
      headers: {
        "Content-Type": "application/x-www-form-urlencoded",
      },
    }
  );

  return response.data; // new access token
}

👉 Best for production apps (less custom security work)

🧩 Option 2: Custom JWT + Refresh Token (Manual Implementation)

If you're not using Cognito:

🗄️ Architecture

  • Access token → JWT (signed)
  • Refresh token → random string (stored in DB like DynamoDB)

1. Generate Tokens (Login Lambda)


const jwt = require("jsonwebtoken");
const crypto = require("crypto");

function generateTokens(userId) {
  const accessToken = jwt.sign(
    { userId },
    process.env.ACCESS_SECRET,
    { expiresIn: "15m" }
  );

  const refreshToken = crypto.randomBytes(64).toString("hex");

  return { accessToken, refreshToken };
}

2. Store Refresh Token (DynamoDB)

Store:

  • userId
  • refreshToken
  • expiration
  • device info (optional)

Example item:


{
    "userId": "123",
    "refreshToken": "abc123...",
    "expiresAt": 1710000000
}

3. Refresh Endpoint (Lambda)


const jwt = require("jsonwebtoken");

exports.handler = async (event) => {
  const { refreshToken } = JSON.parse(event.body);

  // 1. Check DB
  const record = await getTokenFromDB(refreshToken);

  if (!record) {
    return { statusCode: 401, body: "Invalid refresh token" };
  }

  // 2. Issue new access token
  const newAccessToken = jwt.sign(
    { userId: record.userId },
    process.env.ACCESS_SECRET,
    { expiresIn: "15m" }
  );

  return {
    statusCode: 200,
    body: JSON.stringify({ accessToken: newAccessToken }),
  };
};

4. Protect API (Middleware in Lambda)


function verifyAccessToken(token) {
    try {
        return jwt.verify(token, process.env.ACCESS_SECRET);
    } catch (err) {
        return null;
    }
}

🔐 Security Best Practices

✅ DO:

  • Store refresh tokens in HTTP-only cookies
  • Rotate refresh tokens (issue a new one each refresh)
  • Set expiration (e.g., 7–30 days)
  • Use HTTPS only
  • Store hashed refresh tokens (like passwords)

❌ DON'T:

  • Store refresh tokens in localStorage (XSS risk)
  • Make refresh tokens never expire
  • Reuse refresh tokens indefinitely

🔄 Token Rotation (Important)

Instead of reusing:

  1. Client sends refresh token
  2. Server:
    • Deletes old refresh token
    • Issues new refresh + access token

👉 Prevents token theft reuse

🧱 Serverless Flow Summary

  1. Login → Lambda:
    • Generate tokens
    • Store refresh token in Amazon DynamoDB
  2. API Request:
    • Verify access token in Lambda
  3. Expired:
    • Call /refresh endpoint via Amazon API Gateway
  4. Refresh Lambda:
    • Validate refresh token
    • Issue new access token

🧭 When to Use What

  • Use Amazon Cognito if:
    • You want managed auth
    • You don’t want to handle security complexity
  • Use custom solution if:
    • You need full control
    • You already have your own auth system
