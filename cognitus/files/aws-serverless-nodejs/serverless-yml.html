AWS serverless function (Lambda-based) vs traditional Node.js Express app

The key difference between an AWS serverless function (Lambda-based) and a traditional Node.js Express app is how and where your code runs, and how it’s managed.

1. Execution Model

🟣 AWS Serverless Function (AWS Lambda)

  • Code runs as single-purpose functions
  • Triggered by events (HTTP request, queue, cron, etc.)
  • Example: one function per route or feature

👉 Think: “Run this function only when needed”

🔵 Node.js Express App (Traditional Server)

  • Runs as a continuous server process
  • Handles many routes in one application
  • Example: /users, /orders, /login all inside one app

👉 Think: “Always-on server handling requests anytime”

2. Infrastructure Management

AWS Lambda

  • No server to manage (serverless)
  • AWS automatically handles scaling, uptime, and resources
  • You just deploy functions

Express App

  • You manage the server (EC2, VPS, Docker, etc.)
  • You handle scaling, load balancing, uptime

3. Scaling

Lambda

  • Auto-scales instantly per request
  • Can handle 1 → 10,000 requests without manual setup

Express

  • You must configure scaling (load balancer, clusters, autoscaling groups)

4. Cost Model

Lambda

  • Pay per request + execution time
  • Cheap for low/medium traffic
  • Can become expensive at very high traffic

Express

  • Pay for server uptime (even if idle)
  • More cost-effective for constant heavy traffic

5. Performance

Lambda

  • Has cold starts (initial delay when function wakes up)
  • Best for short, stateless operations

Express

  • Always running → no cold start
  • Better for long-lived connections (WebSockets, streaming)

6. Structure

Lambda example


exports.handler = async (event) => {
    return {
        statusCode: 200,
        body: JSON.stringify({ message: "Hello from Lambda" })
    };
};

Each function is isolated.

Express example


const express = require('express');
const app = express();

app.get('/hello', (req, res) => {
  res.json({ message: "Hello from Express" });
});

app.listen(3000);

All routes live in one app.

7. Deployment Style

Lambda (Serverless)

  • Deployed via Serverless Framework / AWS SAM / CDK
  • Often mapped via API Gateway

Express

  • Deploy whole app to a server (EC2, VPS, container)

8. When to Use

Use AWS Serverless (Lambda) when:

  • Microservices
  • Event-driven systems
  • Low to medium traffic APIs
  • Pay-per-use optimization

Use Express when:

  • Full control needed
  • Real-time apps (WebSockets, streaming)
  • High and constant traffic
  • Complex middleware-heavy APIs

Simple Analogy

  • Lambda = Uber ride → you only pay when you ride
  • Express server = owning a car → always available, but you maintain it
