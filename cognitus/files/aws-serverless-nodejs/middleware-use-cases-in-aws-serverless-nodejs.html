Middleware Use Cases in AWS Serverless (Node.js)

In an AWS serverless Node.js setup (typically using Amazon Web Services with AWS Lambda and Amazon API Gateway), middleware works a bit differently compared to traditional Express.js apps—but the concept is the same: intercept requests before/after your handler to apply shared logic.

Since Lambda doesn’t natively support Express middleware chains, developers usually use:

  • middleware frameworks like Middy
  • or custom wrapper functions

🔧 Common Middleware Use Cases in AWS Serverless (Node.js)

1. Authentication / Authorization (JWT, Cognito)

Validate tokens before allowing access to protected routes.

Use case:

  • Verify JWT from headers
  • Integrate with Amazon Cognito

const authMiddleware = async (event) => {
    const token = event.headers.Authorization;
    
    if (!token) throw new Error("Unauthorized");
    
    // verify token logic here
    return { userId: "123" };
};

👉 Applied before Lambda handler to protect endpoints like /contacts.

2. Request Validation

Ensure incoming data is valid before processing.

Use case:

  • Validate body for POST/PUT (e.g., contact creation)
  • Prevent bad data reaching MongoDB

const validateMiddleware = (event) => {
    const body = JSON.parse(event.body);
    
    if (!body.email) {
        throw new Error("Email is required");
    }
};

3. Error Handling Middleware

Catch and standardize errors across all Lambdas.


const errorMiddleware = (handler) => async (event) => {
    try {
        return await handler(event);
    } catch (err) {
        return {
            statusCode: 500,
            body: JSON.stringify({ message: err.message }),
        };
    }
};

👉 Prevents inconsistent error responses.

4. Logging & Monitoring

Track requests, responses, and performance.

Use case:

  • Log request payloads
  • Send logs to Amazon CloudWatch

const loggerMiddleware = (handler) => async (event) => {
    console.log("Request:", event);
    const result = await handler(event);
    console.log("Response:", result);
    return result;
};

5. CORS Handling

Add headers to allow frontend apps (React Native / Web) to access API.


const corsMiddleware = (response) => {
    return {
        ...response,
        headers: {
        ...response.headers,
        "Access-Control-Allow-Origin": "*",
        },
    };
};

6. Database Connection Reuse (MongoDB Optimization)

Important for serverless performance.

Use case:

  • Reuse MongoDB connection instead of reconnecting every invocation

let cachedDb = null;

const dbMiddleware = async () => {
  if (!cachedDb) {
    cachedDb = await connectToMongo();
  }
  return cachedDb;
};

👉 Critical when using MongoDB with Lambda (like your setup).

7. Input Transformation / Parsing

Normalize incoming requests.

Use case:

  • Parse JSON body
  • Normalize headers/query params

8. Rate Limiting / Throttling

Prevent abuse of API endpoints.

Use case:

  • Limit requests per user/IP
  • Protect login endpoints

9. Response Formatting

Standardize API responses across all endpoints.


const responseMiddleware = (data) => ({
    statusCode: 200,
    body: JSON.stringify({
        success: true,
        data,
    }),
});

🧠 Example: Middleware Composition (Custom Wrapper)


const withMiddleware = (handler) => async (event) => {
    try {
        await validateMiddleware(event);
        const user = await authMiddleware(event);
    
        const result = await handler(event, user);
    
        return corsMiddleware({
        statusCode: 200,
        body: JSON.stringify(result),
        });
    } catch (err) {
        return {
        statusCode: 400,
        body: JSON.stringify({ error: err.message }),
        };
    }
};

⚡ Using Middy (Recommended)


const middy = require('@middy/core');
const jsonBodyParser = require('@middy/http-json-body-parser');

const handler = async (event) => {
  return { statusCode: 200, body: JSON.stringify(event.body) };
};

exports.handler = middy(handler)
  .use(jsonBodyParser());

🔑 Key Insight

Middleware in serverless is not built-in like Express—but it’s even more important because:

  • Functions are stateless
  • You must handle cross-cutting concerns manually
  • Performance depends on smart reuse (DB, auth, etc.)
