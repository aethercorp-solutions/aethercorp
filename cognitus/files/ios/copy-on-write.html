Swift’s copy-on-write

Swift’s copy-on-write (COW) is a performance optimization layered on top of value semantics. It lets value types behave like independent copies while avoiding unnecessary deep copies until mutation actually happens.

Core idea

When you copy a value type like Array, Dictionary, or String, Swift does not immediately duplicate the underlying data. Instead, both copies initially share the same heap-allocated storage. Only when one of them is modified does Swift make a real copy.

How it works internally

Most standard library value types use this pattern:

1. Value type (struct) wraps a reference type

A Swift struct (e.g., Array) contains a reference to a heap object that holds the actual storage.

Conceptually:


struct Array<Element> {
    private var storage: _ArrayStorage<Element>
}

_ArrayStorage is a class (reference type) that holds:

  • buffer pointer
  • capacity
  • count
  • elements

So copying the struct is cheap: it just copies a pointer.

2. Reference counting shares storage

The underlying storage is reference-counted (ARC). When you do:


var a = [1, 2, 3]
var b = a

Both a and b point to the same storage object, and its retain count increases.

No element copying happens here.

3. Mutation triggers uniqueness check

Before any mutation, Swift checks whether the storage is uniquely referenced.

This is done using:


isKnownUniquelyReferenced(&storage)

So when you write:


b.append(4)

Swift does something like:


if !isKnownUniquelyReferenced(&storage) {
    storage = storage.copy()
}

Then b gets its own independent storage, and mutation proceeds safely.

Why this is safe

COW relies on two guarantees:

1. Value semantics at the language level

Even though storage is shared, the observable behavior is as if each variable is independent.

2. Uniqueness check before mutation

As long as mutation triggers a copy when needed, no two variables can accidentally mutate the same storage.

Where Swift uses COW

Common standard library types:

  • Array
  • Dictionary
  • Set
  • String
  • Data

All of these:

  • Are structs
  • Wrap reference-counted storage
  • Use COW on mutation

Simple mental model

Think of it like this:

  • Copying = sharing a pointer
  • Reading = free (no copy)
  • Writing = “if shared, clone first”

Minimal illustrative example

A simplified version of what Swift does:


final class Box<T> {
    var value: T
    init(_ value: T) { self.value = value }
}

struct MyValue<T> {
    private var box: Box

    mutating func mutate(_ newValue: T) {
        if !isKnownUniquelyReferenced(&box) {
            box = Box(box.value) // copy storage
        }
        box.value = newValue
    }
}

Key benefits

  • Efficient copies of large collections
  • Predictable value semantics
  • No accidental shared mutation
  • Good balance between performance and safety
