UIViewController

In UIViewController, you should focus on code that manages a single screen’s lifecycle, UI behavior, and user interaction—not everything in your app. Think of it as the “orchestrator” for one view.

Here’s what typically belongs there:

1. Lifecycle Methods

These control what happens as the screen appears/disappears.

  • viewDidLoad() → initial setup (runs once)
  • viewWillAppear() → update UI before it shows
  • viewDidAppear() → start animations, tracking
  • viewWillDisappear() → cleanup before leaving

override func viewDidLoad() {
    super.viewDidLoad()
    setupUI()
}

2. UI Setup & Configuration

Create and configure views, constraints, styles.


func setupUI() {
    view.backgroundColor = .white
    titleLabel.text = "Welcome"
}

3. User Interaction Handling

Respond to taps, gestures, inputs.


@IBAction func buttonTapped(_ sender: UIButton) {
    print("Button tapped")
}

4. Navigation Logic

Handle transitions to other screens.


func goToNextScreen() {
    let vc = NextViewController()
    navigationController?.pushViewController(vc, animated: true)
}

5. Binding / Updating Data to UI

Take data and reflect it in views.


func updateUI(with user: User) {
    nameLabel.text = user.name
}

6. Light State Management

Keep small pieces of screen-specific state.


var isLoading = false

What should NOT be in UIViewController ❌

If everything ends up in your controller, it quickly becomes unmanageable (“Massive View Controller” problem). Avoid:

  • Heavy business logic → move to services or models
  • Networking code → use separate API/service layer
  • Complex data processing → use view models (MVVM)
  • Reusable UI components → separate into custom views

A Good Rule of Thumb

Your UIViewController should:

  • Coordinate, not compute
  • React, not decide everything
  • Display, not own all logic

Where should the code for networking API calls be placed?

Short answer: not inside your UIViewController. Put networking code in a separate layer dedicated to fetching data.

Where it should go instead

1. Service / API Layer (most common)

Create a class or struct responsible only for API calls.


class APIService {
    static let shared = APIService()

    func fetchUsers(completion: @escaping (Result<[User], Error>) -> Void) {
        let url = URL(string: "https://api.example.com/users")!

        URLSession.shared.dataTask(with: url) { data, _, error in
            if let error = error {
                completion(.failure(error))
                return
            }

            let users = try? JSONDecoder().decode([User].self, from: data!)
            completion(.success(users ?? []))
        }.resume()
    }
}

👉 This layer uses URLSession and knows nothing about UI.

2. ViewModel (if using MVVM)

If you’re using MVVM, the ViewModel calls the service and prepares data for the UI.


class UserViewModel {
    func loadUsers(completion: @escaping ([User]) -> Void) {
        APIService.shared.fetchUsers { result in
            switch result {
            case .success(let users):
                completion(users)
            case .failure:
                completion([])
            }
        }
    }
}

3. Then the UIViewController

Your controller just triggers the call and updates UI.


viewModel.loadUsers { users in
    DispatchQueue.main.async {
        self.updateUI(with: users)
    }
}

Why this separation matters

If you put networking inside UIViewController, you’ll run into:

  • Hard-to-test code
  • Reuse problems (same API used in multiple screens)
  • “Massive View Controller” mess
  • Tight coupling between UI and data layer

Separating it gives you:

  • Cleaner structure
  • Easier debugging
  • Reusable API logic
  • Better scalability

Mental model

  • UIViewController → displays data, handles user actions
  • ViewModel (optional) → prepares data
  • Service/API layer → fetches data from network

Yes—you can, and in some architectures you should.

If you’re using something like VIPER or Clean Architecture, the Interactor is actually the right place for coordinating network calls—but with an important nuance.

How it works conceptually

In these architectures:

  • UIViewController → handles UI only
  • Interactor → contains business logic (use cases)
  • Service/API layer → performs the actual network request

👉 The Interactor doesn’t directly “own” networking details—it uses a service that does.

Recommended structure

1. API Service (low-level networking)


protocol APIServiceProtocol {
    func fetchUsers(completion: @escaping (Result<[User], Error>) -> Void)
}

2. Interactor (business logic layer)


class UserInteractor {
    let apiService: APIServiceProtocol

    init(apiService: APIServiceProtocol) {
        self.apiService = apiService
    }

    func getUsers(completion: @escaping ([User]) -> Void) {
        apiService.fetchUsers { result in
            switch result {
            case .success(let users):
                // Business logic can go here (filtering, rules, etc.)
                completion(users)
            case .failure:
                completion([])
            }
        }
    }
}

3. ViewController


interactor.getUsers { users in
    DispatchQueue.main.async {
        self.updateUI(with: users)
    }
}

What NOT to do ❌

Avoid putting raw networking code directly inside the Interactor like this:


// ❌ Not ideal
URLSession.shared.dataTask(with: url) { ... }

That tightly couples your business logic to a specific networking implementation.

The key idea

  • ✅ Interactor = “what to fetch and why”
  • ✅ Service = “how to fetch”

When it’s okay to bend the rule

For very small apps or prototypes, putting API calls directly in the Interactor isn’t catastrophic—but it won’t scale well.

Bottom line

  • ✔ Yes, API calls can be triggered from the Interactor
  • ✔ But the actual networking logic should live in a separate service layer
  • ✔ The Interactor should depend on abstractions (protocols), not implementations
