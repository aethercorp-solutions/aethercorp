Local persistence frameworks for iOS apps

Local persistence frameworks for iOS apps are tools or APIs that let you store data directly on the device so it remains available between app launches, even without an internet connection. Apple and third-party developers provide several options, each suited to different needs.

Here are the main ones you’ll encounter:

1. Core Data

This is Apple’s primary, high-level persistence framework. It’s not just a database—it’s an object graph and data management system.

  • Best for: Complex apps with structured data and relationships
  • Features: Data modeling, querying, change tracking, undo/redo, iCloud sync support
  • Works with SQLite under the hood, but abstracts it away

2. SQLite

A lightweight, file-based relational database built into iOS.

  • Best for: Apps needing direct SQL control or high performance
  • Features: Full SQL support, fast, reliable
  • Downside: Requires writing raw SQL and managing schema manually

3. Realm

A popular third-party mobile database that’s often seen as simpler than Core Data.

  • Best for: Developers who want an easier API than Core Data
  • Features: Object-oriented, fast, real-time updates, cross-platform support
  • Downside: Adds external dependency

4. UserDefaults

A simple key-value storage system.

  • Best for: Small pieces of data (settings, flags, preferences)
  • Features: Easy to use, built-in
  • Not suitable for: Large or complex data

5. Keychain

Secure storage for sensitive data.

  • Best for: Passwords, tokens, credentials
  • Features: Encrypted, secure, persists across app reinstalls (in some cases)
  • Not for general app data

6. File System (using FileManager)

You can store files directly in the app’s sandbox.

  • Best for: Media files, documents, cached data
  • Features: Full control over file storage
  • Requires manual organization and handling

7. SwiftData

A newer Apple framework introduced to modernize Core Data.

  • Best for: SwiftUI apps and modern Swift development
  • Features: Simpler syntax, integrates with SwiftUI, uses Swift types directly
  • Think of it as a more developer-friendly evolution of Core Data

How to choose

  • Simple settings → UserDefaults
  • Sensitive info → Keychain
  • Structured app data (modern) → SwiftData
  • Structured app data (mature/legacy) → Core Data
  • Full SQL control → SQLite
  • Simpler alternative to Core Data → Realm
  • Files/media → FileManager

It’s not just a database—it’s an object graph and data management system.

That line is contrasting a simple database with something more sophisticated.

A traditional database (like a relational database) mainly stores data in tables—rows and columns—and you query it to get pieces of data.

When something is described as “an object graph and data management system,” it usually means:

  • Object graph:

    Data is represented as interconnected objects, similar to how objects reference each other in programming (like in Java or Python). Instead of flat tables, you have a network (graph) of objects linked together—like a user object connected to orders, addresses, friends, etc.

  • Data management system:

    It doesn’t just store data—it also handles things like:

    • Relationships between objects
    • Tracking changes
    • Caching
    • Synchronization
    • Querying in a more object-oriented way

So the sentence is basically saying:

“This isn’t just a place to store data—it actively manages complex, connected data structures in a way that mirrors how your application works.”

In practice, this kind of system is often used in:

  • ORMs (Object-Relational Mappers)
  • Graph databases
  • Frameworks that keep app data in sync with storage
