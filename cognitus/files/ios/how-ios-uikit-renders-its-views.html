How ios uikit renders its views?

iOS UIKit rendering is a layered, highly optimized pipeline that turns your view hierarchy into pixels on screen. Under the hood, it’s not just UIKit—it heavily relies on Core Animation and the GPU.

Let’s break it down in a way that matches how you’d reason about performance as a developer 👇

🧱 1. View Hierarchy (UIKit Level)

At the top level, everything starts with UIView.

  • You build a tree: 
    
UIWindow
  └── UIView
       ├── UILabel
       ├── UIImageView
       └── UIButton

Each UIView:

  • Handles layout (layoutSubviews)
  • Handles drawing (drawRect: if needed)
  • Owns a backing CALayer

👉 Important: UIKit itself does not render pixels directly.

🎨 2. Backing Layer (Core Animation Bridge)

Every UIView is backed by a CALayer (from Core Animation).

  • UIView = high-level API (touch, layout, logic)
  • CALayer = rendering + animation

Example mapping:


view.backgroundColor → layer.backgroundColor
view.frame           → layer.bounds + position
view.alpha           → layer.opacity

👉 The real rendering happens at the layer level, not the view.

🧠 3. Layout Pass (CPU)

Before anything is drawn:

  1. Auto Layout / manual layout runs
  2. Calls:
    • setNeedsLayout
    • layoutIfNeeded
    • layoutSubviews

This happens on the main thread (CPU).

✏️ 4. Display Pass (CPU → Bitmap)

If a view needs custom drawing:

  • setNeedsDisplay → triggers drawRect:
  • Runs on CPU
  • Produces a bitmap (texture)

For standard views:

  • UIKit skips drawRect: and uses optimized layer properties

👉 Key point:

  • Custom drawing = CPU heavy
  • Layer properties (cornerRadius, shadow, etc.) = GPU friendly (usually)

📦 5. Layer Tree Commit

After layout + display:

  • UIKit builds a Layer Tree
  • This tree is sent to Core Animation Render Server

This step is called:

“Commit Transaction”

👉 Happens at the end of the run loop.

🚀 6. Render Server + GPU (Actual Rendering)

Now we leave your app process.

Core Animation:

  • Takes the layer tree
  • Sends it to the Render Server (separate process)

GPU handles:

  • Compositing layers
  • Applying transforms
  • Animations (smooth, 60fps / 120fps)

👉 This is why animations are smooth even if CPU is busy.

🔁 7. Double Buffering & VSync

iOS uses:

  • Double buffering
  • Synced with display refresh (VSync)

Typical flow:

  1. Frame prepared offscreen
  2. Swapped on screen at refresh tick

⚡ Rendering Pipeline Summary


UIView (UIKit)
   ↓
CALayer (Core Animation)
   ↓
Layout Pass (CPU)
   ↓
Display Pass (CPU → bitmap if needed)
   ↓
Layer Tree Commit
   ↓
Render Server
   ↓
GPU Compositing
   ↓
Screen

🔥 Key Performance Insights (VERY IMPORTANT)

  1. Avoid unnecessary drawRect:
    • Expensive CPU work
    • Prefer layer properties
  2. Offscreen Rendering Triggers

    Watch out for:

    • cornerRadius + masksToBounds
    • shadows
    • group opacity

    3. 👉 These force extra GPU passes

  3. Minimize View Hierarchy Depth

    Too many nested views:

    • More layout work
    • More compositing cost
  4. Use Rasterization Carefully 
    
layer.shouldRasterize = YES;
    • Good for static complex views
    • Bad for dynamic content
  5. Animations are GPU-driven
    • UIView.animate → actually Core Animation
    • Doesn’t block main thread

🧩 Mental Model (Simple)

Think of UIKit like this:

  • UIView = declarative structure
  • CALayer = render instructions
  • Core Animation = frame scheduler
  • GPU = final painter

🆚 UIKit vs React Native (since you're working with RN)

  • UIKit: immediate + layer-based rendering
  • React Native:
    • JS → Bridge → Native Views → UIKit → CALayer → GPU

👉 That bridge is why performance tuning matters in RN.
