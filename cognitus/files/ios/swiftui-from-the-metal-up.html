SwiftUI from the metal up

Alright, let’s peel SwiftUI from the metal up—modern, opinionated, and honest about what’s actually happening under the hood. Think of this as the full stack of SwiftUI, bottom → top 🧱⬆️

1. Hardware & OS Layer (The Foundation)

Apple Silicon (CPU / GPU / Neural Engine)

SwiftUI is designed specifically for Apple Silicon-era devices:

  • Massive parallelism (big.LITTLE cores)
  • Metal-accelerated rendering
  • Unified memory (CPU + GPU share RAM)

Why this matters:

  • Animations, transitions, and layout diffs are cheap
  • Declarative updates don’t kill performance

SwiftUI assumes:

“Recompute views often — the system is fast enough.”

Operating System (iOS / macOS / watchOS / tvOS / visionOS)

SwiftUI is not cross-platform like React Native.

It’s a single abstraction layer over multiple native OS frameworks.

Each OS provides:

  • Window & event lifecycle
  • Accessibility
  • Input systems (touch, pointer, keyboard, gaze)
  • Rendering surfaces

SwiftUI adapts automatically:


Button("Save") { }

Same code → different behavior on:

  • iPhone (touch)
  • Mac (mouse + keyboard)
  • Vision Pro (hand + eye)

2. Core System Frameworks (The Hidden Engines)

Core Animation

SwiftUI does not draw pixels directly.

Instead:

  • Generates a render tree
  • Hands it to Core Animation
  • CA batches GPU commands efficiently

This is why:

  • Animations feel “free”
  • State-driven animations don’t block UI

Metal

Metal is the final renderer.

SwiftUI:

  • Compiles views → display lists
  • Uses Metal shaders for:
    • Gradients
    • Blurs
    • Shadows
    • Effects (.material, .blur, .opacity)

When you write:


.background(.ultraThinMaterial)

You’re basically invoking:

“Apply a GPU-backed real-time blur pipeline”

3. Swift Language Features (Critical to SwiftUI)

SwiftUI is impossible without modern Swift.

Result Builders (@ViewBuilder)

This is what allows this magic:


var body: some View {
    if isLoggedIn {
        Dashboard()
    } else {
        Login()
    }
}

Under the hood:

  • Transforms conditional blocks into a type-safe view tree
  • No runtime interpretation
  • No reflection

Compile-time magic 🧠

Value Types & Immutability

SwiftUI views are:

  • structs
  • Cheap
  • Disposable
  • Recreated constantly

Text("Hello")

This is not a UI object.

It’s a description of UI.

SwiftUI views are blueprints, not houses.

Protocol-Oriented Design

Everything is:


protocol View {
    associatedtype Body : View
}

Benefits:

  • Zero-cost abstractions
  • Static dispatch
  • Aggressive compiler optimizations

4. SwiftUI Runtime (The Brain)

This is where SwiftUI becomes special.

Diffing Engine

SwiftUI:

  1. Rebuilds the entire view tree
  2. Compares it to the previous tree
  3. Applies minimal mutations to the render tree

This is similar to React—but:

  • Fully compile-time typed
  • No virtual DOM
  • No string-based reconciliation

Key principle:

“Views are cheap. Side effects are expensive.”

Identity & Stability

SwiftUI tracks views via:

  • Structural position
  • Explicit IDs (.id())
  • ForEach identity

Bad identity = weird bugs 😅


ForEach(items, id: \.id) { item in
    Row(item)
}

This tells SwiftUI:

“This view represents the same conceptual entity across updates.”

5. State System (The Heartbeat)

SwiftUI is state-driven UI.

Property Wrappers

These are not just syntax sugar.

Wrapper Ownership Purpose
@State View Local mutable state
@Binding External Two-way state
@StateObject View Owns reference model
@ObservedObject External Subscribes only
@EnvironmentObject App Dependency injection
@Environment System OS-provided values

Example:


@State private var count = 0

Triggers:

  1. State change
  2. View invalidation
  3. Tree diff
  4. Render update

No manual refresh. Ever.

Observation (Modern SwiftUI)

iOS 17+:


@Observable class UserModel {
    var name: String = ""
}

Replaces:

  • ObservableObject
  • @Published
  • Combine boilerplate

Swift compiler now tracks field-level mutations.

This is huge.

6. Layout System (Constraint-Free but Strict)

SwiftUI layout works in 3 passes:

1️⃣ Propose Size

Parent says:

“You may be up to this big.”

2️⃣ Choose Size

Child responds:

“I want this size.”

3️⃣ Position

Parent places the child.

Unlike Auto Layout:

  • No constraints
  • No solver
  • Deterministic
  • Predictable

This is why SwiftUI layouts are fast.

Layout Protocol (Advanced)

Custom layouts:


struct MyLayout: Layout {
    func sizeThatFits(...)
    func placeSubviews(...)
}

This is lower-level than most devs ever need—and that’s intentional.

7. Navigation & Scene Management

App Protocol


@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}

This replaces:

  • AppDelegate
  • SceneDelegate
  • Storyboards

Everything is declarative.

NavigationStack

SwiftUI models navigation as data:


@State private var path: [Route] = []

UI = projection of state.

8. Animation System (Physics, Not Timelines)

Animations are:

  • State transitions
  • Interpolated by the system

withAnimation(.spring()) {
    expanded.toggle()
}

SwiftUI:

  • Captures before & after states
  • Calculates deltas
  • Applies physics-based interpolation

This is why animations “just work”.

9. Interop Layer (UIKit / AppKit)

SwiftUI is not replacing UIKit.

It’s sitting on top of it.

You can:

  • Embed UIKit views
  • Access view controllers
  • Drop to imperative code when needed

UIViewRepresentable

Apple’s philosophy:

“Use SwiftUI first. Escape when necessary.”

10. Philosophy (Why SwiftUI Feels Different)

SwiftUI is built on these beliefs:

  • UI is a function of state
  • Recompute everything
  • Let the compiler optimize
  • Let the GPU do the work
  • Avoid mutation
  • Make illegal states unrepresentable

This is functional UI married to systems-level performance.

Mental Model (TL;DR)


State changes
   ↓
View tree rebuilds
   ↓
Diffing engine computes minimal updates
   ↓
Core Animation updates render tree
   ↓
Metal draws pixels
