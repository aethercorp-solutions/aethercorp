Swift memory management

Swift memory management is mainly about how the language allocates, uses, and frees memory automatically so you don’t have to manually manage it like in C or older Objective-C code. The core system behind this is called ARC (Automatic Reference Counting).

Let’s break it down clearly and in depth.

1. The Big Idea: ARC (Automatic Reference Counting)

Swift uses ARC to track how many references (variables, constants, properties) point to a class instance.

Rule:

  • When an object’s reference count becomes 0 → it is deallocated (freed from memory).
  • When you assign a class instance to a variable → reference count increases.
  • When a reference is removed → reference count decreases.

Example


class Person {
    var name: String

    init(name: String) {
        self.name = name
        print("\(name) is being initialized")
    }

    deinit {
        print("\(name) is being deinitialized")
    }
}

var p1: Person? = Person(name: "John")  // RC = 1
var p2 = p1                              // RC = 2

p1 = nil                                 // RC = 1
p2 = nil                                 // RC = 0 → deinit called

2. Value Types vs Reference Types

Value Types (Stack-based)

  • Structs, enums, tuples
  • Copied when assigned
  • Each copy is independent
  • Usually stored on stack (fast, auto cleanup)

struct Car {
    var model: String
}

var c1 = Car(model: "BMW")
var c2 = c1   // copy, not reference

✔ No ARC needed

Reference Types (Heap-based)

  • Classes
  • Stored in heap memory
  • Shared references
  • Managed by ARC

class Car {
    var model: String
    init(model: String) { self.model = model }
}

✔ ARC applies here

3. How ARC Works Internally

Swift keeps a reference count table:

Action Effect
Create object RC = 1
Assign to new variable RC +1
Remove reference RC -1
RC = 0 object deallocated

4. Strong, Weak, and Unowned References

These are used to prevent memory leaks (especially retain cycles).

4.1 Strong Reference (Default)


class A {
    var b: B?
}
  • Increases reference count
  • Prevents deallocation

👉 Most references are strong by default

Problem: Retain Cycle


class A {
    var b: B?
}

class B {
    var a: A?
}

If A holds B and B holds A:

  • RC never becomes 0
  • Memory leak occurs ❌

4.2 Weak Reference


class B {
    weak var a: A?
}

Key points:

  • Does NOT increase reference count
  • Automatically becomes nil when object is deallocated
  • Must be optional (A?)

✔ Used in parent-child relationships (common in delegates)

Example:


class Owner {
    var dog: Dog?
}

class Dog {
    weak var owner: Owner?
}

4.3 Unowned Reference


class B {
    unowned var a: A
}

Key points:

  • Does NOT increase reference count
  • NOT optional
  • Assumes object will never be nil when accessed

⚠ Dangerous if object is deallocated → crash

When to use:

  • When lifecycle is guaranteed

Example:


class Customer {
    var card: CreditCard?
}

class CreditCard {
    unowned let customer: Customer

    init(customer: Customer) {
        self.customer = customer
    }
}

5. Closures and Retain Cycles

Closures capture references strongly by default.

Problem:


class MyClass {
    var name = "Swift"

    lazy var closure: () -> Void = {
        print(self.name)
    }
}

Here:

  • closure strongly captures self
  • self strongly holds closure
  • retain cycle ❌

Fix: Capture List


lazy var closure: () -> Void = { [weak self] in
    print(self?.name ?? "")
}

Or:


lazy var closure: () -> Void = { [unowned self] in
    print(self.name)
}

6. Memory Regions in Swift

6.1 Stack

  • Stores function calls, value types
  • Very fast
  • Auto-managed

6.2 Heap

  • Stores class instances
  • Slower but flexible
  • Managed by ARC

6.3 Static/Data segment

  • Global/static variables
  • Lives for entire app lifecycle

7. Lifecycle of an Object

  1. Allocation (heap)
  2. Initialization (init)
  3. Usage (references tracked)
  4. ARC updates count
  5. When RC = 0 → deinit
  6. Memory freed

8. Common Memory Issues

1. Retain Cycles

  • Two objects strongly referencing each other

2. Closure capturing self strongly

3. Delegates not marked weak


protocol Delegate: AnyObject {}

class A {
    weak var delegate: Delegate?
}

9. Best Practices

  • Prefer value types (structs) when possible
  • Use weak for delegates
  • Use weak/unowned in closures carefully
  • Avoid unnecessary strong references
  • Break retain cycles explicitly

10. Quick Summary

Concept Meaning
ARC Automatic memory management system
Strong Increases reference count
Weak Doesn’t increase count, becomes nil
Unowned Doesn’t increase count, non-optional
Retain cycle Memory leak due to circular references
