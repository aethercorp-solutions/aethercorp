Keychain Services and UserDefaults

In iOS development, Keychain Services and UserDefaults serve very different purposes. Mixing them up can create serious security issues, so it’s worth being precise.

🔐 Keychain Services

Use this for sensitive data.

Keychain is an encrypted, secure storage managed by the OS.

Best for:

  • Passwords
  • OAuth / JWT tokens
  • API keys
  • Private credentials

Characteristics:

  • Data is encrypted at rest
  • Survives app reinstall (in many cases)
  • Protected by device security (Face ID, passcode, etc.)
  • Not easily accessible even if device is compromised (compared to normal storage)

Example use case:


saveTokenToKeychain("eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...")

🧾 UserDefaults

Use this for non-sensitive preferences.

UserDefaults is basically a simple key-value store.

Best for:

  • App settings (dark mode, language)
  • Flags (isFirstLaunch, hasSeenOnboarding)
  • Lightweight cached values

Characteristics:

  • Not encrypted
  • Stored in plain plist files
  • Fast and simple
  • Easily readable if someone inspects the app container

Example:


UserDefaults.standard.set(true, forKey: "isLoggedIn")

⚠️ Critical Difference

Aspect Keychain Services 🔐 UserDefaults 🧾
Security Encrypted Not encrypted
Use case Secrets Preferences
Persistence Long-term, secure Simple storage
Risk level Low (secure) High if misused

🚨 Practical Rule (Don’t skip this)

  • If it can compromise a user account → use Keychain
  • If it just changes UI/behavior → use UserDefaults

💡 Real-world (your setup)

Given you’re working with:

  • JWT auth
  • AWS Lambda APIs
  • Mobile app auth flows

👉 Store your JWT / refresh tokens in Keychain, not UserDefaults.

Storing tokens in UserDefaults is a common mistake and can expose users to token theft (especially on jailbroken devices or backups).

Secure Enclave

A Secure Enclave is a dedicated, isolated hardware component inside a device that is designed to protect sensitive data by keeping it completely separate from the main processor and operating system.

Think of it as a locked mini-computer inside your phone or laptop that handles the most sensitive security tasks.

Key idea

Even if your main operating system is compromised (malware, jailbreak, etc.), the Secure Enclave is still designed to keep certain data safe and inaccessible.

🧠 How it works

The Secure Enclave has its own:

  • CPU
  • Secure memory
  • Operating system (a very minimal, security-focused OS)
  • Encrypted storage

It communicates with the main system only through tightly controlled channels.

🔐 What it protects

On devices like those from Apple, the Secure Enclave is used for:

  • Face ID / Touch ID biometric data
  • Device passcodes
  • Encryption keys (used to encrypt files, messages, etc.)
  • Apple Pay payment credentials
  • Keychain secrets (in some cases)

Importantly:

Your biometric data (like Face ID) never leaves the Secure Enclave.

🧩 Why it exists

The goal is to prevent attackers from accessing critical data even if they:

  • Hack the operating system
  • Gain root access
  • Install malicious software

So instead of storing secrets in normal memory or disk, they’re stored in a physically isolated chip area.

📱 Where you’ll find it

Secure Enclave is commonly found in:

  • iPhones (A-series chips)
  • iPads
  • Macs with Apple Silicon (M-series chips)
  • Apple Watches

⚡ Simple analogy

  • Main system = your house
  • Secure Enclave = a locked safe embedded in the wall
  • Even if someone breaks into the house, they still can’t open the safe
