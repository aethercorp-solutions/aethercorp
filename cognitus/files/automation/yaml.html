YAML

YAML stands for “YAML Ain’t Markup Language.”

It’s a human-readable data format commonly used for configuration files.

You’ll see it a lot in:

  • DevOps (Docker, Kubernetes, GitHub Actions)
  • Backend configs (Serverless Framework, CI/CD)
  • App settings and environment configs

Basic rules

  • Uses indentation (spaces, not tabs) to show structure
  • Key–value pairs
  • Very readable compared to JSON

Simple example


app:
  name: TryCastanet
  version: 1.0.0
  debug: true

server:
  port: 3000
  host: localhost

Lists


languages:
  - JavaScript
  - Python
  - Go

Objects inside lists


users:
  - name: Paul
    role: admin
  - name: Anna
    role: editor

YAML vs JSON (quick vibe)

  • YAML → more readable, less symbols
  • JSON → stricter, better for APIs
  • YAML → great for configs
  • JSON → great for data exchange

Ah nice — GitHub Actions + YAML ⚙️

This is where Serverless + CI/CD really starts to feel pro.

Below is a clean, battle-tested GitHub Actions workflow for deploying a Serverless Framework app to AWS.

Basic workflow: deploy Serverless on push

.github/workflows/deploy.yml


name: Deploy Serverless API

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 18

      - name: Install dependencies
        run: npm ci

      - name: Deploy to AWS (prod)
        run: npx serverless deploy --stage prod
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          AWS_REGION: ap-southeast-1
          MONGODB_URI: ${{ secrets.MONGODB_URI }}

Deploy dev vs prod automatically


on:
  push:
    branches:
      - main
      - develop
            

- name: Set stage
  run: |
    if [[ "${GITHUB_REF}" == "refs/heads/main" ]]; then
      echo "STAGE=prod" >> $GITHUB_ENV
    else
      echo "STAGE=dev" >> $GITHUB_ENV
    fi

- name: Deploy
  run: npx serverless deploy --stage $STAGE

Sure — let’s break this down in plain English 👇

This is a GitHub Actions workflow snippet that decides which environment (stage) to deploy to, and then runs a Serverless Framework deployment.

What’s happening here

  • GITHUB_REF is a built-in GitHub Actions variable.
  • It contains the branch or tag that triggered the workflow.

Examples:

  • refs/heads/main → push to main branch
  • refs/heads/dev → push to dev branch
  • refs/pull/123/merge → pull request

The logic

  • ✅ If the workflow runs from main branch

    → set STAGE=prod

  • ❌ If it runs from any other branch

    → set STAGE=dev

Why >> $GITHUB_ENV?

  • Writing to $GITHUB_ENV makes STAGE available to all following steps in the job.
  • After this step, you can safely use $STAGE.

2️⃣ “Deploy” step


- name: Deploy
  run: npx serverless deploy --stage $STAGE

What this does

  • Runs Serverless Framework via npx
  • Deploys your service using the stage decided above

So effectively:

Branch Command run
main serverless deploy --stage prod
any other branch serverless deploy --stage dev

3️⃣ Why this is useful

This pattern lets you:

  • 🚀 Auto-deploy production only from main
  • 🧪 Deploy dev/staging from feature branches
  • Keep one workflow instead of separate prod/dev pipelines

Very common in AWS Lambda + Serverless setups 👍

name: Serverless CI/CD

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      id-token: write
      contents: read

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18

      - run: npm ci

      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-serverless
          aws-region: ap-southeast-1

      - run: npx serverless deploy --stage prod

Nice, this is a clean, real-world GitHub Actions deploy job 👍

Let’s walk through it line by line and also talk about what actually happens when this runs.

High-level: what this workflow does

This job:

  • Runs on GitHub-hosted Ubuntu
  • Checks out your repo
  • Sets up Node.js 18
  • Installs dependencies
  • Deploys your Serverless Framework app to AWS (prod stage)
  • Uses GitHub Secrets for AWS + MongoDB credentials

In short:

👉 Every time this workflow runs, it deploys to AWS production.

Breakdown section by section

jobs:


jobs:
  deploy:
  • Defines a job named deploy
  • A workflow can have multiple jobs; this one has just one

runs-on


runs-on: ubuntu-latest
  • GitHub spins up a fresh Ubuntu VM
  • Nothing is cached by default (unless you add caching)

steps:

Each step runs sequentially inside that VM.

1️⃣ Checkout repo


- name: Checkout repo
  uses: actions/checkout@v4
  • Pulls your GitHub repository into the runner
  • Without this, the VM would be empty
  • Required for npm ci and serverless deploy

2️⃣ Setup Node.js


- name: Setup Node.js
  uses: actions/setup-node@v4
  with:
    node-version: 18
  • Installs Node.js v18
  • Ensures:
    • Same Node version every deploy
    • Matches your local / Lambda runtime

💡 Good choice — Node 18 is stable and supported by AWS Lambda.

3️⃣ Install dependencies


- name: Install dependencies
  run: npm ci
  • Installs dependencies from package-lock.json
  • Faster and more deterministic than npm install
  • Fails if package-lock.json is out of sync (good for CI)

👉 This ensures reproducible builds

4️⃣ Deploy to AWS (prod)


- name: Deploy to AWS (prod)
  run: npx serverless deploy --stage prod
  • Runs the Serverless Framework without global install
  • --stage prod:
    • Uses prod stage in serverless.yml
    • Typically maps to:
      • prod Lambda functions
      • prod API Gateway
      • prod env vars

This command:

  • Packages your Lambda
  • Uploads it to S3
  • Creates/updates CloudFormation
  • Deploys API Gateway + Lambdas

  AWS_REGION: ap-southeast-1
  MONGODB_URI: ${{ secrets.MONGODB_URI }}

These are injected only for this step.

AWS credentials

  • Used by:
    • Serverless Framework
    • AWS SDK
    • CloudFormation

Best practice ✔️

(never hardcode these)

AWS_REGION

  • Sets default AWS region
  • Matches your Lambda / API Gateway region

MONGODB_URI

  • Available during deploy
  • Typically used in serverless.yml, like:

provider:
  environment:
    MONGODB_URI: ${env:MONGODB_URI}
