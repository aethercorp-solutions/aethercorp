Kong Gateway

Kong Gateway is a popular open-source API gateway used to manage, secure, and route API traffic between clients (like apps or browsers) and backend services (like microservices).

🧠 Simple explanation

Think of Kong Gateway as a “traffic controller” for APIs:

  • Clients → send requests
  • Kong Gateway → receives & processes them
  • Backend services → handle the logic
  • Kong Gateway → returns the response

It acts as a single entry point for all your APIs.

⚙️ What it actually does

Kong Gateway sits in front of your services and handles common API concerns:

  • Routing → sends requests to the correct service
  • Authentication & security → verifies users (JWT, OAuth, etc.)
  • Rate limiting → prevents abuse (e.g., too many requests)
  • Load balancing → distributes traffic across services
  • Logging & monitoring → tracks API usage
  • Transformation → modifies requests/responses

👉 All of this happens without changing your backend code.

🏗️ How it fits in architecture

In modern systems (especially microservices), you might have dozens of APIs.

Instead of exposing all of them directly:


Client → Kong Gateway → Multiple Services (User, Payment, Orders...)

This simplifies things because:

  • Clients talk to one endpoint
  • Kong handles complexity behind the scenes

🚀 Key features of Kong Gateway

  • High performance (built on NGINX)
  • Plugin system (easy to extend functionality)
  • Cloud-native & Kubernetes support
  • Works anywhere (on-prem, cloud, hybrid)
  • Declarative config & APIs for automation

💡 Why people use Kong

  • Centralized API management
  • Strong security layer
  • Scales easily with microservices
  • Reduces complexity for frontend apps
  • Improves performance and reliability

🧠 Core Responsibilities of an API Gateway

At minimum, your custom gateway should:

  • Route requests → correct backend service
  • Handle authentication (JWT, API keys, OAuth)
  • Rate limit / throttle
  • Transform requests/responses (optional)
  • Logging & monitoring

🚀 Step-by-Step (Node.js from scratch)

1. Basic HTTP Server

Use Express as your base:


npm init -y
npm install express http-proxy-middleware jsonwebtoken
            

// gateway.js
const express = require('express');
const app = express();

app.use(express.json());

app.listen(3000, () => {
  console.log('API Gateway running on port 3000');
});

2. Request Routing (Reverse Proxy)


const { createProxyMiddleware } = require('http-proxy-middleware');

app.use('/users', createProxyMiddleware({
  target: 'http://localhost:4001',
  changeOrigin: true
}));

app.use('/payments', createProxyMiddleware({
  target: 'http://localhost:4002',
  changeOrigin: true
}));

👉 Now:

  • /users → User service
  • /payments → Payment service

3. Authentication Middleware (JWT)


const jwt = require('jsonwebtoken');

function authMiddleware(req, res, next) {
  const token = req.headers['authorization'];

  if (!token) return res.status(401).send('No token');

  try {
    const decoded = jwt.verify(token, 'SECRET_KEY');
    req.user = decoded;
    next();
  } catch (err) {
    return res.status(403).send('Invalid token');
  }
}

Apply it:


app.use('/payments', authMiddleware);

4. Rate Limiting


npm install express-rate-limit
            

const rateLimit = require('express-rate-limit');

const limiter = rateLimit({
  windowMs: 1 * 60 * 1000, // 1 min
  max: 100
});

app.use(limiter);

5. Logging


npm install morgan
            

const morgan = require('morgan');
app.use(morgan('combined'));

6. Request Transformation (Optional)


app.use('/users', (req, res, next) => {
    req.headers['x-gateway'] = 'custom-api-gateway';
    next();
});
