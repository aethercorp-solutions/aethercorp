Unified sign-in + middleware setup

If you’re aiming for a clean, production-ready unified sign-in + middleware setup in AWS Lambda (Node.js), the key is to stop treating each provider (Google, Cognito, Azure AD, etc.) differently at the route level and instead normalize everything into one token verification layer + one middleware.

Here’s a practical architecture that fits your stack (Lambda + API Gateway + MongoDB + React Native):

🔐 1. High-Level Flow (Unified Sign-In)

Client (React Native)

  • User signs in via:
    • Google
    • Azure AD
    • Cognito Hosted UI (optional)
  • Client sends ID token / access token to backend

Backend (Lambda)

  1. Detect provider
  2. Verify token
  3. Normalize user profile
  4. Issue your own JWT (session token)

🧠 2. Core Idea: Your Backend Owns the Session

Never rely on external tokens for authorization after login.

Instead:


[Google ID Token] → verify → create/find user → issue JWT (your system)

After that:


Client → sends your JWT → middleware validates → access granted

🔑 4. Token Verification (Provider Layer)

Example: Google


const { OAuth2Client } = require('google-auth-library');
const client = new OAuth2Client(process.env.GOOGLE_CLIENT_ID);

async function verifyGoogleToken(idToken) {
    const ticket = await client.verifyIdToken({
        idToken,
        audience: process.env.GOOGLE_CLIENT_ID
    });

    const payload = ticket.getPayload();

    return {
        email: payload.email,
        name: payload.name,
        picture: payload.picture,
        provider: 'google',
        providerId: payload.sub
    };
}

module.exports = { verifyGoogleToken };

🔄 5. Unified Auth Service


const { verifyGoogleToken } = require('./providers/google');
// add azure, cognito...

async function verifyProviderToken(provider, token) {
    switch (provider) {
        case 'google':
            return await verifyGoogleToken(token);

        case 'azure':
            return await verifyAzureToken(token);

        case 'cognito':
            return await verifyCognitoToken(token);

        default:
            throw new Error('Unsupported provider');
    }
}

👤 6. Sign-In Lambda Handler


const jwt = require('jsonwebtoken');
const UserRepo = require('../user.repository');
const { verifyProviderToken } = require('./auth.service');

module.exports.signIn = async (event) => {
    const body = JSON.parse(event.body);
    const { provider, token } = body;

    const profile = await verifyProviderToken(provider, token);

    let user = await UserRepo.findByEmail(profile.email);

    if (!user) {
        user = await UserRepo.create(profile);
    }

    const appToken = jwt.sign(
        {
            userId: user._id,
            email: user.email
        },
        process.env.JWT_SECRET,
        { expiresIn: '7d' }
    );

    return {
        statusCode: 200,
        body: JSON.stringify({
            token: appToken,
            user
        })
    };
};

🛡️ 7. Middleware for API Gateway (Lambda)

Since Lambda doesn’t have Express middleware, you simulate it.

Option A: Reusable Middleware Wrapper


const jwt = require('jsonwebtoken');

function authMiddleware(handler) {
    return async (event) => {
        try {
            const authHeader = event.headers.Authorization || '';
            const token = authHeader.replace('Bearer ', '');

            if (!token) {
                return { statusCode: 401, body: 'Unauthorized' };
            }

            const decoded = jwt.verify(token, process.env.JWT_SECRET);

            event.user = decoded;

            return await handler(event);

        } catch (err) {
            return { statusCode: 401, body: 'Invalid token' };
        }
    };
}

module.exports = authMiddleware;

Usage:


const authMiddleware = require('./auth.middleware');

const handler = async (event) => {
    return {
        statusCode: 200,
        body: JSON.stringify({
            message: 'Protected route',
            user: event.user
        })
    };
};

module.exports.protectedRoute = authMiddleware(handler);

⚡ 8. Alternative (Better for Scale): API Gateway Authorizer

Instead of per-Lambda middleware:

👉 Use Lambda Authorizer

Flow:


Request → Authorizer → Validate JWT → Allow/Deny → Lambda

Benefits:

  • Centralized auth
  • Faster execution (cached)
  • Cleaner Lambdas

Example Authorizer


const jwt = require('jsonwebtoken');

exports.handler = async (event) => {
    try {
        const token = event.authorizationToken.replace('Bearer ', '');
        const decoded = jwt.verify(token, process.env.JWT_SECRET);

        return {
            principalId: decoded.userId,
            policyDocument: {
                Version: '2012-10-17',
                Statement: [
                    {
                        Action: 'execute-api:Invoke',
                        Effect: 'Allow',
                        Resource: event.methodArn
                    }
                ]
            },
            context: {
                userId: decoded.userId,
                email: decoded.email
            }
        };
    } catch (e) {
        throw new Error('Unauthorized');
    }
};

🧩 9. Recommended Setup (Best Practice)

For your case:

✅ Use:

  • Provider verification (Google/Azure)
  • Issue your own JWT
  • Use API Gateway Authorizer

🚫 Avoid:

  • Passing Google tokens to every API call
  • Verifying Google token on every request (slow + costly)
