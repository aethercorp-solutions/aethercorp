Practical breakdown of Amazon DynamoDB usage and implementation

Here’s a clear, practical breakdown of Amazon DynamoDB usage and implementation, especially useful for your serverless Node.js + AWS Lambda setup.

🚀 What is Amazon DynamoDB?

A fully managed NoSQL database from Amazon Web Services designed for:

  • High performance (single-digit millisecond latency)
  • Massive scalability
  • Serverless architecture (no servers to manage)

📌 Common Use Cases

DynamoDB is ideal for:

1. Serverless Applications

  • Backend for AWS Lambda APIs
  • Event-driven systems

2. Real-Time Apps

  • Chat apps
  • Gaming leaderboards
  • IoT telemetry

3. User Data / Auth Systems

  • Store user profiles
  • Session storage
  • JWT refresh tokens (fits your current work)

4. E-commerce

  • Product catalogs
  • Orders
  • Inventory

🧠 Core Concepts

1. Table

Like a collection in MongoDB


Users
Orders
Sessions

2. Primary Key (VERY IMPORTANT)

Two types:

  • Partition Key (PK)
  • Partition + Sort Key (PK + SK)

Example:


{
    "userId": "123",   // PK
    "createdAt": "2026-04-13" // SK
}

3. Items

Each row = JSON object


{
    "userId": "123",
    "name": "Paul",
    "email": "paul@email.com"
}

4. Indexes

  • GSI (Global Secondary Index) → query by other fields
  • LSI (Local Secondary Index) → alternate sort key

⚙️ Implementation (Node.js + AWS SDK)

1. Install SDK


npm install @aws-sdk/client-dynamodb @aws-sdk/lib-dynamodb

2. Setup DynamoDB Client


import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import { DynamoDBDocumentClient } from "@aws-sdk/lib-dynamodb";

const client = new DynamoDBClient({
  region: "ap-southeast-1"
});

export const ddb = DynamoDBDocumentClient.from(client);

3. Create Item (Put)


import { PutCommand } from "@aws-sdk/lib-dynamodb";

export const createUser = async () => {
  await ddb.send(new PutCommand({
    TableName: "Users",
    Item: {
      userId: "123",
      name: "Paul",
      email: "paul@email.com"
    }
  }));
};

4. Get Item


import { GetCommand } from "@aws-sdk/lib-dynamodb";

export const getUser = async () => {
  const result = await ddb.send(new GetCommand({
    TableName: "Users",
    Key: { userId: "123" }
  }));

  return result.Item;
};

5. Query (Using Sort Key)


import { QueryCommand } from "@aws-sdk/lib-dynamodb";

export const getUserOrders = async () => {
  const result = await ddb.send(new QueryCommand({
    TableName: "Orders",
    KeyConditionExpression: "userId = :uid",
    ExpressionAttributeValues: {
      ":uid": "123"
    }
  }));

  return result.Items;
};

6. Update Item


import { UpdateCommand } from "@aws-sdk/lib-dynamodb";

export const updateUser = async () => {
  await ddb.send(new UpdateCommand({
    TableName: "Users",
    Key: { userId: "123" },
    UpdateExpression: "set #n = :name",
    ExpressionAttributeNames: {
      "#n": "name"
    },
    ExpressionAttributeValues: {
      ":name": "New Name"
    }
  }));
};

7. Delete Item


import { DeleteCommand } from "@aws-sdk/lib-dynamodb";

export const deleteUser = async () => {
  await ddb.send(new DeleteCommand({
    TableName: "Users",
    Key: { userId: "123" }
  }));
};

🔐 IAM Permissions (Important for your setup)

Your Lambda role must allow:


{
    "Effect": "Allow",
    "Action": [
        "dynamodb:PutItem",
        "dynamodb:GetItem",
        "dynamodb:Query",
        "dynamodb:UpdateItem",
        "dynamodb:DeleteItem"
    ],
    "Resource": "*"
}

🧩 Example Serverless Integration

serverless.yml


provider:

  runtime: nodejs20.x
  iam:
    role:
      statements:
        - Effect: Allow
          Action:
            - dynamodb:*

⚡ Best Practices (VERY IMPORTANT)

1. Design for Queries (Not Tables)

DynamoDB is query-first, unlike SQL.

👉 Ask:

"How will I query this data?"

2. Use Single Table Design (Advanced)

Instead of:


Users table
Orders table

Use:


MainTable
PK: USER#123
SK: ORDER#456

3. Avoid Scans 🚫


ScanCommand ❌ (slow & expensive)
QueryCommand ✅ (fast)

4. Use TTL (Auto Delete)

Perfect for:

  • Sessions
  • Refresh tokens

{
    "ttl": 1710000000
}

5. Use GSIs for Flexibility

Example:

  • Query by email instead of userId
