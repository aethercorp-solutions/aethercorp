Manually set up a custom domain for a React web app

To manually set up a custom domain for a React web app, the steps depend on where you're hosting it. Since you work heavily with AWS (CloudFront, S3, Route 53, ACM), I’ll show both:

  1. ✅ Generic hosting (Vercel / Netlify / any server)
  2. ✅ Manual AWS setup (S3 + CloudFront + Route 53 + ACM) — production-ready approach

✅ OPTION 1: Generic Hosting (Vercel / Netlify / Any Host)

If your React app is already deployed:

Step 1 — Buy a Domain

You can buy from:

  • Namecheap
  • GoDaddy
  • Google Domains

Step 2 — Add Custom Domain in Hosting Dashboard

Example:

  • Go to Project Settings
  • Click Domains
  • Add: yourdomain.com

The host will give you either:

  • A CNAME record, or
  • An A record (IP address)

Step 3 — Configure DNS at Your Domain Registrar

Go to DNS settings and add:

If CNAME:


Type: CNAME
Host: www
Value: yourapp.hostingplatform.com

If A record:


Type: A
Host: @
Value: 123.45.67.89

Wait 5–30 minutes for propagation.

✅ OPTION 2: Manual AWS Setup (S3 + CloudFront + Route 53)

This is the correct scalable architecture for production.

Architecture Overview


React Build → S3 → CloudFront → Route 53 → Custom Domain
            ↑
          ACM SSL

STEP 1 — Build Your React App


npm run build

This generates the /build folder.

STEP 2 — Upload to S3

  1. Go to Amazon S3
  2. Create bucket: 
    
yourdomain.com
  3. Disable "Block all public access"
  4. Enable Static Website Hosting
  5. Set:
    • Index: index.html
    • Error: index.html (important for React routing)

Upload your /build files.

STEP 3 — Request SSL Certificate

Go to:

  • AWS Certificate Manager

⚠ IMPORTANT:

Request certificate in us-east-1 (required for CloudFront)

Add:


yourdomain.com
www.yourdomain.com

Choose DNS validation

Create validation record in Route 53.

Wait until status = Issued.

STEP 4 — Create CloudFront Distribution

Go to:

  • Amazon CloudFront

Origin:


S3 bucket (NOT the website endpoint)

Enable:

  • Redirect HTTP → HTTPS
  • Add Alternate Domain Name: 
    
yourdomain.com
www.yourdomain.com
  • Attach ACM certificate

Default Root Object:


index.html

Deploy (takes ~10–15 mins)

STEP 5 — Point Domain to CloudFront

Go to:

  • Amazon Route 53

Create records:

For root domain:


Type: A
Alias: YES
Alias Target: CloudFront distribution

For www:


Type: A
Alias: YES
Alias Target: CloudFront distribution

Important for React Router (SPA Fix)

If using React Router:

CloudFront → Error Pages → Add:

Error Code Response
403 /index.html
404 /index.html

This prevents route refresh errors.

Optional: Redirect www → non-www

You can:

  • Create separate S3 bucket www.yourdomain.com
  • Redirect to root domain

🔐 How to secure S3 with OAC (no public access)

Perfect — this is the correct modern setup for production 🔥

Since you're building AWS serverless systems already, this is the architecture you should be using.

We’ll secure:


React App → Private S3 → CloudFront (OAC) → Custom Domain

No public S3 access at all.

🔐 What is OAC?

OAC (Origin Access Control) is the new secure way for

Amazon CloudFront to access

Amazon S3

It replaces the older OAI (Origin Access Identity).

With OAC:

  • S3 bucket stays completely private
  • Only CloudFront can access it
  • Signed requests are enforced
  • Uses SigV4 authentication

🏗 Secure Architecture


User
  ↓
Route 53
  ↓
CloudFront (OAC signs request)
  ↓
Private S3 bucket (no public access)

✅ STEP-BY-STEP: Secure S3 with OAC

STEP 1 — Create S3 Bucket (Private)

  1. Go to S3
  2. Create bucket: 
    
yourdomain.com
  3. ✅ KEEP: 
    
Block ALL public access = ENABLED
  4. Do NOT enable static website hosting

    (We use CloudFront origin, not website endpoint)

Upload your React /build files.

STEP 2 — Create CloudFront Distribution

Go to CloudFront → Create distribution

Origin Settings:

Origin Domain:


Select S3 bucket (not website endpoint)

Origin Access:


Origin access control settings (recommended)

Click:


Create new OAC

Settings:

  • Signing behavior: Sign requests
  • Signing protocol: SigV4

Save.

STEP 3 — Attach Bucket Policy (IMPORTANT)

After creating OAC, CloudFront will show:

“Update bucket policy”

Click Copy policy

Go to:

S3 → Permissions → Bucket Policy → Paste policy

It will look like:


{
    "Version": "2012-10-17",
    "Statement": [
        {
        "Sid": "AllowCloudFrontServicePrincipalReadOnly",
        "Effect": "Allow",
        "Principal": {
            "Service": "cloudfront.amazonaws.com"
        },
        "Action": "s3:GetObject",
        "Resource": "arn:aws:s3:::yourdomain.com/*",
        "Condition": {
            "StringEquals": {
            "AWS:SourceArn": "arn:aws:cloudfront::ACCOUNT-ID:distribution/DISTRIBUTION-ID"
            }
        }
        }
    ]
}

This means:

  • Only YOUR CloudFront distribution can access S3
  • Direct S3 URL access = ❌ Forbidden

STEP 4 — Viewer Protocol Policy

Set:


Redirect HTTP to HTTPS

STEP 5 — Default Root Object

Set:


index.html

STEP 6 — React SPA Fix (Important)

CloudFront → Error Pages → Add:

HTTP Code Response Page Response Code
403 /index.html 200
404 /index.html 200

This prevents route refresh errors.

🔍 How to Verify It's Secure

Try opening:


https://yourbucket.s3.amazonaws.com/index.html

You should get:


Access Denied

But:


https://yourcloudfrontdomain.com

Should work.

If direct S3 works → it's NOT secure.

🚀 Production-Level Hardening (What You Should Do)

Since you're building production systems:

1️⃣ Enable CloudFront Security Headers

Add response headers policy:

  • Strict-Transport-Security
  • X-Content-Type-Options
  • X-Frame-Options
  • Content-Security-Policy

2️⃣ Enable WAF

Attach:

AWS WAF

Protect against:

  • SQL injection
  • XSS
  • Bots
  • Rate abuse

3️⃣ Enable Logging

Enable:

  • CloudFront access logs
  • S3 access logs

4️⃣ Use Origin Shield (Optional for High Traffic)

Improves cache efficiency.

🔥 Enterprise-Level Setup (Best Practice)

For something like TryCastanet:

  • Private S3
  • CloudFront OAC
  • ACM SSL (us-east-1)
  • Route 53 alias
  • WAF
  • CI/CD auto invalidation
  • Cache-control headers for assets
  • Versioned build folders

⚡ Pro Tip for React Performance

Set long cache headers for static files:

Example:


/static/*.js → Cache 1 year
index.html → No cache

This dramatically improves performance.
