access token

An access token is a short-lived credential used to authorize a user or client to access protected resources (like APIs).

🔐 Use cases of an Access Token

1. API Authentication

After login, the client receives an access token and uses it for every API request:


Authorization: Bearer <access_token>

Example:

  • Get user profile
  • Fetch contacts
  • Update data
  • Call AWS API Gateway / Lambda endpoints

2. Stateless Authentication (JWT common case)

Instead of storing sessions on the server, the server trusts the token itself.

Typical flow:

  1. User logs in
  2. Server returns access token (JWT)
  3. Client stores token (memory / secure storage)
  4. Client sends token on every request
  5. Server validates signature + expiry

3. Permission Control (Scopes/Roles)

Access tokens can contain claims like:


{
    "sub": "userId123",
    "role": "admin",
    "scope": "read:contacts write:contacts"
}

So APIs can decide:

  • Can read?
  • Can write?
  • Can access admin routes?

4. Third-party access (OAuth2)

Used when apps access another service:

  • “Login with Google”
  • “Login with Apple”
  • Access Google Drive, Facebook API, etc.

⏳ How Access Token Expiration Works

Access tokens are intentionally short-lived for security.

Typical lifespan:

  • 5 minutes → 1 hour (common)
  • Rarely more than 24 hours

🔥 Why expire them?

If stolen:

  • Attacker can only use it for a short time
  • Limits damage window

🧠 How expiration is implemented

1. JWT exp claim (most common)


{
    "exp": 1714380000
}

Server checks:

  • If current time > exp → reject token

2. Server-side session store (less common with JWT)

  • Store token/session in DB or Redis
  • Mark expired manually or via TTL

🔄 Best Practice: Use Refresh Tokens

Because access tokens expire quickly, systems usually use:

Flow:

  • Access token (short-lived, e.g. 15 min)
  • Refresh token (long-lived, e.g. 7–30 days)

When access token expires:

Client calls:


POST /refresh-token

Server returns a new access token without forcing login.

🏗️ Typical Production Flow


Login → Access Token + Refresh Token

API calls → Access Token

Expired?
   ↓
Use Refresh Token → Get new Access Token

⚠️ Common mistakes

  • ❌ Long-lived access tokens (bad security)
  • ❌ No refresh token rotation
  • ❌ Storing tokens in insecure storage (like localStorage for sensitive apps)
  • ❌ Not validating exp claim properly

💡 In your AWS / Node.js setup (important)

Since you're using AWS Lambda + API Gateway:

Best practice setup:

  • JWT access token (15 min)
  • Refresh token stored in DB (MongoDB or DynamoDB)
  • API Gateway authorizer validates JWT
  • Lambda checks scopes/roles

🔑 What a JWT looks like

JWT (JSON Web Token) is a compact, secure way to transmit information between parties, commonly used for authentication.

I’ll walk you through the core idea and a simple implementation.

A JWT has 3 parts:


header.payload.signature

Example:


eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  • Header → algorithm + token type
  • Payload → user data (id, email, etc.)
  • Signature → ensures token is valid

🧠 Basic Flow (Auth System)

  1. User logs in with username/password
  2. Server verifies credentials
  3. Server generates JWT
  4. Client stores token (cookie/localStorage)
  5. Client sends token in requests
  6. Server verifies token before granting access

⚙️ Example: Node.js (Express)

1. Install dependencies


npm install jsonwebtoken express body-parser

2. Generate JWT (Login route)


const express = require('express');
const jwt = require('jsonwebtoken');

const app = express();
app.use(express.json());

const SECRET_KEY = "your_secret_key";

app.post('/login', (req, res) => {
    const { username, password } = req.body;

    // fake user validation
    if (username === "admin" && password === "1234") {
        const token = jwt.sign(
            { username },
            SECRET_KEY,
            { expiresIn: '1h' }
        );

        return res.json({ token });
    }

    res.status(401).json({ message: "Invalid credentials" });
});

3. Verify JWT (Middleware)


function authenticateToken(req, res, next) {
    const authHeader = req.headers['authorization'];
    const token = authHeader && authHeader.split(' ')[1];

    if (!token) return res.sendStatus(401);

    jwt.verify(token, SECRET_KEY, (err, user) => {
        if (err) return res.sendStatus(403);

        req.user = user;
        next();
    });
}

4. Protect routes


app.get('/protected', authenticateToken, (req, res) => {
    res.json({ message: "Protected data", user: req.user });
});

5. Start server


app.listen(3000, () => console.log("Server running"));

🔐 How client sends token


Authorization: Bearer <your_token>

⚠️ Best Practices

  1. Use HTTPS only
  2. Keep secret key safe (env variables)
  3. Use short expiration time
  4. Store tokens in HTTP-only cookies (safer than localStorage)
  5. Avoid storing sensitive data in payload
  6. Use refresh tokens for long sessions

🧩 Other Languages (quick mention)

  • Python → PyJWT

  • .NET → built-in JWT middleware

🚀 Simple mental model

Think of JWT like a signed ID card:

  • Payload = your info
  • Signature = tamper-proof seal
  • Server checks the seal, not the database every time
