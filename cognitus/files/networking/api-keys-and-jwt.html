API keys and JWTs (JSON Web Tokens)

At a high level, API keys and JWTs (JSON Web Tokens) are both used to control access to APIs—but they solve different problems and operate at different levels of security.

🔑 API Key (Simple Identification)

An API key is basically a static secret string that identifies the application making the request.

How it works:

  • Client sends a request with a key (e.g., in headers: x-api-key)
  • Server checks if the key is valid
  • If valid → request is allowed

Characteristics:

  • Simple and fast
  • Usually does NOT represent a user, just the app
  • No built-in expiration (unless you implement it)
  • No payload (just a string)
  • Easily leaked if not protected

Example:


GET /api/data
x-api-key: abc123xyz

👉 Think of API keys like a door key to a building—anyone who has it can enter.

🪪 JWT (Authentication + Authorization)

A JWT (JSON Web Token) is a signed token containing data (claims) about a user or entity.

How it works:

  1. User logs in
  2. Server generates a JWT
  3. Client sends JWT in requests (usually Authorization: Bearer )
  4. Server verifies signature and reads data

Structure:

  • Header
  • Payload (e.g., user ID, roles)
  • Signature (to prevent tampering)

Characteristics:

  • Represents a specific user/session
  • Contains data (claims) inside
  • Has expiration (exp)
  • Cryptographically signed (secure if done right)
  • Stateless (no DB lookup required if designed properly)

Example:


GET /api/profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

👉 Think of JWT like a passport—it proves identity and contains information about you.

⚖️ Key Differences

Feature API Key JWT
Purpose Identify application Authenticate user
Contains data ❌ No ✅ Yes (claims)
Expiration ❌ Usually none ✅ Built-in (exp)
Security level Low–Medium Medium–High
Use case Public APIs, server-to-server User login/auth sessions
Stateless ✅ Yes ✅ Yes
Tamper protection ❌ No ✅ Yes (signature)

🧠 When to Use Each

Use API Keys when:

  • You just need to identify which app is calling
  • Public APIs (e.g., rate limiting, quota tracking)
  • Internal service-to-service communication (simple cases)

Use JWT when:

  • You need user authentication
  • You need roles/permissions (e.g., admin vs user)
  • You want stateless auth in systems like your AWS Lambda + API Gateway setup

🔥 Real-World Combo (What pros actually do)

In production systems (like your serverless setup), it's common to use both:

  • API Key → identifies the client/app
  • JWT → identifies the user

Example:


Authorization: Bearer <jwt>
x-api-key: <api-key>
