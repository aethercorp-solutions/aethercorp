JavaScript module system

A JavaScript module system is the way JavaScript organizes code into separate, reusable files (modules) and controls how those files import and export functionality.

Instead of putting everything in one huge file, modules let you split your app into smaller pieces—each with its own responsibility.

🧠 Core Idea

A module is just a file with its own scope that can:

  • export values (functions, variables, classes)
  • import values from other modules

🔑 Why module systems exist

Before modules, JavaScript had problems:

  • Global scope pollution (var x everywhere)
  • No standard way to reuse code
  • Hard to maintain large apps

Module systems solve this by:

  • Encapsulating code
  • Making dependencies explicit
  • Improving scalability

⚙️ Main JavaScript Module Systems

1. CommonJS (CJS) — Node.js classic

Used in Node.js (especially older versions)


// export
module.exports = function greet() {
  console.log("Hello");
};

// import
const greet = require('./greet');

Characteristics:

  • Synchronous (loads modules instantly)
  • Good for backend (like your AWS Lambda setup)
  • Uses require and module.exports

2. ES Modules (ESM) — Modern standard

Native in modern JavaScript (browser + Node.js)


// export
export function greet() {
  console.log("Hello");
}

// import
import { greet } from './greet.js';

Characteristics:

  • Asynchronous (better for browsers)
  • Static structure (analyzable at build time)
  • Official JS standard

3. AMD (Asynchronous Module Definition)

Used mainly in browsers (older systems like RequireJS)


define(['dep'], function(dep) {
    return function() {};
});

Characteristics:

  • Async loading
  • Mostly obsolete today

4. UMD (Universal Module Definition)

Hybrid system that works with both CommonJS and AMD


(function (root, factory) {
    if (typeof module === 'object') {
        module.exports = factory();
    } else {
        root.myLib = factory();
    }
    })(this, function () {
    return {};
});

Use case:

  • Libraries that must work everywhere

Standard No Yes

🧩 In your context (AWS + Node.js)

Since you're working with:

  • AWS Lambda
  • Node.js backend
  • Express / serverless

You're likely using:

  • CommonJS (require) OR
  • Moving toward ES Modules (import)

👉 Example in your middleware:


const jwt = require('jsonwebtoken'); // CommonJS

Modern equivalent:


import jwt from 'jsonwebtoken'; // ESM

🧠 Simple Analogy

Think of modules like microservices inside your codebase:

  • Each file = one service
  • export = API
  • import = calling another service
