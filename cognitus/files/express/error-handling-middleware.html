Error Handling Middleware

Here’s a clean and practical Error Handling Middleware example in Node.js using Express.js.

🔧 Basic Error Handling Middleware

In Express, error-handling middleware always has 4 parameters:


// error.middleware.js
module.exports = (err, req, res, next) => {
  console.error(err.stack);

  res.status(err.status || 500).json({
    success: false,
    message: err.message || 'Internal Server Error',
  });
};

That’s a clean and valid Express error-handling middleware. Let’s break down exactly what your module.exports is doing here and why this function looks a bit different from normal middleware.

🔹 What you exported


module.exports = (err, req, res, next) => { ... }

You are exporting a function, not an object.

So when you import it:


const errorMiddleware = require('./error.middleware');

👉 errorMiddleware is directly that function.

🔹 Why 4 parameters?

This is the key part.

In Express.js, middleware types are identified by parameters:

Normal middleware:


(req, res, next)

Error middleware:


(err, req, res, next)

👉 The 4 arguments tell Express:

“This middleware handles errors”

🔹 How it works in flow

Example:


app.get('/test', (req, res) => {
    throw new Error('Something broke!');
});

Express will skip normal middleware and go straight to your error middleware:


app.use(require('./error.middleware'));

🔹 Your code explained


console.error(err.stack);
  • Logs full error trace (good for debugging)

res.status(err.status || 500)
  • Uses custom status if provided
  • Defaults to 500

.json({
    success: false,
    message: err.message || 'Internal Server Error',
});
  • Clean API response format

🔹 Real-world improvement (important)

Right now, your middleware assumes err.status exists. In many cases, it won’t.

You can make it more robust:


module.exports = (err, req, res, next) => {
    console.error(err.stack);
    
    const statusCode = err.status || err.statusCode || 500;
    
    res.status(statusCode).json({
        success: false,
        message: err.message || 'Internal Server Error',
    });
};

🔹 Even better (production-safe)

Avoid leaking internal errors in production:


module.exports = (err, req, res, next) => {
    console.error(err.stack);
    
    const statusCode = err.status || err.statusCode || 500;
    
    res.status(statusCode).json({
        success: false,
        message:
        process.env.NODE_ENV === 'production'
            ? 'Internal Server Error'
            : err.message,
    });
};

🔹 How this fits your AWS Lambda setup

Since you're using:


module.exports.handler = serverless(app);

Your Express middleware (including this one) still works — it's just wrapped by the serverless-http adapter so AWS Lambda can run it.

🔹 Common mistake to avoid

Make sure this is last middleware:


app.use(errorMiddleware); // must be last

If you put it before routes → it won’t catch errors properly.
