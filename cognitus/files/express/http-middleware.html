HTTP

HTTP stands for HyperText Transfer Protocol. It’s the fundamental protocol used for communication on the web—basically how your browser talks to a server.

Simple idea

When you open a website:

  1. Your browser sends an HTTP request to a server
  2. The server sends back an HTTP response
  3. The browser renders the result (HTML, JSON, images, etc.)

Example (real-world flow)

If you visit:


https://example.com

Your browser sends something like:


GET / HTTP/1.1
Host: example.com

Server responds:


HTTP/1.1 200 OK
Content-Type: text/html

<html>...</html>

Key concepts

1. Request Methods

These define what you want to do:

  • GET → retrieve data
  • POST → send/create data
  • PUT → update data
  • DELETE → remove data

2. Status Codes

Server responses include status:

  • 200 → success
  • 404 → not found
  • 500 → server error

3. Stateless

HTTP is stateless, meaning:

  • Each request is independent
  • Server doesn’t remember previous requests
  • That’s why we use things like cookies or JWT tokens (you’ve used this in your Node.js middleware)

HTTP vs HTTPS

  • HTTP → not secure (data is plain text)
  • HTTPS → secure (encrypted using SSL/TLS)

In your context (Node.js / AWS)

When you build:

  • Express apps → handle HTTP requests (req, res)
  • AWS Lambda + API Gateway → API Gateway receives HTTP requests, then triggers your Lambda

Quick analogy

Think of HTTP like a messenger system:

  • You (client) send a request letter
  • Server reads it and sends back a reply
  • Each letter is separate (stateless)

🔗 1. Big Picture Flow

Great—let’s connect HTTP → Express → AWS Lambda in a way that matches what you’re building.

When a client (mobile app / browser) calls your API:


Client (React Native)
        ↓ HTTP Request
API Gateway (AWS)
        ↓
Lambda Handler
        ↓
Express App (middleware + routes)
        ↓
Response back to client

🧠 2. HTTP in Express (what you already use)

In a normal Express app:


app.get('/users', (req, res) => {
    res.json({ users: [] });
});

What happens under the hood:

  • req = HTTP Request
    • headers, body, method, URL
  • res = HTTP Response
    • status code, JSON, headers

Example:


req.method   // GET
req.headers  // authorization, content-type
req.body     // JSON payload

⚙️ 3. Middleware = HTTP Pipeline

Your middleware is literally processing the HTTP request step-by-step.

Example (like what you wrote):


module.exports = function (req, res, next) {
    const token = req.headers['authorization'];
    
    if (!token) {
        return res.status(401).json({ message: 'No token' });
    }
    
    next();
};

Flow:


HTTP Request
   ↓
Auth Middleware
   ↓
Route Handler
   ↓
HTTP Response

So middleware = interceptors of HTTP requests

☁️ 4. Now add AWS Lambda

Lambda does NOT natively understand Express.

It receives an event like this:


{
    httpMethod: "GET",
    path: "/users",
    headers: {...},
    body: "..."
}

🔁 5. What serverless-http does


const serverless = require('serverless-http');
const app = require('./app');

module.exports.handler = serverless(app);

Translation layer:

HTTP Concept API Gateway Express
Method event.httpMethod req.method
Headers event.headers req.headers
Body event.body req.body
Response eturn object res.json()

👉 serverless-http converts:


API Gateway event → Express req/res

🔄 6. Full Flow (REAL execution)

Step-by-step when your React Native app calls API:

1. Client sends HTTP request


GET /users
Authorization: Bearer xxx

2. API Gateway receives it

Transforms into:


event = {
    httpMethod: "GET",
    path: "/users",
    headers: { authorization: "Bearer xxx" }
}

3. Lambda handler runs


module.exports.handler = serverless(app);

4. Express kicks in

Middleware runs:


authMiddleware → verifies JWT

Route runs:


app.get('/users', ...)

5. Response sent back


{
    "users": []
}

Then:


Express → serverless-http → API Gateway → Client

⚠️ Important Insight

Even though you use Express…

👉 In serverless:

  • There is no real server running
  • Each HTTP request = new Lambda execution

So:

  • No persistent memory
  • No session (unless external like DB/Redis)
