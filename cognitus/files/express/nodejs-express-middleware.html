Node.js + Express.js app, middleware

In a Node.js + Express.js app, middleware is basically a function that runs between the request and the response. It can read, modify, validate, or stop a request before it reaches your main logic (controllers/routes).

Think of it like a pipeline:


Request → Middleware → Middleware → Route Handler → Response

🔧 Common Middleware Use Cases

1. Authentication (Very common in your setup)

Validate JWT or session before allowing access.


const jwt = require('jsonwebtoken');

function authMiddleware(req, res, next) {
  const token = req.headers.authorization;

  if (!token) {
    return res.status(401).json({ message: 'No token' });
  }

  try {
    const decoded = jwt.verify(token.split(' ')[1], process.env.JWT_SECRET);
    req.user = decoded;
    next(); // continue
  } catch (err) {
    return res.status(401).json({ message: 'Invalid token' });
  }
}

👉 Use case: Protect APIs like /profile, /orders

2. Logging Requests

Track every incoming request (useful for debugging or analytics).


function logger(req, res, next) {
    console.log(`${req.method} ${req.url}`);
    next();
}

👉 Use case: Monitoring API usage in AWS Lambda logs

3. Input Validation

Ensure request body is correct before hitting business logic.


function validateUser(req, res, next) {
    const { email, password } = req.body;
    
    if (!email || !password) {
        return res.status(400).json({ message: 'Missing fields' });
    }
    
    next();
}

👉 Use case: Prevent bad data from reaching MongoDB

4. Error Handling Middleware

Centralized error management.


function errorHandler(err, req, res, next) {
    console.error(err.stack);
    res.status(500).json({ message: 'Something went wrong' });
}

👉 Use case: Avoid repeating try/catch everywhere

5. CORS Handling

Allow cross-origin requests (important for React Native / frontend apps).


const cors = require('cors');
app.use(cors());

👉 Use case: Your React Native app calling your AWS API

6. Rate Limiting / Security

Prevent abuse (DDoS, brute force).


const rateLimit = require('express-rate-limit');

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100
});

app.use(limiter);

👉 Use case: Protect login endpoints

7. Parsing Request Body

Convert incoming JSON to usable object.


app.use(express.json());

👉 Use case: Read req.body properly

8. Role-Based Authorization

Restrict access by user roles.


function isAdmin(req, res, next) {
    if (req.user.role !== 'admin') {
        return res.status(403).json({ message: 'Forbidden' });
    }
    next();
}

👉 Use case: Admin-only APIs

🧠 In Your AWS Serverless Setup

Since you're using Lambda + API Gateway:

  • Middleware is not built-in like Express
  • You either:
    • Use Express + serverless-http (what you're doing)
    • Or manually chain middleware functions

Example in your setup:


app.get('/protected', authMiddleware, (req, res) => {
    res.json({ user: req.user });
});

⚡ Key Insight

Middleware helps you:

  • Reuse logic
  • Keep routes clean
  • Separate concerns (auth, validation, logging)

Without middleware, your route becomes messy:


// BAD (no middleware)
app.get('/profile', (req, res) => {
  // auth logic
  // validation
  // logging
  // business logic
});

🧠 Core Idea

At the JavaScript level, middleware is nothing magical—it’s just functions calling other functions in sequence, using closures and callbacks.

A middleware is just a function like this:


function middleware(req, res, next) {
    // do something
    next(); // pass control
}

The important part is next() → it tells JavaScript:

“Call the next function in the chain.”

🔁 What’s really happening under the hood

Express (from Express.js running on Node.js) stores middleware in an array (stack):


const stack = [mw1, mw2, mw3];

Then it executes them one by one.

⚙️ Simplified Engine (Pure JavaScript)

Here’s a minimal version of how middleware works internally:


function runMiddleware(stack, req, res) {
    let index = 0;
    
    function next() {
        const fn = stack[index++];
        if (fn) {
        fn(req, res, next);
        }
    }
    
    next(); // start chain
}

▶ Example usage:


const mw1 = (req, res, next) => {
    console.log('Middleware 1');
    req.name = 'Paul';
    next();
};

const mw2 = (req, res, next) => {
    console.log('Middleware 2');
    req.age = 30;
    next();
};
    
const handler = (req, res) => {
    console.log('Final Handler', req);
};

runMiddleware([mw1, mw2, handler], {}, {});

🧾 Output:


Middleware 1
Middleware 2
Final Handler { name: 'Paul', age: 30 }

🔄 Execution Flow (Step-by-step)

  1. next() is called
  2. First middleware runs
  3. It calls next()
  4. Second middleware runs
  5. It calls next()
  6. Final handler runs
  7. No more functions → stop

❗ Important Behavior

1. If next() is NOT called → chain stops


const mw = (req, res, next) => {
    console.log('Stops here');
    // no next()
};

👉 Everything after this middleware will never execute

2. Middleware can END the response


const auth = (req, res, next) => {
    if (!req.user) {
        return res.send('Unauthorized'); // stops chain
    }
    next();
};

👉 No next() → request lifecycle ends

3. Middleware can MODIFY shared objects

All middleware share the same req and res references:


req.user = { id: 1 };

👉 Next middleware can use it

🧩 Why this works (JS concepts involved)

Middleware relies on:

  • Closures → next() remembers index
  • First-class functions → functions passed as arguments
  • Mutable objects → req and res shared across chain
  • Control flow delegation → each function decides when to continue

⚡ Async Middleware (Real-world)


const mw = async (req, res, next) => {
    await someAsyncTask();
    next();
};

⚠️ If you forget next() or await, you can break the flow or create bugs.

🔥 Key Insight

Middleware is basically:

A controlled function pipeline where each function decides whether to pass control forward.
