Understanding custom domains deeply will help you design production-ready systems

Since you're working heavily with AWS (Lambda, API Gateway, Cognito, etc.), understanding custom domains deeply will help you design production-ready systems.

Let’s break this down clearly and practically.

🔎 Big Picture: What Is a Custom Domain?

Instead of:

https://d3k2abcxyz.cloudfront.net

You want:

https://api.trycastanet.com

To make that work in AWS using:

  • Amazon Web Services
  • Amazon S3
  • Amazon CloudFront
  • Amazon Route 53

You must connect 4 systems:

  1. DNS (Route 53)
  2. CDN (CloudFront)
  3. SSL Certificate (ACM)
  4. Origin (S3 bucket)

🧠 How It Works (Conceptual Flow)

When a user types:

https://app.trycastanet.com

Here’s what happens step by step:

  1. Browser asks DNS:

    👉 “Where is app.trycastanet.com?”

  2. Route 53 replies:

    👉 “Go to CloudFront distribution XYZ.”

  3. Browser connects to CloudFront over HTTPS.
  4. CloudFront:
    • Uses SSL certificate (ACM)
    • Fetches content from S3 bucket (origin)
    • Caches and returns it
  5. User sees your site.

🧱 Step-by-Step Setup (Detailed)

STEP 1️⃣: Register or Have a Domain

Example:

trycastanet.com

You can:

  • Register inside Route 53
  • Or use external registrar (GoDaddy, Namecheap, etc.)

STEP 2️⃣: Create S3 Bucket

Create bucket:

app.trycastanet.com

Important settings:

Enable:

  • Static website hosting (optional but recommended for web apps)
  • Block public access → DISABLED (if directly public)

    OR

  • Keep private (recommended) and use CloudFront OAC

Upload:

  • index.html
  • JS/CSS
  • React build files

STEP 3️⃣: Create SSL Certificate (VERY IMPORTANT)

Go to:

👉 AWS Certificate Manager (ACM)

Request certificate for:

app.trycastanet.com

Validation:

  • DNS validation (recommended)
  • Route 53 auto-creates validation record

⚠ IMPORTANT:

Certificate MUST be created in:

us-east-1

Why?

Because CloudFront only reads certificates from us-east-1.

STEP 4️⃣: Create CloudFront Distribution

Origin:

S3 bucket

Settings:

  • Viewer Protocol Policy → Redirect HTTP to HTTPS
  • Alternate Domain Name (CNAME):

    app.trycastanet.com

  • Attach ACM certificate

Security (Recommended):

  • Use Origin Access Control (OAC)
  • Keep S3 bucket private
  • Allow only CloudFront to access bucket

Now CloudFront gives you:

d123abcd.cloudfront.net

STEP 5️⃣: Configure Route 53

Go to:

Hosted Zones → trycastanet.com

Create Record:

Type:

A record

Enable:

Alias = Yes

Alias target:

→ Select CloudFront distribution

Now:


app.trycastanet.com
→ points to CloudFront
→ which pulls from S3

🔐 HTTPS & SSL Flow Explained

When user connects:

  1. Browser sends TLS handshake request
  2. CloudFront presents ACM certificate
  3. Browser verifies:
    • Certificate validity
    • Domain match
    • Trusted CA
  4. Secure encrypted connection established

Without ACM certificate:

  • HTTPS won’t work
  • Browser shows "Not Secure"

🚀 What CloudFront Actually Does

CloudFront is:

  • CDN (global edge caching)
  • SSL terminator
  • DDoS protection layer
  • Performance optimizer

If user is in Philippines:

→ Request served from nearest edge location

→ Faster than direct S3

🔁 Full Request Lifecycle

Let’s simulate:

User in Diliman opens:

https://app.trycastanet.com

1️⃣ DNS Resolution

Route 53 returns CloudFront IP

2️⃣ TLS Handshake

CloudFront uses ACM certificate

3️⃣ Cache Check

CloudFront checks edge cache

If cached:

→ Return instantly

If not cached:

→ Fetch from S3

1️⃣ What Is a Custom Domain (Architectural View)

A custom domain is when users access your system using:


https://yourdomain.com

Instead of:


https://random-id.cloudfront.net
https://bucket-name.s3.amazonaws.com
https://xyz.execute-api.us-east-1.amazonaws.com

A custom domain gives you:

✅ Brand trust

✅ HTTPS security

✅ SEO benefits

✅ Multi-environment control (dev/staging/prod)

✅ TLS termination control

✅ Traffic routing flexibility

2️⃣ Core AWS Services Involved

In most production setups, these services work together:

  • DNS → Amazon Route 53
  • SSL/TLS → AWS Certificate Manager
  • CDN → Amazon CloudFront
  • Static hosting → Amazon S3
  • API hosting → Amazon API Gateway

Each plays a specific role in the request lifecycle.

3️⃣ End-to-End Request Flow (Production Example)

Let’s say a user visits:


https://app.yourdomain.com

Here’s what happens:

Step 1: DNS Resolution

DNS is managed by Route 53.

It resolves:


app.yourdomain.com → CloudFront Distribution

Technically:

  • You create an A Record (Alias) in Route 53.
  • It points to CloudFront.

Step 2: TLS/SSL Handshake

CloudFront needs a certificate.

You create it in:

👉 AWS Certificate Manager

Important:

  • The certificate must be created in us-east-1 for CloudFront.
  • You validate ownership via DNS or email.
  • Route 53 can auto-create DNS validation records.

Step 3: CloudFront Behavior

CloudFront then:

  • Accepts HTTPS request
  • Terminates TLS
  • Caches content
  • Forwards to origin

The origin could be:

  • S3 (static frontend)
  • API Gateway (backend API)
  • Load Balancer
  • EC2

Step 4: Origin Response

Example:

  • S3 returns React build files
  • API Gateway forwards to Lambda
  • CloudFront caches response

Then response goes back to the user.

4️⃣ Production Architecture Patterns

🔹 Static Website (React / Next.js Static Export)

User

→ Route 53

→ CloudFront

→ S3

Best practice:

  • ❌ Do NOT use S3 website endpoint in production
  • ✅ Use S3 private bucket
  • ✅ Use CloudFront Origin Access Control (OAC)

This prevents direct S3 access.

🔹 API Backend (Serverless)

User

→ Route 53

→ CloudFront (optional)

→ API Gateway

→ Lambda

API Gateway supports custom domains directly too.

🔹 Full Production Setup


yourdomain.com
│
├── app.yourdomain.com → CloudFront → S3
├── api.yourdomain.com → API Gateway → Lambda
└── admin.yourdomain.com → ALB → EC2

This allows microservice separation.

6️⃣ Common Mistakes in Custom Domain Setup

❌ Creating certificate in wrong region

CloudFront requires ACM cert in us-east-1.

❌ Using S3 public bucket

Use OAC instead.

❌ Not using Alias records

Use Route 53 Alias → avoids extra DNS lookup cost.

❌ Mixing HTTP and HTTPS

Force redirect to HTTPS via CloudFront behavior.

7️⃣ Deep DNS Concepts (Important for Architects)

A Record vs CNAME

  • A record → IP address
  • CNAME → another domain
  • Route 53 Alias → AWS resource (preferred)

Propagation Time

DNS changes:

  • TTL affects caching
  • Can take minutes to hours globally
