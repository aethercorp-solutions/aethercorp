Internet Gateway (IGW)

An Internet Gateway (IGW) is an AWS VPC component that lets resources inside your Virtual Private Cloud (VPC) talk to the public internet — both outbound and inbound.

Think of it as the front door of your VPC to the internet 🌐🚪

What an Internet Gateway actually does

An IGW does two critical things:

1️⃣ Enables outbound internet access

It allows instances in a VPC (like EC2) to send traffic to the internet.

Example:

  • EC2 → Google.com ✅

2️⃣ Enables inbound internet access

It allows traffic from the internet to reach your VPC if routing + security rules allow it.

Example:

  • User → EC2 public IP ✅ (only if security groups & NACLs permit)

When an IGW is required

You need an IGW if:

  • Your EC2 instance has a public IPv4 address or Elastic IP
  • Your route table has a route like:

0.0.0.0/0 → Internet Gateway

Without an IGW:

❌ No internet access — even with a public IP

How IGW fits into the VPC flow


EC2 (public subnet)
   ↓
Route Table (0.0.0.0/0 → IGW)
   ↓
Internet Gateway
   ↓
Internet

Key properties of an Internet Gateway

  • ✔️ Horizontally scalable (no bandwidth limits to configure)
  • ✔️ Highly available (AWS-managed)
  • ✔️ One IGW per VPC
  • ✔️ Must be attached to a VPC
  • ❌ Does NOT provide security (that’s Security Groups & NACLs)

Public subnet vs Private subnet (important!)

Subnet Type Route to IGW Internet Access
Public Subnet Yes ✅ Direct
Private Subnet No ❌ Direct (uses NAT instead)

Private subnets use NAT Gateway, not IGW, for outbound internet.

Common confusion cleared up

❓ IGW vs NAT Gateway

IGW NAT Gateway
Inbound + Outbound Outbound only
Public subnets Private subnets
Needs public IP No public IP on private instances

IGW vs NAT Gateway in a serverless/Lambda setup

Great question — this is one of the most confusing parts of serverless on AWS, so let’s make it click 👌

IGW vs NAT Gateway in a Lambda (serverless) setup

(fast truth)

Component Used by Lambda? Purpose
Internet Gateway (IGW) ❌ Directly Lets public subnets talk to the internet
NAT Gateway ✅ Lets private subnets (Lambda) access the internet

👉 Lambda never talks to the internet through an IGW directly.

How Lambda networking actually works

Case 1️⃣ Lambda NOT in a VPC (default)


Lambda
→ AWS-managed internet access
→ Internet
  • ✅ Can access the internet automatically
  • ❌ Cannot access VPC resources (RDS, EC2, Redis)
  • 💡 Best for pure APIs, webhooks, 3rd-party calls

No IGW. No NAT. AWS handles everything.

Case 2️⃣ Lambda IN a VPC (this is where things break)


Lambda (private subnet)
→ ? → Internet ❌

The moment you attach a Lambda to a VPC:

  • ❌ Internet access is lost
  • ❌ Even axios, fetch, npm install, Stripe calls fail

Why?

👉 Private subnets cannot use IGW

Correct setup: Lambda + NAT Gateway

Architecture


Lambda (private subnet)
→ Route Table (0.0.0.0/0 → NAT Gateway)
→ NAT Gateway (public subnet)
→ Internet Gateway
→ Internet

What each piece does

🔹 NAT Gateway

  • Used by private subnets
  • Allows outbound-only internet access
  • Lambda uses this, not IGW

🔹 Internet Gateway

  • Attached to the VPC
  • Used by the NAT Gateway, not Lambda

Why Lambda must be in a private subnet

  • Lambda never accepts inbound connections
  • It only makes outbound calls
  • Public IPs are useless for Lambda
  • Security best practice

Common mistakes (🔥 very common)

❌ “I attached an IGW but Lambda still can’t access internet”

✔️ Correct — Lambda can’t use IGW directly

❌ “I put Lambda in a public subnet”

✔️ Doesn’t matter — Lambda still has no public IP

❌ “I don’t want NAT to save money”

✔️ Then don’t put Lambda in a VPC

Cost reality check 💸

Option Cost
Lambda NOT in VPC 💸 $0
NAT Gateway 💸 ~$32/month + data
VPC Endpoints 💸 Cheaper (for AWS services only)

💡 Use VPC Endpoints (S3, DynamoDB, Secrets Manager) to avoid NAT costs.

Best practice summary

✅ Use IGW when:

  • You have EC2 in public subnets
  • You need inbound internet traffic

✅ Use NAT Gateway when:

  • Lambda is in a VPC
  • Lambda needs outbound internet access

❌ Do NOT use anything when:

  • Lambda does not need VPC access

One-sentence rule to remember

IGW is for public resources.

NAT Gateway is for private resources like Lambda.

Why Lambda in a VPC loses internet without NAT/IGW

This happens because of how AWS isolates VPC networking, and it’s 100% by design (not a bug). Let’s walk it step-by-step so it really sticks.

Short answer (the core reason)

When you put a Lambda in a VPC, AWS removes its default internet access. From that point on, the Lambda behaves like a private subnet resource and needs NAT + IGW to reach the internet.

What Lambda has BEFORE VPC attachment

Lambda not in a VPC (default)


Lambda
→ AWS-managed network
→ Internet
  • AWS gives Lambda implicit outbound internet
  • No routing tables
  • No subnets
  • No IGW / NAT needed

This is why Stripe, SendGrid, Google APIs “just work”.

What changes the moment you attach a VPC

When you attach a Lambda to a VPC, AWS does three important things:

1️⃣ Lambda gets ENIs inside your VPC

  • Elastic Network Interfaces are created in your subnets
  • Lambda now lives inside your VPC network
  • It follows your route tables, not AWS’s

2️⃣ AWS removes the default internet path

AWS cuts off:


Lambda → AWS-managed internet

Why?

👉 Security + isolation

  • Your VPC must be fully controlled by you
  • AWS will not assume you want internet access

3️⃣ Private subnets cannot talk to IGW

Even if:

  • Subnet route = 0.0.0.0/0 → IGW
  • Lambda is in that subnet

❌ Lambda still cannot use IGW

Because:

  • Lambda never gets a public IP
  • IGW requires a public IP on the source

Why IGW alone does NOT fix it

IGW works only if:

  • Resource has a public IP
  • Resource is in a public subnet

Lambda:

  • ❌ No public IP
  • ❌ No inbound connections
  • ❌ No direct IGW support

So traffic dies here:


Lambda → IGW ❌

Why NAT Gateway fixes everything

Correct path


Lambda (private subnet)
→ Route Table (0.0.0.0/0 → NAT Gateway)
→ NAT Gateway (public subnet)
→ Internet Gateway
→ Internet

Why this works

  • NAT Gateway has a public IP
  • NAT translates Lambda’s private IP → public IP
  • IGW allows NAT to reach internet

Security reason AWS designed it this way 🔒

If AWS allowed:


Lambda in VPC → IGW

Then:

  • Lambdas could become accidentally public
  • VPC isolation would be meaningless
  • Massive security foot-guns

AWS forces you to explicitly choose internet access.

Visual summary

Lambda State Internet Access Why
Not in VPC ✅ Yes AWS-managed networking
In VPC, no NAT ❌ No No route + no public IP
In VPC + NAT ✅ Yes NAT provides public egress

Mental model to remember 🧠

Attaching a Lambda to a VPC is like locking it inside a private room. If you want it to call the outside world, you must build a controlled exit (NAT).

Pro tip (important for your setups)

If your Lambda only needs:

  • DynamoDB
  • S3
  • Secrets Manager
  • SQS / SNS

👉 Use VPC Endpoints instead of NAT

Cheaper, faster, more secure.

How this applies to your AWS API Gateway + Lambda setup you’ve been working on

Alright, let’s map this directly to your typical API Gateway + Lambda setup, no theory fluff — just what actually happens and why it matters.

I’ll walk it in decision form, because this is usually where things break in real projects.

Your common setup (API Gateway → Lambda)


Client (Web / Mobile)
→ API Gateway (public)
→ Lambda
→ (DB / 3rd-party / AWS services)

API Gateway is always public.

Lambda may or may not be in a VPC — that’s the key fork.

Scenario 1️⃣ Lambda NOT in a VPC (simplest & cheapest)

Architecture


Internet
→ API Gateway
→ Lambda (no VPC)
→ Internet / AWS services

What works automatically

  • ✅ Stripe / PayPal / SendGrid
  • ✅ External REST APIs
  • ✅ npm packages calling the internet
  • ✅ DynamoDB, S3, Secrets Manager

What does NOT work

  • ❌ RDS
  • ❌ EC2
  • ❌ ElastiCache
  • ❌ Private services

When this is best

  • Pure REST API
  • Serverless-first design
  • No legacy DB
  • Cost-sensitive (no NAT)

👉 This is the best default unless you need VPC access.

Scenario 2️⃣ Lambda IN a VPC (most production apps)

Architecture


Internet
→ API Gateway
→ Lambda (private subnet)
→ RDS / MongoDB / Redis

What suddenly breaks

  • ❌ Stripe
  • ❌ Firebase
  • ❌ Google APIs
  • ❌ Any external HTTPS call

This is the “why is axios failing??” moment 😅

Why API Gateway still works (important!)

People ask this a lot:

“If Lambda has no internet, why can API Gateway reach it?”

Because:

  • API Gateway runs outside your VPC
  • AWS connects API Gateway → Lambda via internal AWS networking
  • This path does NOT use IGW or NAT

So this works even with zero internet:


API Gateway → Lambda ✅

Correct production setup (what you actually want)

Full architecture


Client
→ API Gateway
→ Lambda (private subnet)
   → RDS / MongoDB
   → NAT Gateway
   → Internet (Stripe, Firebase, etc.)

Required components

✔️ VPC

✔️ Private subnets (for Lambda)

✔️ Public subnet (for NAT)

✔️ NAT Gateway

✔️ Internet Gateway (attached to VPC)

Route tables (this is the critical part)

Private subnet (Lambda)


0.0.0.0/0 → NAT Gateway

Public subnet (NAT)


0.0.0.0/0 → Internet Gateway

Miss any one of these → internet is dead.

Cost-optimized version (VERY important)

Instead of sending all traffic through NAT:

Use VPC Endpoints for AWS services

  • DynamoDB
  • S3
  • Secrets Manager
  • SQS / SNS

Then NAT is used only for:

  • Stripe
  • Google
  • Firebase
  • External APIs

💡 This can cut NAT costs by 60–80%.

Your mental checklist when debugging 🔍

If Lambda can’t reach the internet, check in this order:

  1. Is Lambda attached to a VPC?
  2. Is it in a private subnet?
  3. Does the private subnet route to NAT Gateway?
  4. Is NAT in a public subnet?
  5. Does public subnet route to IGW?
  6. Are NACLs allowing outbound traffic?

99% of issues are #3 or #4.
