Just raw AWS services, how they actually work under the hood. Let’s strip the magic away. 🧱⚙️

I’ll explain this like you’re building it from scratch, the way AWS itself sees it.

1. What “bare AWS services” really means

When you remove wrappers, you are responsible for:

  • Creating every resource
  • Wiring permissions explicitly
  • Handling deployments manually
  • Managing environments (dev/prod) yourself
  • Debugging IAM 😈

Nothing is “automatic”.

2. Core building blocks (mental model)

Think of AWS like LEGO pieces. Wrappers just snap them together for you.

Bare AWS = you snap them together by hand.


Client → API Gateway → Lambda → (DB / S3 / other services)
        ↑
        IAM

Everything revolves around IAM, networking, and service-to-service trust.

3. IAM (Identity & Access Management) — the spine of AWS

3.1 IAM Users (humans)

Used for:

  • Console login
  • CLI access (aws configure)

Contains:

  • Access Key ID
  • Secret Access Key

Example policy (read-only S3):


{
    "Version": "2012-10-17",
    "Statement": [
        {
        "Effect": "Allow",
        "Action": ["s3:GetObject"],
        "Resource": "arn:aws:s3:::my-bucket/*"
        }
    ]
}

⚠️ Best practice:

  • Humans → IAM Users
  • Services → IAM Roles

3.2 IAM Roles (services)

Services assume roles.

Lambda never uses access keys.

It assumes a role at runtime.

Example Lambda role trust policy:


{
    "Version": "2012-10-17",
    "Statement": [
        {
        "Effect": "Allow",
        "Principal": { "Service": "lambda.amazonaws.com" },
        "Action": "sts:AssumeRole"
        }
    ]
}

Then attach permissions:


{
    "Effect": "Allow",
    "Action": ["dynamodb:PutItem"],
    "Resource": "arn:aws:dynamodb:ap-southeast-1:*:table/Users"
}

4. Networking — VPC (most people don’t really understand this)

4.1 VPC

A private network inside AWS.

  • CIDR block: 10.0.0.0/16
  • You control traffic, routing, isolation

Default VPC exists, but serious systems don’t rely on it.

4.2 Subnets

Split VPC into zones:

  • Public subnet (internet access)
  • Private subnet (no internet)

Example:


10.0.1.0/24 → public
10.0.2.0/24 → private

4.3 Internet Gateway + Route Tables

Public subnet needs:

  • Internet Gateway attached to VPC
  • Route 0.0.0.0/0 → igw

Without this: no internet

4.4 Security Groups (stateful firewall)

Attached to:

  • EC2
  • RDS
  • Load Balancers

Example:

  • Allow HTTP 80
  • Allow HTTPS 443

Inbound:
  TCP 443 from 0.0.0.0/0
Outbound:
  All traffic

5. Compute services (no abstraction)

5.1 EC2 — raw virtual machines

You manage:

  • OS
  • Updates
  • Security patches
  • Scaling
  • Failures

Launch flow:

  1. Choose AMI
  2. Choose instance type
  3. Attach security group
  4. Attach IAM role
  5. SSH in

You become sysadmin again.

5.2 Lambda — managed compute, still bare wiring

What AWS gives:

  • Runtime
  • Scaling
  • Execution environment

What you must do:

  • Create role
  • Set permissions
  • Deploy code
  • Define triggers

Create function:


aws lambda create-function \
  --function-name hello \
  --runtime nodejs18.x \
  --handler index.handler \
  --role arn:aws:iam::123:role/lambda-role \
  --zip-file fileb://function.zip

6. API Gateway (no magic)

6.1 REST API vs HTTP API

  • REST API → full control, expensive
  • HTTP API → lighter, cheaper

Bare setup requires:

  1. Create API
  2. Create resource (/users)
  3. Create method (GET)
  4. Integrate Lambda
  5. Grant permission to invoke Lambda
  6. Deploy to stage

Lambda permission example:


aws lambda add-permission \
  --function-name hello \
  --statement-id apigw \
  --action lambda:InvokeFunction \
  --principal apigateway.amazonaws.com

Miss one step → 403 error 😅

7. Storage

7.1 S3 — object storage

Buckets are global namespace.

You manage:

  • Bucket policy
  • Object ACL
  • Encryption
  • Lifecycle rules

Example bucket policy:


{
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-site/*"
}

7.2 Databases

DynamoDB

  • No servers
  • Partition keys matter
  • IAM-based access

RDS

  • Managed EC2
  • Still needs:
    • Subnet groups
    • Security groups
    • Backups
    • Patch windows

8. Environment separation (WITHOUT wrappers)

Wrappers hide this — bare AWS makes it explicit.

Option A: Separate accounts (best)


aws-dev
aws-prod
aws-staging

Option B: Same account (manual discipline)

  • Resource naming
  • Separate IAM roles
  • Separate VPCs

Example:


my-api-dev
my-api-prod

9. Logging & Monitoring

CloudWatch

  • Logs → per service
  • Metrics → CPU, memory, invocations
  • Alarms → SNS

Lambda logs:


/aws/lambda/my-function

10. Deployment (no CI fairy)

Bare AWS means:

  • Zip files
  • CLI commands
  • Manual rollbacks

Typical flow:


zip function.zip index.js
aws lambda update-function-code \
  --function-name hello \
  --zip-file fileb://function.zip

11. Why wrappers exist (hard truth)

Wrappers solve:

  • IAM hell
  • Ordering dependencies
  • Rollbacks
  • Repeatability
  • Human error

Bare AWS gives:

  • Maximum control
  • Maximum responsibility
  • Maximum pain 😄

12. When bare AWS makes sense

✅ Learning AWS deeply

✅ Regulated / locked-down environments

✅ Highly customized networking

❌ Fast-moving startups

❌ Solo devs shipping fast
