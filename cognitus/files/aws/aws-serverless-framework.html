AWS Serverless Framework

The AWS Serverless Framework (usually just called Serverless Framework) is a developer toolchain that lets you define, deploy, and manage serverless applications on AWS using configuration instead of manual AWS console work.

Since you already work with AWS Lambda, API Gateway, Cognito, MongoDB, and Node.js, I’ll explain it from the ground up, then map it to how it actually runs on AWS, and finally show real-world flows and best practices.

1. What “Serverless Framework” actually is

Serverless Framework is NOT a cloud service.

It is a CLI + configuration system that:

  • Translates your app definition into AWS CloudFormation
  • Deploys AWS resources in the correct order
  • Packages and uploads your code to AWS
  • Manages stages (dev, prod, etc.)
  • Handles permissions (IAM)
  • Supports many cloud providers (AWS, Azure, GCP), but AWS is the most common

👉 Think of it as Terraform + Lambda-specific automation, but more opinionated for serverless apps.

2. Core building blocks

A. serverless.yml (the brain of everything)

This file defines what AWS should create.

Example (simplified):


service: my-api

provider:
  name: aws
  runtime: nodejs18.x
  region: ap-southeast-1
  stage: dev

functions:
  hello:
    handler: handler.hello
    events:
      - http:
          path: hello
          method: get

This single file eventually becomes:

  • 1 Lambda function
  • 1 API Gateway endpoint
  • IAM roles
  • CloudWatch logs
  • Permissions to link everything

B. Functions → AWS Lambda

Each functions: entry becomes one AWS Lambda function.


functions:
  login:
    handler: auth.login

Maps to:

  • Lambda function name: my-api-dev-login
  • Runtime: Node.js
  • Entry point: auth.js → exports.login

C. Events → Triggers

Events define what triggers the Lambda.

Common ones:

  • API Gateway (HTTP / REST)
  • SQS
  • SNS
  • EventBridge
  • S3
  • DynamoDB Streams
  • Scheduled cron jobs

Example:


events:
  - http:
      path: users/{id}
      method: get

Creates:

  • API Gateway route
  • Integration to Lambda
  • IAM permission allowing API Gateway → Lambda

3. What happens when you run serverless deploy

This is the most important part.

Step-by-step deployment flow

1️⃣ Parse serverless.yml

The framework reads:

  • provider settings
  • functions
  • plugins
  • environment variables

2️⃣ Package your code

  • Bundles Node.js files
  • Excludes node_modules if using layers
  • Zips everything

You can control this with:


package:
  individually: true

3️⃣ Generate CloudFormation

Serverless converts your config into CloudFormation templates.

This includes:

  • Lambda functions
  • API Gateway
  • IAM roles & policies
  • Log groups
  • Permissions

👉 Serverless itself does NOT create resources directly — CloudFormation does.

4️⃣ Upload artifacts to S3

  • Zip files are uploaded to an auto-created S3 bucket
  • CloudFormation references these objects

5️⃣ CloudFormation executes

AWS:

  • Creates or updates resources
  • Rolls back if something fails
  • Ensures consistency

This is why deployments are safe and repeatable.

4. Request lifecycle (runtime flow)

Let’s say a client calls your API.

Example: React Native → API Gateway → Lambda → MongoDB


Mobile App
   ↓ HTTPS
API Gateway
   ↓
Lambda (Node.js)
   ↓
MongoDB Atlas

Detailed runtime steps:

  1. Client hits API Gateway
  2. API Gateway:
    • Validates request
    • Applies authorizer (Cognito / JWT)
  3. API Gateway invokes Lambda
  4. Lambda:
    • Cold start (if not warm)
    • Executes handler
    • Reuses DB connection if possible
  5. Lambda returns response
  6. API Gateway sends HTTP response

5. IAM: how permissions work

Serverless auto-creates an IAM role per service (or per function if configured).

Example:


provider:
  iam:
    role:
      statements:
        - Effect: Allow
          Action:
            - s3:GetObject
          Resource: "*"

This becomes:

  • An IAM Role attached to Lambda
  • Least-privilege access

👉 This is critical for security and compliance.

6. Environment variables & stages

Stages


serverless deploy --stage prod

Creates:

  • my-api-dev-*
  • my-api-prod-*

You can define stage-specific values:


provider:
  environment:
    DB_URI: ${ssm:/my-api/${self:provider.stage}/db-uri}

Uses:

  • AWS SSM Parameter Store
  • Secrets Manager

7. Plugins (why Serverless is powerful)

Serverless is plugin-driven.

Common plugins you may already use or should use:

A. serverless-offline

Run API Gateway + Lambda locally


serverless offline

Great for React Native backend testing.

B. serverless-esbuild

Fast bundling for Node.js Lambdas


plugins:
  - serverless-esbuild

C. serverless-domain-manager

Custom domains for APIs

8. How it compares to other approaches

Approach Pros Cons
AWS Console Visual Manual, error-prone
CloudFormation raw Powerful Verbose, slow
Terraform Multi-cloud More boilerplate
Serverless Framework Fast, dev-friendly Opinionated

For Lambda-heavy APIs, Serverless is usually the fastest.

10. Key advantages

✔ Infrastructure as code

✔ Easy multi-environment deployments

✔ Automatic IAM

✔ Scales to zero

✔ Cheap for low/medium traffic

✔ Easy rollback

11. Common pitfalls (important)

❌ Too many Lambdas → harder tracing

❌ Cold starts (especially VPC + MongoDB)

❌ Large bundles increase latency

❌ Poor DB connection reuse

You already touched these in your MongoDB serverless work 😉

12. Mental model (best way to remember)

Serverless Framework = A compiler that converts your app definition into CloudFormation, which then creates real AWS resources.
