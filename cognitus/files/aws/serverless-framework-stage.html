Serverless Framework Stage

In Serverless Framework, stage represents the environment your service is deployed to.

In your config:


provider:
  stage: dev

What “stage” means (in simple terms)

A stage is a named deployment environment, commonly used to separate:

  • dev → development
  • test / qa → testing
  • staging → pre-production
  • prod → production

Each stage is isolated in AWS.

What changes when you change the stage?

1. API Gateway URL

Each stage gets its own URL.

Example:


https://abcd1234.execute-api.ap-southeast-1.amazonaws.com/dev/hello
https://abcd1234.execute-api.ap-southeast-1.amazonaws.com/prod/hello

So dev and prod do NOT affect each other.

2. Separate AWS Resources

When you deploy with different stages, Serverless creates separate resources:

  • Lambda functions
  • API Gateway stages
  • CloudWatch logs
  • (Optionally) DynamoDB tables, S3 buckets, etc.

Example Lambda names:


my-api-dev-hello
my-api-prod-hello

3. Environment-Specific Config

You typically use stage for:

  • Different databases
  • Different API keys
  • Different AWS accounts
  • Different feature flags

Example:


provider:
  environment:
    STAGE: ${self:provider.stage}

How to deploy different stages

Default (uses dev)


serverless deploy

Deploy to prod


serverless deploy --stage prod

This creates a new, independent deployment.

Common real-world setup


provider:
  stage: ${opt:stage, 'dev'}

Now you can do:


serverless deploy --stage dev
serverless deploy --stage prod

Why stage is important

Without stages:

  • Dev bugs can break production
  • No safe testing environment
  • Hard to manage credentials

With stages:

  • Clean separation
  • Safer releases
  • Professional deployment workflow

This is a Serverless Framework configuration snippet that defines an environment variable for your AWS Lambda functions.


provider:
  environment:
    STAGE: ${self:provider.stage}

What it means

  • provider.environment

    Declares environment variables that will be injected into all Lambda functions in the service.

  • STAGE

    The name of the environment variable available at runtime.

  • ${self:provider.stage}

    A Serverless variable reference that resolves to the current deployment stage (e.g. dev, prod, staging).

So if you deploy with:


serverless deploy --stage dev

Then inside Lambda:


process.env.STAGE === 'dev'

If you deploy with:


serverless deploy --stage prod

Then:


process.env.STAGE === 'prod'

Why this is useful

  • Environment-specific logic (dev vs prod)
  • Selecting correct databases, API keys, or feature flags
  • Cleaner configs without hardcoding values

Common pattern (recommended)

You’ll often see this combined with stage-based configs:


custom:
  dbName:
    dev: myapp-dev-db
    prod: myapp-prod-db

provider:
  environment:
    STAGE: ${self:provider.stage}
    DB_NAME: ${self:custom.dbName.${self:provider.stage}}

Then in Node.js Lambda:


const stage = process.env.STAGE;
const dbName = process.env.DB_NAME;

In the Serverless Framework, there is no built-in CLI command that directly lists all deployed stages for a service.

However, here are the correct and practical ways to see stages 👇

1️⃣ Check deployed stages via CloudFormation stacks (recommended)

Each stage = one CloudFormation stack.


serverless info --verbose

This shows the currently selected stage and stack details.

To list all stages, use AWS CLI:


aws cloudformation list-stacks \
  --stack-status-filter CREATE_COMPLETE UPDATE_COMPLETE \
  --query "StackSummaries[].StackName"

Then look for:


your-service-name-dev
your-service-name-prod
your-service-name-staging

2️⃣ List stages using Serverless variables (local config)

Check what stage would be used:


serverless print

or explicitly:


serverless print --stage dev
serverless print --stage prod

This prints the resolved configuration for that stage.

3️⃣ See deployed endpoints per stage


serverless info --stage dev
serverless info --stage prod

If the stage exists, you’ll see endpoints.

If not, it will fail.

4️⃣ Quick one-liner to list stages (AWS CLI)


aws cloudformation describe-stacks \
  --query "Stacks[].StackName" \
  | grep your-service-name

Important note (common confusion)

serverless stage ❌ does NOT exist

Stages are not centrally stored—they’re inferred from:

  • Stack names
  • Deployment history in AWS

Pro tip (best practice)

Define your stages explicitly to avoid guessing:


custom:
  stages:
    - dev
    - staging
    - prod

Then enforce:


serverless deploy --stage dev

aws cloudformation list-stacks \
  --stack-status-filter CREATE_COMPLETE UPDATE_COMPLETE \
  --query "StackSummaries[].StackName"

[
    "castanet-api",
    "awseb-e-mwzgqu4vr2-stack"
]

That output simply means CloudFormation found two stacks in a “complete” state in your AWS account:

What each stack likely is

  1. castanet-api
    • This is almost certainly your own stack.
    • Likely created via Serverless Framework, SAM, or a manual CloudFormation template.
    • Contains resources like:
      • Lambda functions
      • API Gateway
      • IAM roles
      • DynamoDB / other services (depending on your setup)
  2. awseb-e-mwzgqu4vr2-stack
    • This is an Elastic Beanstalk–managed CloudFormation stack
    • awseb = AWS Elastic Beanstalk
    • AWS creates and manages this automatically when you deploy an EB environment
    • You generally do not edit this stack directly

Useful next commands

See full details of a stack


aws cloudformation describe-stacks --stack-name castanet-api

List resources inside a stack


aws cloudformation list-stack-resources --stack-name castanet-api

Check which service created it (tags help a lot)


aws cloudformation describe-stacks \
  --stack-name castanet-api \
  --query "Stacks[0].Tags"

Delete a stack (⚠️ destructive)


aws cloudformation delete-stack --stack-name castanet-api

Important notes

  • ✅ It’s safe to keep Elastic Beanstalk stacks unless you are deleting the EB environment
  • ❌ Don’t manually modify awseb-* stacks
  • ✅ Serverless Framework stacks are usually safe to redeploy or delete if you know the app lifecycle

An AWS environment is a logically separated setup of AWS resources used for a specific purpose, such as development, testing, or production.

Think of it as “a complete AWS workspace” where your app runs.

Simple definition

AWS Environment = AWS account + region + configuration + resources

It defines where and how your application runs in AWS.

Common AWS environments (real-world)

Most teams use multiple environments:

Environment Purpose
dev Local development, experiments
test / qa Automated testing
staging Production-like testing
prod Real users, real data

Each environment is isolated to avoid breaking production.

What makes an AWS environment different?

1️⃣ AWS Account

  • Strongest isolation
  • Recommended:
    • mycompany-dev
    • mycompany-prod

Example:


Account A → dev
Account B → prod

2️⃣ AWS Region

Physical location of resources.

Examples:

  • ap-southeast-1 (Singapore)
  • us-east-1 (Virginia)

Why it matters:

  • Latency
  • Legal/compliance
  • Cost

3️⃣ AWS Resources

Each environment has its own:

  • Lambda functions
  • API Gateway
  • EC2 / ECS
  • DynamoDB / RDS
  • S3 buckets
  • IAM roles
  • CloudWatch logs

Example:


my-api-dev-hello
my-api-prod-hello

4️⃣ Configuration & Secrets

Each environment uses different values:

  • Database URLs
  • API keys
  • OAuth credentials
  • Feature flags

Example:


DEV_DB_URL=mongodb://dev-db
PROD_DB_URL=mongodb://prod-db

AWS environment vs Serverless stage

They are related but not the same.

Term Meaning
AWS Environment The actual AWS setup
Stage (Serverless) A logical label inside AWS

👉 A stage lives inside an AWS environment.

Example:


AWS Account: mycompany
Region: ap-southeast-1
Stages: dev, prod

Typical AWS environment setups

✅ Small projects


1 AWS Account
├── dev stage
└── prod stage

✅ Medium / enterprise (best practice)


AWS Account (dev)
└── dev stage

AWS Account (prod)
└── prod stage

How this applies to Serverless Framework

In your config:


provider:
  region: ap-southeast-1
  stage: dev

This means:

  • Deploy to Singapore region
  • Use dev environment configuration
  • Create dev AWS resources

Why AWS environments matter

  • 🔒 Security isolation
  • 💥 Prevent production outages
  • 🧪 Safe testing
  • 💰 Cost tracking per environment
  • 🚀 Professional deployment pipelines

Yes — absolutely yes 👍

You can (and very often do) have multiple stages in a single AWS account.

This is actually very common, especially for small teams and startups.

Short answer

1 AWS account → many stages

Example:


AWS Account: mycompany
Region: ap-southeast-1
Stages: dev, test, staging, prod

How this works in practice (Serverless Framework)


provider:
  name: aws
  region: ap-southeast-1
  stage: ${opt:stage, 'dev'}

Deploy:


serverless deploy --stage dev
serverless deploy --stage prod

Serverless creates separate resources per stage.

What stays the same?

  • ✅ AWS account
  • ✅ AWS billing
  • ✅ IAM users (unless you separate them)

What is different per stage?

🔹 API Gateway

Different URLs:


/dev/hello
/prod/hello

🔹 Lambda functions

Different names:


my-api-dev-hello
my-api-prod-hello

🔹 Logs

Separate CloudWatch log groups:


/aws/lambda/my-api-dev-hello
/aws/lambda/my-api-prod-hello

🔹 Environment variables


provider:
  environment:
    STAGE: ${self:provider.stage}

Example: different DB per stage (same account)


custom:
  db:
    dev: mongodb://dev-db
    prod: mongodb://prod-db

provider:
  environment:
    DB_URL: ${self:custom.db.${self:provider.stage}}

Pros of single AWS account + multiple stages

✅ Simple setup

✅ Lower overhead

✅ Easy to manage

✅ Perfect for:

  • Solo devs
  • Startups
  • MVPs
  • Side projects

Cons / risks ⚠️

❌ Accidental prod deletion

❌ IAM permissions harder to lock down

❌ Dev bugs could affect prod if misconfigured

(Still manageable with good discipline.)

When to use multiple AWS accounts instead

Use separate AWS accounts when:

  • You’re handling sensitive production data
  • You need strict security isolation
  • You have a large team
  • Compliance requirements exist

Best-practice hybrid (very common)


AWS Account (dev)
└── dev, test stages

AWS Account (prod)
└── prod stage

Bottom line

✔ Yes, multiple stages in one AWS account is totally fine

✔ Serverless Framework fully supports it

✔ Many production apps run this way
