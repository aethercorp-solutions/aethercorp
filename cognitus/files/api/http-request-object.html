request object

In HTTP, a request object is basically the structured representation of a client’s request sent to a server. When your app (like your Node.js API on AWS Lambda) receives an HTTP request, frameworks turn that raw request into a request object so you can easily work with it in code.

What an HTTP request contains

At the protocol level, an HTTP request has:

  • Method → what action to perform (GET, POST, PUT, DELETE)
  • URL / Path → the endpoint (/users, /login)
  • Headers → metadata (e.g. Authorization, Content-Type)
  • Body → data sent by the client (for POST/PUT)
  • Query parameters → values in the URL (?page=1&limit=10)

Example (raw HTTP request)


POST /login HTTP/1.1
Host: example.com
Content-Type: application/json
Authorization: Bearer token123

{
  "email": "test@example.com",
  "password": "123456"
}

Request object in Node.js (Express-style)

Frameworks like Express.js convert that into a request object (req):


app.post('/login', (req, res) => {
    console.log(req.method);      // POST
    console.log(req.path);        // /login
    console.log(req.headers);     // headers object
    console.log(req.body);        // parsed JSON body
    console.log(req.query);       // query params
});

In AWS Lambda (API Gateway)

Since you're using AWS, your request object looks slightly different:


exports.handler = async (event) => {
    console.log(event.httpMethod);   // POST
    console.log(event.path);         // /login
    console.log(event.headers);      // headers
    console.log(event.body);         // raw string (JSON)
};

Here, the event object = request object.

Key idea

The request object is:

A program-friendly wrapper around the incoming HTTP request, so you don’t have to manually parse raw text.

Why it matters (especially for your setup)

Since you're working with:

  • OAuth (Google / Azure AD)
  • JWT authentication
  • Serverless APIs

You’ll constantly use the request object to:

  • Read tokens → req.headers.authorization
  • Get auth codes → req.headers.code
  • Parse user data → req.body
  • Handle query params → req.query

Where and How data is sent in an HTTP request

🧱 1. HTTP Request Structure (big picture)

An HTTP request has 4 main parts:


POST /users?id=123 HTTP/1.1
Host: example.com
Authorization: Bearer abc123

{
  "name": "Paul"
}
  • Request line → URL + query params
  • Headers → metadata (Authorization, Content-Type, etc.)
  • Body → actual payload (data)
  • Query params → part of URL

🔑 2. Authorization Header (what it is)

This is a header used for authentication.


Authorization: Bearer abc123
  • Lives in: req.headers.authorization
  • Purpose: tell server who you are

🪪 3. Bearer Token (what it is)

“Bearer” is just a type of Authorization scheme.


Authorization: Bearer <token>

Meaning:

“Whoever bears this token is authenticated”

Common token types:

  • JWT
  • OAuth access token (like Google)

📦 4. Request Body (what it is)

This is where you send actual data.


{
    "email": "paul@example.com",
    "password": "123456"
}
  • Access in Node.js: req.body
  • Used for:
    • form submissions
    • JSON APIs
    • POST/PUT requests

🔗 5. Request Parameters (2 types)

a) Query Parameters


GET /users?role=admin&page=2
  • Access: req.query.role
  • Used for:
    • filtering
    • searching
    • pagination

b) Path Parameters


GET /users/123
  • Access: req.params.id
  • Used for:
    • identifying a specific resource

⚖️ Side-by-side comparison

Type Where it lives Example Purpose
Authorization Header req.headers.authorization Bearer abc123 Authentication
Bearer Inside Authorization Bearer Token type
Body req.body { email, password } Data payload
Query Param req.query ?page=1 Filters/search
Path Param req.params /users/123 Resource ID

🧠 Real-world example (your Google OAuth flow)

❌ Your current approach


req.headers.code

You're putting OAuth code in headers.

✅ Standard approach

Step 1: send code in body


POST /auth/google
Content-Type: application/json

{
  "code": "4/0AX4XfWh..."
}

Step 2: backend receives it


const { code } = req.body;

✅ After login (protected routes)


GET /profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
            

const token = req.headers.authorization;

🔥 Simple mental model

  • Headers → identity / metadata
  • Bearer → type of auth token
  • Body → actual data
  • Params → filtering or identifying resources

⚠️ Common mistakes (you’re close to one)

❌ Putting everything in headers

Bad for:

  • debugging
  • standards
  • proxies/CDNs

❌ Putting tokens in body


{ "token": "abc123" }

Bad because:

  • not standard
  • harder to secure

✅ Best practice summary

  • Auth → Authorization header
  • Data → Body
  • Filters → Query params
  • Resource IDs → Path params
