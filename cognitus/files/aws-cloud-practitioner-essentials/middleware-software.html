Middleware is software

Middleware is software that acts as a bridge between different applications, systems, or components, allowing them to communicate and work together smoothly.

Simple explanation

Think of middleware like a translator or middle layer:

  • One application sends a request
  • Middleware processes or modifies it
  • Then passes it to another application
  • And returns the response back

In everyday terms

Imagine ordering food:

  • You (client) → waiter (middleware) → kitchen (server)
  • The waiter takes your order, delivers it, and brings back the food

The waiter is middleware.

In software (especially web development)

Middleware sits between the request and the response in an application.

For example, in a web app:

  1. User sends a request (e.g., login)
  2. Middleware can:
    • Check authentication
    • Log the request
    • Validate data
  3. Then the request reaches the main application logic
  4. Response goes back through middleware again

Common uses of middleware

  • Authentication (checking if a user is logged in)
  • Logging (tracking requests)
  • Data validation
  • Error handling
  • Security filtering

Example (conceptual)

In frameworks like Express.js or Django, middleware functions run before your main code:


Request → Middleware → Application → Middleware → Response

Key idea

Middleware = "in-between software that manages communication and processing"

1. Client Layer

This is your React Native app.


fetch('https://api.example.com/users/profile', {
    method: 'GET',
    headers: {
        Authorization: `Bearer ${token}`
    }
});

The client:

  • Sends HTTP request
  • Adds JWT token
  • Sends JSON payload
  • Waits for response

2. API Gateway

Amazon API Gateway acts as the entry point.

Responsibilities:

  • Receives HTTP request
  • Routes request to correct Lambda
  • Handles CORS
  • Rate limiting
  • Authorization
  • Request validation

Example:


GET /users/profile
    ↓
routes to
    ↓
Lambda function: getProfile

3. Lambda Handler (Entry Point)

Your Lambda receives the request event.

Example:


module.exports.getProfile = async (event) => {
    console.log(event);
};

The event contains:

  • headers
  • body
  • query params
  • path params
  • auth data

Example:


event.headers.Authorization

4. Middleware Layer

Middleware executes BEFORE business logic.

In Node.js serverless apps, middleware is commonly implemented using:

  • Middy
  • custom wrapper functions
  • Express-like patterns

Example architecture:


Request
  ↓
Auth Middleware
  ↓
Validation Middleware
  ↓
Logging Middleware
  ↓
Business Logic

5. Authentication Middleware

This checks JWT tokens.

Example flow:


Client sends JWT
    ↓
Middleware extracts token
    ↓
Verifies token
    ↓
Adds user info to request context
    ↓
Continues to business logic

Example:


const authMiddleware = async (event) => {
    const token = event.headers.Authorization;
    
    const decoded = verifyJWT(token);
    
    event.user = decoded;
};

If invalid:


return {
    statusCode: 401,
    body: 'Unauthorized'
};

6. Validation Middleware

Checks request data before business logic runs.

Example:


POST /users

Middleware checks:

  • email exists
  • password length
  • valid JSON

Example:


if (!body.email) {
    throw new Error('Email required');
}

7. Logging Middleware

Tracks requests for monitoring/debugging.

Example:


console.log({
    path: event.path,
    user: event.user.id
});

Useful for:

  • CloudWatch logs
  • analytics
  • debugging production issues

8. Business Logic Layer

This is the actual application functionality.

Example:


const user = await User.findById(event.user.id);

Business logic should:

  • contain app rules
  • not handle authentication directly
  • focus on operations

Examples:

  • create invoice
  • calculate totals
  • process payments
  • generate analytics

9. Database Layer

Your Lambda connects to MongoDB.

Example:


await mongoose.connect(process.env.MONGO_URI);

Flow:


Business Logic
   ↓
Mongoose Model
   ↓
MongoDB

Example:


const invoices = await Invoice.find({
    userId: event.user.id
});

10. Response Middleware

Middleware can also modify responses.

Example:


return {
    statusCode: 200,
    headers: {
        'Access-Control-Allow-Origin': '*'
    },
    body: JSON.stringify(data)
};

Response middleware may:

  • add headers
  • compress response
  • standardize API format
  • handle errors

11. Response Back to Client

Final response flow:


MongoDB
   ↓
Business Logic
   ↓
Middleware
   ↓
API Gateway
   ↓
React Native App

Client receives:


{
    "success": true,

In AWS serverless architecture, a Lambda Handler is the main function AWS Lambda executes when an event triggers your function.

Think of it as:

  • the entry point
  • the starting execution function
  • the bridge between AWS services and your business logic

In Node.js serverless apps, the handler receives:

  1. the incoming event/request
  2. execution context
  3. optionally a callback

Detailed Anatomy

1. Event Object

The event contains the incoming request data.

If using API Gateway:


exports.handler = async (event) => {
    console.log(event);
};

Example event:


{
    "httpMethod": "GET",
    "path": "/users/123",
    "headers": {
        "Authorization": "Bearer token"
    },
    "queryStringParameters": {
        "expand": "profile"
    },
    "pathParameters": {
        "id": "123"
    },
    "body": "{\"name\":\"Paul\"}"
}

Event Sources

Provides Lambda runtime info.

Example Professional Handler


const middy = require('@middy/core');
const authMiddleware = require('../middleware/auth');
const validatorMiddleware = require('../middleware/validator');
const userService = require('../services/userService');

const baseHandler = async (event) => {

    const userId = event.pathParameters.id;

    const user = await userService.getUser(userId);

    return {
        statusCode: 200,
        body: JSON.stringify(user)
    };
};

exports.handler = middy(baseHandler)
    .use(authMiddleware())
    .use(validatorMiddleware());

Example Authentication Middleware


const authMiddleware = () => {
    return {
        before: async (request) => {

            const token =
                request.event.headers.Authorization;

            if (!token) {
                throw new Error('Unauthorized');
            }
        }
    };
};

Lambda Execution Environment

AWS creates isolated containers.

Each Lambda invocation runs in:


Micro VM / Container

Contains:

  • Node.js runtime
  • memory
  • temporary storage /tmp
  • environment variables

Important Lambda Concepts

1. Statelessness

Do NOT rely on memory persistence.

Bad:


let currentUser;

Lambda containers may disappear anytime.

Use:

  • DynamoDB
  • MongoDB
  • Redis
  • S3

2. Reused Containers

Global variables may persist temporarily.

Example:


const mongoose = require('mongoose');

let cachedConnection;

Useful for DB optimization.

MongoDB Optimization Example

Common in your AWS + Mongoose architecture:


let isConnected = false;

module.exports.connectDB = async () => {

    if (isConnected) {
        return;
    }

    await mongoose.connect(process.env.MONGO_URI);

    isConnected = true;
};

Without this:

  • every invocation reconnects to MongoDB
  • performance degrades

3. Timeout

Every Lambda has timeout:


timeout: 30

If exceeded:


Task timed out

4. Memory

Configured memory:


memorySize: 512

More memory:

  • more CPU power
  • faster execution
  • higher cost

Error Handling

Example:


exports.handler = async () => {

    try {

        return {
            statusCode: 200,
            body: 'Success'
        };

    } catch (error) {

        return {
            statusCode: 500,
            body: JSON.stringify({
                message: error.message
            })
        };
    }
};

Example JWT Verification


const jwt = require('jsonwebtoken');

exports.handler = async (event) => {

    const token =
        event.headers.Authorization;

    const decoded =
        jwt.verify(token, process.env.JWT_SECRET);

    return {
        statusCode: 200,
        body: JSON.stringify(decoded)
    };
};

Best Practices

Keep Handler Thin

Good:


Handler = orchestration only

Avoid:

  • huge business logic
  • direct DB complexity
  • large validation blocks

Use Layers

For shared:

  • utilities
  • SDKs
  • common libraries

Reuse Connections

Especially:

  • MongoDB
  • Redis
  • PostgreSQL

Use Environment Variables


process.env.MONGO_URI

Never hardcode secrets.

Structured Logging


console.log(JSON.stringify({
    action: 'GET_USER',
    userId
}));

Better for CloudWatch analysis.

Summary

A Lambda Handler is:


The main serverless execution entry point

It:

  • orchestrates middleware
  • calls business logic
  • returns responses

In professional AWS serverless architecture:

  • handler = thin controller
  • middleware = cross-cutting concerns
  • services = business logic
  • repositories = data access
  • AWS Lambda = execution environment
