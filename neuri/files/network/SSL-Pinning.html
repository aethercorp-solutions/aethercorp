SSL pinning

SSL pinning (also known as certificate pinning) is a security technique used in mobile and web applications to help prevent man-in-the-middle (MITM) attacks by ensuring the client connects only to a trusted server with a known SSL/TLS certificate.

What Problem Does SSL Pinning Solve?

Normally, during an HTTPS connection:

  • The client (e.g., app or browser) connects to a server.
  • The server provides its SSL/TLS certificate.
  • The client verifies that the certificate is signed by a trusted Certificate Authority (CA).

Problem: If an attacker can trick the device into trusting a malicious certificate (e.g., via rogue Wi-Fi, compromised CA, or custom root certificates), they can intercept and decrypt the traffic — a classic MITM attack.

How SSL Pinning Works

SSL pinning helps prevent this by "pinning" a specific certificate or public key to the client app.

  • The app has a copy of the expected server certificate or public key hardcoded or securely embedded.
  • When connecting, the app checks that the server's certificate matches the pinned one.
  • If it doesn’t match, the connection is rejected, even if the certificate is valid and signed by a trusted CA.

Example

Let’s say your app communicates with https://api.mybank.com.

With SSL pinning:

  • You pin the public key or certificate of api.mybank.com in your app.
  • On every connection, the app compares the received certificate/key with the pinned one.
  • If a MITM attacker presents a fake certificate (even one signed by a trusted CA), the app blocks the connection.

Types of Pinning

  • Certificate Pinning: Pins the entire server certificate.
  • Public Key Pinning: Pins only the public key from the certificate.
  • SPKI Pinning: A specific format of public key pinning using the Subject Public Key Info field.

Considerations

  • Pros:
    • Strong defense against MITM.
    • Works even if a CA is compromised or a custom CA is added to the device.
  • Cons:
    • Hardcoding certificates can cause outages if they expire or are rotated without app updates.
    • Must handle renewal or fallback strategies properly.

Common Use Cases

  • Banking and finance apps
  • Apps handling sensitive personal data
  • Environments where devices might be rooted or tampered with
