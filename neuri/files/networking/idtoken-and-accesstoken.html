idToken and accessToken

The terms idToken and accessToken are often used in the context of authentication and authorization protocols like OAuth 2.0 and OpenID Connect (OIDC). They serve different purposes even though they are both issued by an authorization server (like Auth0, Firebase, AWS Cognito, etc.).

üîê idToken ‚Äì Identity Token

  • Purpose: To authenticate the user and convey who the user is.
  • Used By: The client application (frontend, SPA, mobile app) to identify the user.
  • Contains:
    • User's identity info (e.g., sub, name, email)
    • Issuer info (who issued the token)
    • Expiry time
  • Format: Usually a JWT (JSON Web Token)
  • Issued When: Using OpenID Connect (OIDC)
  • Sent To: Client application after login (not to APIs)

üî∏ Think of it as the user's profile, digitally signed by the identity provider.

  • Purpose: To authorize access to protected resources (like APIs).
  • Used By: The client application to access APIs on behalf of the user.
  • Contains:
    • Scopes (permissions granted)
    • Resource access claims
    • Expiry time

      Authorization: Bearer <accessToken>

  • Format: Often a JWT, but not always (depends on provider)
  • Issued When: Using OAuth 2.0 or OIDC
  • Sent To: API servers in the Authorization header:

üî∏ Think of it as a key to access resources, not a source of identity.
