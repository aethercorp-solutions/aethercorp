OAuth 2.0 client

An OAuth 2.0 client is an application that accesses a resource (like an API) on behalf of a user by obtaining authorization through the OAuth 2.0 protocol. It's one of the key roles defined in the OAuth 2.0 framework.

üîë Key Roles in OAuth 2.0

To understand the client, it's helpful to know the main entities in OAuth:

Role Description
Resource Owner The user who owns the data (e.g., a person).
Client The app requesting access to the resource owner's data.
Authorization Server The server that authenticates the user and issues access tokens.
Resource Server The server hosting the protected resources (e.g., an API).

‚úÖ What the Client Does

The OAuth client:

  • Initiates the authorization flow.
  • Redirects the user to the authorization server.
  • Exchanges an authorization code (or other credentials) for an access token.
  • Uses the access token to access protected resources on behalf of the user.

üõ†Ô∏è Common OAuth 2.0 Grant Types for Clients

Grant Type Use Case
Authorization Code Most secure; used by web/mobile apps with server-side component.
Client Credentials Used for server-to-server communication; no user involved.
Implicit Deprecated; previously used by client-side apps (e.g., SPAs).
Resource Owner Password Deprecated; used when the app asks for user credentials directly.
Device Authorization Used on devices with limited input (e.g., TVs, IoT).

üîê Example Flow (Authorization Code)

  1. Client redirects user to Authorization Server.
  2. User logs in and authorizes the client.
  3. Server redirects back with authorization code.
  4. Client exchanges code for access token.
  5. Client uses access token to call Resource Server.

üîë Key Tokens in Google Sign-In

Token Purpose
Access Token Lets your app access Google APIs on behalf of the user.
ID Token Confirms the user's identity (contains their email, name, etc.).
Refresh Token (optional) Lets your app get a new access token without re-authenticating.

üîÅ Example Flow (Web App)

  1. User clicks "Sign in with Google".
  2. Your app redirects them to Google‚Äôs OAuth 2.0 Authorization Endpoint.
  3. User authenticates and consents.
  4. Google redirects back to your app with:
    • An authorization code (for server-side apps).
  5. Your backend exchanges the code for:
    • Access Token
    • ID Token
    • (Optional) Refresh Token
  6. You use:
    • ID Token to verify the user's identity.
    • Access Token to call Google APIs.

üîí Why OAuth + OpenID Connect?

  • OAuth 2.0 = Authorization (what the app can do)
  • OpenID Connect = Authentication (who the user is)

Google Sign-In combines both to give you identity and access in one flow.

üîê 1. What Is Firebase Authentication?

Firebase Authentication is a service that provides backend services, easy-to-use SDKs, and ready-made UI libraries to authenticate users to your app. It supports:

  • Email and password
  • Phone number
  • Google, Facebook, Twitter, Apple, GitHub (federated identity providers)
  • Anonymous authentication
  • Custom authentication system

üß† 2. How Google Sign-In Works with Firebase

üîÑ Flow Overview:

  1. User Clicks ‚ÄúSign In with Google‚Äù
    • Your app presents a Google sign-in UI (either via popup or redirect).
  2. Google Sign-In
    • The user selects their Google account and grants permissions.
    • Google returns an ID token and an access token.
  3. Firebase Auth Handles the Token
    • Your app sends the ID token to Firebase.

    • Firebase returns a User object, including UID, email, display name, photo URL, etc.
