Middleware

Middleware is one of the most important concepts in a Node.js API—especially when using Express.js.

Instead of jumping straight to the answer, think of middleware as a pipeline that every request passes through before reaching your main logic.

🧠 What is Middleware?

Middleware = a function that runs between the request and the response

It has access to:


(req, res, next)
  • req → incoming request
  • res → response object
  • next() → passes control to the next middleware

🔄 Full Middleware Flow (Step-by-Step)

Let’s say you call:


GET /profile

Here’s what happens internally:


Client Request
      ↓
[Middleware 1] → Logging
      ↓
[Middleware 2] → Authentication
      ↓
[Middleware 3] → Validation
      ↓
[Route Handler] → Business Logic
      ↓
Response Sent Back

🔧 Example Flow in Code


app.get(
  '/profile',
  logMiddleware,
  authMiddleware,
  validateMiddleware,
  (req, res) => {
    res.json({ message: "User profile data" });
  }
);

🔍 Each Step Explained

1. Logging Middleware


function logMiddleware(req, res, next) {
    console.log(`${req.method} ${req.url}`);
    next(); // move to next
}

👉 Runs first

👉 Does not stop the request

2. Authentication Middleware


function authMiddleware(req, res, next) {
    const token = req.headers.authorization;

    if (!token) {
        return res.status(401).json({ message: "Unauthorized" });
    }

    // validate token here
    next();
}

👉 Can stop the flow if invalid

👉 If valid → continues

3. Validation Middleware


function validateMiddleware(req, res, next) {
    if (!req.query.id) {
        return res.status(400).json({ message: "Missing ID" });
    }

    next();
}

4. Final Route Handler


(req, res) => {
    res.json({ message: "Success" });
}

⚡ Key Concept: next()

This is the most important part.

  • next() → go to next middleware
  • No next() → request gets stuck
  • return res... → stops everything

🧱 Types of Middleware

1. Application-level

Runs on all routes:


app.use(express.json());

2. Route-level

Runs only on specific routes:


app.get('/profile', authMiddleware, handler);

3. Global Middleware


app.use((req, res, next) => {
    console.log("Runs for every request");
    next();
});

4. Error Middleware


app.use((err, req, res, next) => {
    res.status(500).json({ error: err.message });
});

🔁 Real Flow (Your AWS Serverless Setup)

In your case (Lambda + API Gateway):


API Gateway
     ↓
Lambda Handler
     ↓
Middleware (auth, headers, logging)
     ↓
Controller
     ↓
MongoDB

Even in serverless, middleware still works—you just structure it manually or with frameworks.

🧠 Simple Analogy

Middleware is like security checkpoints at an airport:

  • ID check (auth middleware)
  • Bag scan (validation middleware)
  • Boarding gate (controller)

Fail any step → you don’t proceed

🚨 Common Mistakes

  • ❌ Forgetting next()
  • ❌ Sending response twice
  • ❌ Blocking request unintentionally
  • ❌ Putting heavy logic in middleware

✅ Best Practices

  • Keep middleware small and reusable
  • Separate concerns (auth, validation, logging)
  • Always handle errors
  • Use async/await properly

🚀 Example (Clean Structure)


app.use(logMiddleware);

app.get('/profile', authMiddleware, async (req, res) => {
    const user = await getUser(req.user.id);
    res.json(user);
});
