Nodemon

Nodemon is basically a watchdog for your Node.js app 🐶. It sits there, watches your files, and automatically restarts your server whenever something changes—so you don’t have to keep stopping and starting it manually.

Here’s how it works under the hood, step by step.

1. What problem Nodemon solves

Normally, with plain Node.js:


node app.js

If you edit app.js (or any required file), nothing happens.

You must stop the process and run it again.

Nodemon automates this loop:

Edit code → stop server → restart server ❌

Edit code → Nodemon restarts server ✅

2. Core idea: file watching

Nodemon watches your project files for changes.

Internally it uses file-system watchers:

  • fs.watch
  • fs.watchFile
  • or efficient OS-level watchers (via libraries like chokidar)

It listens for events like:

  • file modified
  • file added
  • file deleted

Example:


src/server.js changed

3. Process lifecycle (important part)

When you run:


nodemon app.js

This is what happens:

Step 1: Nodemon starts Node

Nodemon spawns a child process:


nodemon
 └── node app.js

Nodemon itself is not your server — it’s just the manager.

Step 2: You edit a file

You save a file Nodemon is watching (e.g. .js, .json).

Step 3: Nodemon detects the change

Nodemon:

  1. Receives a file change event
  2. Debounces it (waits briefly so multiple saves don’t restart repeatedly)

Step 4: Nodemon restarts the app

It does this cleanly:

  1. Sends a termination signal to Node:
    • SIGUSR2 (default)
    • or SIGTERM
  2. Kills the current Node process
  3. Spawns a new Node process
  4. Runs the same command again

[nodemon] restarting due to changes...

4. What Nodemon watches by default

By default Nodemon:

  • Watches the current directory
  • Watches these extensions:
    • .js
    • .json
    • .mjs
  • Ignores:
    • node_modules
    • .git

You can customize this.

5. Configuration (nodemon.json)

Example nodemon.json:


{
    "watch": ["src"],
    "ext": "js,json",
    "ignore": ["src/logs/*"],
    "exec": "node src/server.js",
    "delay": "300"
}

This tells Nodemon:

  • Watch only src
  • Restart only on .js and .json changes
  • Ignore logs
  • Restart after 300ms

6. Signals & graceful shutdown (important in real apps)

You can handle Nodemon restarts properly:


process.on("SIGTERM", () => {
    console.log("Shutting down...");
    server.close(() => process.exit(0));
});

This matters if you use:

  • HTTP servers
  • DB connections (MongoDB, PostgreSQL)
  • AWS Lambda local emulation

7. Nodemon vs production

🚨 Nodemon is for development only

Why?

  • File watching costs resources
  • Restarting processes in prod is risky
  • Production uses:
    • pm2
    • forever
    • Docker health checks
    • Kubernetes restarts

Typical setup:


"scripts": {
    "dev": "nodemon src/server.js",
    "start": "node src/server.js"
}

8. Mental model (simple analogy)

Think of Nodemon as:

“A developer assistant that keeps one eye on your files and one hand on the restart button.”

You write code → Nodemon handles the boring part.
