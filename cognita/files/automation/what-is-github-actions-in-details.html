GitHub Actions

Think of GitHub Actions as GitHub’s built-in robot that watches your repo and does stuff automatically when something happens 🤖

Here’s the mental model, step by step.

1️⃣ What triggers a GitHub Action?

An event happens in your repo, for example:

  • You push code
  • You open a pull request
  • You merge to main
  • You manually click “Run workflow”

These events are defined in YAML using on:.


on:
  push:
    branches:
      - main

👉 This means: “Run this workflow whenever someone pushes to main.”

2️⃣ Where does GitHub Actions live?

All workflows live here:


.github/workflows/

Each file is a workflow, written in YAML.

Example:


.github/workflows/deploy.yml

3️⃣ What is a workflow made of?

A workflow has jobs, and jobs have steps.

High level:


Workflow
 └── Job
      └── Step

Example:


name: Deploy API

on: push

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npm test

4️⃣ What is a job?

A job:

  • Runs on a fresh virtual machine (runner)
  • Has its own OS (ubuntu, windows, macos)
  • Can run in parallel with other jobs

jobs:
  build:
    runs-on: ubuntu-latest

5️⃣ What is a step?

A step is a single action inside a job.

Two types:

  1. Use an existing action 
    
- uses: actions/checkout@v4
  2. Run shell commands 
    
- run: npm install

Steps run top to bottom.

6️⃣ What is a runner?

A runner is the machine that executes your job.

  • GitHub-hosted runner → free VM from GitHub
  • Self-hosted runner → your own server

Each run:

  • Starts clean
  • Runs your steps
  • Gets destroyed after

7️⃣ How do environment variables work?

You can:

  • Set env variables
  • Pass values between steps
  • Use GitHub secrets

Example:


env:
  NODE_ENV: production

Or dynamically:


echo "STAGE=prod" >> $GITHUB_ENV

(This is exactly what you used in your deploy example.)

8️⃣ How secrets work 🔐

Secrets are stored in:

  • Repo settings
  • Org settings

Usage:


env:
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}

👉 Secrets are:

  • Encrypted
  • Hidden in logs
  • Injected at runtime

9️⃣ Typical real-world flow (CI/CD)

For a backend like yours (Serverless + AWS):

  1. Push to GitHub
  2. GitHub Action triggers
  3. Runner spins up
  4. Code is checked out
  5. Dependencies installed
  6. Tests run
  7. App deployed

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx serverless deploy --stage prod

- uses: actions/checkout@v4

That line is one step in a GitHub Actions workflow. Here’s what it does, plain and simple 👇


- uses: actions/checkout@v4

What it means

  • uses:

    Tells GitHub Actions to run a prebuilt action (not a shell command).

  • actions/checkout

    This is the official GitHub action that checks out (clones) your repository into the runner.

  • @v4

    The version of the action (v4 is the current stable one).

Why it’s needed

Without this step:

  • Your workflow runner has no access to your code
  • npm install, serverless deploy, tests, builds → ❌ will fail

With this step:

  • Your repo is cloned into the runner
  • Files are available at GITHUB_WORKSPACE
  • All following steps can read your code

What happens under the hood

It basically runs:


git clone https://github.com/your-org/your-repo.git

(using a secure GitHub token automatically)

Typical placement

Almost always the first step in a job:


steps:
  - uses: actions/checkout@v4

  - name: Install dependencies
    run: npm ci

  - name: Deploy
    run: npx serverless deploy

Common options (optional)


- uses: actions/checkout@v4
  with:
    fetch-depth: 0   # full git history (needed for tags, versioning)
