JavaScript configuration object

A JavaScript configuration object is simply a plain JavaScript object used to store settings, options, or parameters that control how something behaves.

Think of it as a bundle of preferences you pass into a function, library, or component instead of passing many separate arguments.

Simple definition

A configuration object is an object that contains key–value pairs used to configure or customize behavior.

Basic example


const config = {
    theme: "dark",
    language: "en",
    debug: true
};

Here:

  • theme, language, and debug are configuration options
  • The object itself is the configuration object

Why configuration objects are used

1. Cleaner function calls

Instead of this ❌:


initApp("dark", "en", true, false, 3000);

You do this ✅:


initApp({
    theme: "dark",
    language: "en",
    debug: true,
    cache: false,
    timeout: 3000
});

Much easier to read and maintain.

2. Order doesn’t matter


initApp({
    timeout: 3000,
    debug: true
});

No need to remember parameter positions.

3. Optional settings

You only pass what you need.


function initApp(config = {}) {
    const debug = config.debug ?? false;
}

Real-world examples

1. Function configuration


function connectDatabase(config) {
    console.log(config.host, config.port);
}
    
connectDatabase({
    host: "localhost",
    port: 27017,
    ssl: false
});

2. Library usage (very common)


fetch(url, {
    method: "POST",
    headers: {
        "Content-Type": "application/json"
    },
    body: JSON.stringify(data)
});

The second argument is a configuration object.

3. React / React Native


<Button
  title="Save"
  color="blue"
  disabled={false}
/>

Props behave like a configuration object for the component.

4. Framework config files


// webpack.config.js
module.exports = {
  entry: "./src/index.js",
  output: {
    filename: "bundle.js"
  }
};

This entire file is a configuration object.

Common patterns in configuration objects

Default values


const defaultConfig = {
    retries: 3,
    timeout: 5000
};
    
const finalConfig = { ...defaultConfig, ...userConfig };

Nested configuration


const config = {
    api: {
        baseUrl: "https://api.example.com",
        version: "v1"
    },
    auth: {
        token: "abc123"
    }
};

Key takeaway

A JavaScript configuration object is:

  • Just a plain object
  • Used to configure behavior
  • Makes code readable, flexible, and scalable
  • Widely used in APIs, libraries, React, Node.js, and frameworks

Here’s a clear, real-world example of using global settings in a React Native app, with patterns you’d actually use in production.

Use case: Global App Settings (Theme, Language, Notifications)

Imagine a React Native app where:

  • Theme (light / dark)
  • Language (en / fil)
  • Notification toggle

must be accessible everywhere in the app.

1️⃣ Define a global settings configuration object


// settings/defaultSettings.js
export const defaultSettings = {
  theme: "light",
  language: "en",
  notificationsEnabled: true
};

This is your global configuration object.

2️⃣ Create a Settings Context (Global State)


// settings/SettingsContext.js
import React, { createContext, useContext, useState } from "react";
import { defaultSettings } from "./defaultSettings";

const SettingsContext = createContext(null);

export const SettingsProvider = ({ children }) => {
  const [settings, setSettings] = useState(defaultSettings);

  const updateSettings = (updates) => {
    setSettings((prev) => ({ ...prev, ...updates }));
  };

  return (
    <SettingsContext.Provider value={{ settings, updateSettings }}>
      {children}
    </SettingsContext.Provider>
  );
};

export const useSettings = () => useContext(SettingsContext);

This makes your settings global.

3️⃣ Wrap your app with the provider


// App.js
import { SettingsProvider } from "./settings/SettingsContext";

export default function App() {
  return (
    <SettingsProvider>
        <MainNavigator />
    </SettingsProvider>
  );
}

Now every screen can read/update global settings.

4️⃣ Use global settings anywhere

Example: Apply theme globally


                import { View, Text } from "react-native";
import { useSettings } from "../settings/SettingsContext";

export default function HomeScreen() {
  const { settings } = useSettings();

  return (
    <View
      style={{
        flex: 1,
        backgroundColor: settings.theme === "dark" ? "#000" : "#fff"
      }}
    >
        <Text style={{ color: settings.theme === "dark" ? "#fff" : "#000" }}>
        Home Screen
        </Text>
    </View>
  );
}

5️⃣ Update settings from a Settings screen


import { Switch, Button } from "react-native";
import { useSettings } from "../settings/SettingsContext";

export default function SettingsScreen() {
  const { settings, updateSettings } = useSettings();

  return (
    <>
        <Switch
        value={settings.notificationsEnabled}
        onValueChange={(value) =>
          updateSettings({ notificationsEnabled: value })
        }
      />

    <Button
        title="Dark Mode"
        onPress={() => updateSettings({ theme: "dark" })}
      />
    </>
  );
}

6️⃣ Persist settings (important in real apps)


import AsyncStorage from "@react-native-async-storage/async-storage";

const saveSettings = async (settings) => {
  await AsyncStorage.setItem("APP_SETTINGS", JSON.stringify(settings));
};

Load them on app startup and override defaults.

When to use global settings in React Native

✅ Good use cases

  • Theme
  • Language / locale
  • User preferences
  • Feature flags
  • API base URL (non-secret)
  • Accessibility settings

❌ Bad use cases

  • Temporary UI state (modals, input text)
  • Screen-specific state
  • Large dynamic datasets

Production note (important)

For larger apps (like your TryCastanet platform), global settings often evolve into:

  • Context + AsyncStorage
  • Or Zustand / Redux Toolkit
  • Or Expo’s Appearance + custom theme system

What this function does


const updateSettings = (updates) => {
    setSettings((prev) => ({ ...prev, ...updates }));
};

✅ It updates part of a settings object in React state

✅ It preserves existing values

✅ It safely handles async state updates

Step-by-step explanation

  1. setSettings

    This is the state setter from useState:

    
const [settings, setSettings] = useState({
    theme: 'light',
    notifications: true,
    language: 'en',
});
  2. Functional update (prev) 
    
setSettings((prev) => ...)
    • React gives you the latest state value
    • Prevents bugs when multiple updates happen quickly
  3. Object spread 
    
{ ...prev, ...updates }
    • Copies all existing settings
    • Overwrites only the keys provided in updates

Example usage

Update one setting


updateSettings({ theme: 'dark' });

Before


{ theme: 'light', notifications: true, language: 'en' }

After


{ theme: 'dark', notifications: true, language: 'en' }

Update multiple settings


updateSettings({
    notifications: false,
    language: 'fil',
});

Result:


{ theme: 'dark', notifications: false, language: 'fil' }

Why this pattern is recommended

✔ Avoids mutating state

✔ Safe for concurrent rendering (React 18+)

✔ Scales well for global settings (Context, Redux-like patterns)

Common real-world use cases (React / React Native)

  • App preferences (theme, language, timezone)
  • User profile edits
  • Feature toggles
  • Form state management
  • Global config objects

⚠️ Important limitation (nested objects)

This is a shallow merge.

❌ This will overwrite nested objects entirely:


updateSettings({
    profile: { name: 'Paul' }
});

If you need deep updates, you must spread nested levels:


updateSettings({
    profile: {
        ...settings.profile,
        name: 'Paul',
    },
});

(or use useReducer for complex state)

Pro tip (TypeScript)


const updateSettings = (updates: Partial) => {
    setSettings(prev => ({ ...prev, ...updates }));
};
