Penetration‑testing — practical playbook and checklist

Nice — here’s a compact, practical, and usable penetration‑testing playbook you can apply to engagements or learning. I can’t browse the web right now, so everything below sticks to stable, broadly relevant practices, tools, and examples (no time‑sensitive references). If you want a version tailored to a specific target type (web app, cloud, AD, IoT), say which and I’ll adapt it right away.

1. High‑level phases (do these in order)

  1. Engagement setup / legal — written authorization, scope, allowed tools/techniques, safe hours, escalation contacts, data handling rules.
  2. Reconnaissance (passive → active) — gather public info (DNS, subdomains, credentials leaks, OSINT), then active discovery (port/host discovery).
  3. Enumeration & scanning — identify services, versions, directories, API endpoints, user accounts, etc.
  4. Vulnerability analysis — map findings to candidate vulnerabilities and prioritize by impact/likelihood.
  5. Exploitation / validation (only if in scope) — safely confirm vulnerabilities; avoid destructive tests unless explicitly allowed.
  6. Post‑exploitation (if permitted) — persistence, lateral movement, data access validated per scope.
  7. Reporting & remediation verification — deliver executive summary + technical report, include mitigation steps and retest if agreed.

2. Rules of engagement (ROE) checklist — must be documented

  • Authorized targets (IP ranges, hostnames) and excluded assets (production printers, safety systems).
  • Time windows and contact person(s) for emergencies.
  • Allowed techniques (social engineering? physical testing?) — explicit yes/no.
  • Data handling & storage rules (sensitive data treatment, exfiltration limits).
  • Agreed deliverables (format, level of detail, retest policy).
  • Liability/insurance and incident handling process.

3. Recon & enumeration — prioritized tasks

  • Passive OSINT: domain WHOIS, public subdomains, certificate transparency, employee names, public GitHub leaks.
  • Subdomain discovery & DNS mapping.
  • Port & service discovery for live hosts.
  • Web enumeration: directory brute force, hidden endpoints, common parameters.
  • Technology fingerprinting: frameworks, server types, exposed APIs.
  • Credential/credential stuffing checks only if explicitly allowed.

Example (non‑destructive) command snippets you’ll often use:

  • nmap -sS -sV -p- -T4 --open target.com (TCP SYN + service/version scan, show open ports)
  • nmap -sU -sV -pU:53,161 target (UDP where needed)
  • gobuster dir -u https://example.com -w /path/wordlist.txt -x php,html,txt (web directory discovery)

    (Use these only with authorization and adapt flags to the environment.)

4. Web app focus — core checklist

  • Authentication / session management testing (weak auth, session fixation).
  • Input validation & output encoding (SQLi, XSS, command injection).
  • Access control: horizontal & vertical.
  • Logic flaws and API abuse (rate limits, object IDs, parameter manipulation).
  • File upload handling and insecure direct object references (IDOR).
  • CSRF protections and CORS misconfigurations.

    Tools commonly used: intercepting proxy (Burp/OWASP ZAP), scanners, curl, browser devtools.

5. Network & infrastructure focus — core checklist

  • Unpatched services (versions and known CVEs).
  • Weak/default credentials, unsecured management interfaces.
  • Misconfigured VPNs, exposed internal services.
  • Weak SMB/NetBIOS/LDAP configurations.
  • Segmentation validation — can you reach sensitive network zones?

    Tools: Nmap, Nessus/OpenVAS (for automated scanning), Wireshark for traffic analysis, Metasploit for proof‑of‑concept (only if in scope).

6. Active Directory / enterprise (if in scope)

  • Enumerate users, groups, trusts, ACLs.
  • Identify overly permissive groups and service accounts.
  • Look for Kerberoasting, delegation misconfigs, unconstrained delegation (conceptually — validate per scope).
  • Map privilege escalation and lateral movement paths (e.g., with graphing tools).

    Always follow strict ROE — AD testing can be disruptive.

7. Cloud & containers (if in scope)

  • Check IAM policies and over‑permissive roles.
  • Misconfigured storage (open S3/GCS buckets).
  • Publicly exposed management consoles or keys in code.
  • Container image vulnerabilities and insecure orchestration configs.

    Use cloud provider audit tools and IaC scanning in combination with manual checks.

8. Post‑exploitation cautions & limits

  • Never exfiltrate production data unless explicitly allowed and necessary for proof.
  • Prefer screenshots or hashed evidence over downloading large datasets.
  • Avoid actions that cause downtime or data corruption unless scoped.
  • Keep detailed logs of every action and time stamp everything for reporting and legal traceability.

9. Risk rating & reporting — what to include

  • Executive summary — business impact, highest risks, remediation priorities.
  • Technical findings — clear title, affected asset, CVSS (if used), evidence (screenshots/logs), reproduction steps (concise), remediation steps.
  • Remediation timeline & verification steps — how to check the fix.
  • Appendices — tools used, IPs scanned, raw logs (sensitive info redacted), compliance mapping if requested.

Quick template for a finding:

  • Title: [Short descriptor]
  • Severity: High / Medium / Low (justify)
  • Affected asset: IP / URL / hostname
  • Summary: what the issue is and business impact
  • Evidence: screenshot, request/response, log excerpt (redact sensitive data)
  • Reproduction: steps to reproduce (concise)
  • Recommendation: detailed fix and references

10. Tools & learning resources (stable list)

  • Recon/Scanning: Nmap, Amass (subdomain mapping).
  • Web testing: Burp Suite, OWASP ZAP, Nikto, SQLmap (authorized use only).
  • Exploitation: Metasploit (use only when allowed).
  • Packet analysis: Wireshark.
  • AD: BloodHound, PowerView (use per scope).
  • Binary/RE: Ghidra, radare2.
  • Passwords: Hashcat, John the Ripper (offline only).
  • Labs/Practice: TryHackMe, Hack The Box, OWASP Juice Shop, CTFs.
  • Methodologies: OWASP Top Ten, MITRE ATT&CK, NIST 800‑115.

11. Learning / certification path (practical)

  • Fundamentals: TCP/IP, Linux, scripting (Python/Bash), HTTP, databases.
  • Practical labs: start with web app CTFs and progressively do network and AD labs.
  • Certifications if desired: OSCP, GPEN, OSWE — they teach practical, methodological testing and reporting.

12. Quick starter checklist you can print

  • Written authorization & scope signed
  • Emergency contact & blackout windows defined
  • Recon: subdomains, certs, public leaks done
  • Scanning: ports & services identified, non‑intrusive first
  • Web enumeration: directories, endpoints, APIs listed
  • Vulnerability mapping: prioritized list created
  • Exploitation: PoC created only within scope
  • Report: exec summary + technical findings drafted
  • Retest / verification plan agreed
