How browsers render HTML and Canvas

Below is a clear, end-to-end explanation of how browsers render HTML and Canvas, from bytes → pixels, simplified but detailed.

How Browsers Render HTML & Canvas

Browsers have two main rendering paths:

  1. HTML / CSS rendering (DOM-based)
  2. Canvas rendering (immediate-mode drawing)

They share the GPU, but work very differently.

PART 1: How Browsers Render HTML (DOM + CSS)

High-Level Flow


HTML → DOM
CSS  → CSSOM
DOM + CSSOM → Render Tree
Render Tree → Layout
Layout → Paint
Paint → GPU → Screen

1. Download & Parse HTML

Browser downloads HTML bytes and:

  • Reads text top-to-bottom
  • Converts tags into nodes

Result → DOM Tree


<div>
    <p>Hello</p>
</div>

Becomes:


DIV
 └─ P
     └─ "Hello"

2. Parse CSS → CSSOM

CSS is parsed into rules:


p {
    color: red;
    font-size: 16px;
}

Becomes a CSS Object Model (CSSOM):

  • Selectors
  • Properties
  • Values

3. DOM + CSSOM → Render Tree

Browser combines:

  • DOM (structure)
  • CSSOM (styles)

Rules:

  • Invisible elements (display:none) are removed
  • Only visible nodes remain

📌 Render Tree knows:

  • What to draw
  • How it should look

4. Layout (Reflow)

Browser calculates:

  • Width / height
  • X / Y position
  • Box model (margin, border, padding)

This step answers:

“Where does each element go on the screen?”

⚠️ Expensive operation

(Resizing window or changing layout causes reflow)

5. Paint (Drawing Commands)

Browser converts elements into draw commands:

  • Text → glyphs
  • Borders → rectangles
  • Backgrounds → fills
  • Images → bitmaps

Example:


Draw rect at (x,y)
Draw text at (x,y)
Draw image at (x,y)

6. Layers & Compositing (GPU Acceleration)

Some elements become layers:

  • position: fixed
  • transform
  • opacity
  • video
  • canvas

Each layer:

  • Painted separately
  • Uploaded to GPU

GPU:

  • Composes layers
  • Applies transforms
  • Handles animations efficiently

📌 This avoids full repaints.

7. Final Display

  • GPU sends final frame to the screen
  • Happens ~60 times per second

PART 2: How Browsers Render Canvas

Canvas is very different.

Key Difference

HTML Canvas
Retained mode Immediate mode
Browser remembers elements Browser forgets after draw
DOM nodes Pixels only
Easy interaction Manual interaction

1. Create a Canvas


<canvas id="c" width="300" height="150"></canvas>

Browser:

  • Allocates a pixel buffer
  • Exposes drawing API

2. JavaScript Issues Draw Commands


ctx.fillStyle = "red";
ctx.fillRect(10, 10, 100, 50);

This means:

“Paint these pixels right now”

3. Immediate Mode Rendering

Once drawn:

  • Browser does not remember shapes
  • Only pixels exist

To animate:


ctx.clearRect(0,0,w,h);
drawAgain();

You redraw everything every frame.

4. Canvas Rendering Pipeline

CPU Path (2D Canvas)

  • JS → CPU rasterization
  • Writes pixels into bitmap
  • Bitmap uploaded to GPU

GPU Path (WebGL / WebGPU)

  • JS → GPU shaders
  • Same pipeline as games
  • Much faster for complex scenes

5. Canvas vs WebGL vs WebGPU

Tech How it works
Canvas 2D CPU → bitmap
WebGL OpenGL ES → GPU
WebGPU Modern GPU API

PART 5: HTML vs Canvas — When to Use What

Use HTML/CSS When:

  • UI
  • Forms
  • Text
  • Accessibility
  • SEO

Use Canvas When:

  • Games
  • Charts
  • Image editing
  • Visualizations
  • Particle effects

PART 6: Mental Model

HTML Rendering

“Describe what exists → browser figures out drawing”

Canvas Rendering

“Tell browser how to draw pixels → you manage everything”

One-Sentence Summary

HTML rendering is declarative and retained (DOM-based), while Canvas is imperative and immediate (pixel-based), both ultimately using the GPU to turn draw commands into pixels on your screen.
