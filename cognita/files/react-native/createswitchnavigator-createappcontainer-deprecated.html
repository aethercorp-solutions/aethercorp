createSwitchNavigator and createAppContainer come from React Navigation v3 / v4. They were commonly used to control high-level app flow (auth, splash, onboarding) where screens do not go back to each other.

Since you‚Äôre reviewing older navigation code (like your SwitchNavigator example earlier), here‚Äôs a deep, structured explanation of usage, internals, and when to use it.

1. What createSwitchNavigator Is

createSwitchNavigator creates a navigator that:

  • Shows only one screen at a time
  • Does NOT keep navigation history
  • Does NOT allow back navigation
  • Immediately switches between routes

Think of it as a state machine, not a stack.

Common use cases

  • Splash screen
  • Authentication flow
  • Walkthrough / onboarding
  • Deciding between logged-in vs logged-out UI

2. What createAppContainer Is

createAppContainer:

  • Wraps the root navigator
  • Manages navigation state
  • Connects navigation to the React root
  • Handles deep linking, state persistence, back button handling (Android)

In v3/v4, every app must have exactly one AppContainer.

3. Basic Usage Example


import { createSwitchNavigator, createAppContainer } from 'react-navigation';
import AuthScreen from './AuthScreen';
import AppScreen from './AppScreen';
import SplashScreen from './SplashScreen';

const RootNavigator = createSwitchNavigator(
  {
    Splash: SplashScreen,
    Auth: AuthScreen,
    App: AppScreen,
  },
  {
    initialRouteName: 'Splash',
  }
);

export default createAppContainer(RootNavigator);

4. How the Navigation Flow Works

Step-by-step flow

  1. App launches
  2. AppContainer mounts
  3. SwitchNavigator shows initialRouteName
  4. Inside SplashScreen, you decide where to go
  5. You navigate once and cannot go back

Example inside SplashScreen:


componentDidMount() {
    const isLoggedIn = checkAuth();
    
    this.props.navigation.navigate(
        isLoggedIn ? 'App' : 'Auth'
    );
}

6. Why SwitchNavigator Has No Back Button

Design reason

  • Prevent users from navigating back to:
    • Login after login
    • Splash after app load
    • Onboarding after completion

Internally

SwitchNavigator:

  • Replaces the entire route state
  • Does not push routes onto a stack
  • Clears previous routes on navigation

7. Navigation Methods Available

Even though it behaves differently, it still uses:


this.props.navigation.navigate('RouteName');

But not recommended:


goBack()
pop()
push()

Because there is no stack.

10. Common Mistakes

‚ùå Using SwitchNavigator for normal screens

Bad for:

  • Forms
  • Detail pages
  • Settings

‚ùå Expecting header/back behavior

SwitchNavigator:

  • Has no header
  • Has no gestures
  • Has no history

11. Why It Was Removed in React Navigation v5+

React Navigation v5+ replaced it with:


<NavigationContainer>
    {isLoggedIn ? <AppStack /> : <AuthStack />}
</NavigationContainer>

This:

  • Uses React state
  • Is more declarative
  • Easier to reason about

12. When You SHOULD Use It (v4 Apps)

Use createSwitchNavigator if:

  • You maintain a legacy v4 app
  • You need hard route separation
  • You want no back navigation
