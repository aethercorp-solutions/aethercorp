Handling Server Trust

"Handling Server Trust" typically refers to how a client (like a mobile app or web application) determines whether to trust a server it's communicating with, especially over HTTPS. This is critical to prevent man-in-the-middle attacks and ensure secure communication.

Below is a breakdown of what “Handling Server Trust” involves, especially in the context of platforms like iOS (Swift), Android (Java/Kotlin), or web apps:

General Concept

When a client connects to a server using HTTPS, it:

  1. Receives the server’s SSL/TLS certificate.
  2. Validates the certificate’s trust chain.
  3. Ensures the certificate is not expired or revoked.
  4. Optionally checks that the server’s domain matches the certificate.

iOS (Swift): Handling Server Trust

In iOS (using URLSession), you can handle server trust like this:


let session = URLSession(
    configuration: .default,
    delegate: self,
    delegateQueue: nil
)

extension YourClass: URLSessionDelegate {
    func urlSession(_ session: URLSession,
                    didReceive challenge: URLAuthenticationChallenge,
                    completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) {

        let serverTrust = challenge.protectionSpace.serverTrust

        // Example: Trust all certificates (NOT recommended for production)
        if let serverTrust = serverTrust {
            let credential = URLCredential(trust: serverTrust)
            completionHandler(.useCredential, credential)
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}

Important:

  • Never skip validation in production.
  • Use certificate pinning or public key pinning for higher security.

Android (Java/Kotlin): Handling Server Trust

You can create a custom TrustManager, for example:


TrustManager[] trustAllCerts = new TrustManager[]{
    new X509TrustManager() {
        public java.security.cert.X509Certificate[] getAcceptedIssuers() { return new X509Certificate[0]; }
        public void checkClientTrusted(X509Certificate[] certs, String authType) {}
        public void checkServerTrusted(X509Certificate[] certs, String authType) {}
    }
};

Then use it to build an SSLContext.

Warning:

  • Trusting all certificates is dangerous unless it's strictly for development or testing.
  • Use Network Security Configuration to define trusted CAs and pinning policies.

Web: Handling Server Trust

Browsers handle server trust automatically via the OS trust store. As a web developer, you should:

  • Use HTTPS via a valid certificate.
  • Configure HSTS headers.
  • Consider Subresource Integrity (SRI) for third-party scripts.

Best Practices

  • Use valid SSL/TLS certificates from trusted Certificate Authorities (CAs).
  • Implement certificate pinning where applicable.
  • Handle certificate revocation (OCSP or CRLs) if required.
  • Don’t disable trust checks in production code.
  • Log and monitor SSL errors or certificate failures.
