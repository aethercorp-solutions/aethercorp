Firebase Crashlytics in a development build in Xcode

  1. Crashlytics Works in Dev Mode

    Crashlytics will report crashes and non-fatal issues in development builds. This is helpful for debugging and testing before production releases.

  2. Symbolication Still Happens

    As long as you're uploading your dSYM files (which Xcode usually does automatically with the Firebase SDK), the crashes will be symbolicated just like in release builds.

  3. Set a Custom Build Type / Environment

    To help differentiate dev vs prod crashes in the Crashlytics dashboard, it‚Äôs a good idea to set a custom key or log:

    
Crashlytics.crashlytics().setCustomValue("development", forKey: "build_environment")

    Or:

    
Crashlytics.crashlytics().log("Running development build")
  4. Force a Test Crash

    To verify it's working in dev, you can force a test crash:

    
Crashlytics.crashlytics().log("Testing crash in dev")
fatalError("Test crash for dev build")
  5. Crashes May Not Appear Instantly

    Crash reports are not sent immediately ‚Äî they are sent on the next app launch after a crash, and it can take a few minutes for them to show up in the Firebase Console.

  6. Make Sure Debug Symbols Are Uploaded

    For dev builds, Debug Info must be present. If symbolication isn‚Äôt working:

    • Check that DWARF with dSYM File is set in Build Settings.
    • Ensure dSYMs are uploaded (automatic with Firebase Run Script).

‚úÖ What Is a Release Build?

In the context of iOS development with Xcode, a release build typically refers to a version of your app built using the Release configuration, as opposed to the Debug configuration.

üì¶ 1. "Release Build" (Technical Definition in Xcode)

  • In Xcode, there are usually two build configurations:
    • Debug ‚Üí used for development, with extra logs, debugger support, etc.
    • Release ‚Üí optimized for performance, with debugging disabled.
  • You can view/edit these in: 
    
Xcode > Project > Build Settings > Configurations

So, a release build is just:

A version of the app built using the Release configuration ‚Äî usually for TestFlight, App Store, or production use.

üì§ 2. Is a Release Build Only One Uploaded to the App Store?

No. A release build doesn't have to be uploaded to the App Store. It could also be:

  • Uploaded to TestFlight
  • Shared with testers via ad hoc distribution
  • Installed locally with tools like Xcode Organizer or Diawi
  • Shared using Firebase App Distribution

The key is: if the app was built using the Release configuration, it‚Äôs considered a release build, regardless of whether it‚Äôs live on the App Store.

üß™ 3. What About Dev Builds?

Dev (development) builds are typically:

  • Built with the Debug configuration
  • Run locally in the Simulator or on a connected device via Xcode
  • Have debugging and logging enabled
  • Not optimized for performance
