SwiftUI navigation

SwiftUI navigation has evolved a lot, so I’ll walk you through it cleanly—from concept → old way → modern way → real patterns, with mental models that actually stick.

1️⃣ What “navigation” means in SwiftUI

Navigation in SwiftUI is state-driven.

You don’t say:

“Push this view now”

You say:

“When this state becomes true or this value exists, show that view”

SwiftUI watches state → updates UI → handles the transition.

This is the single most important mindset shift.

2️⃣ The old world (iOS 13–15)

NavigationView + NavigationLink


NavigationView {
    NavigationLink("Go to Details") {
        DetailView()
    }
}

Problems:

  • Hard to control programmatically
  • Deep linking = painful
  • Weird behavior on iPad / split view
  • State & navigation tightly coupled to UI layout

Apple deprecated this approach conceptually, though it still works.

3️⃣ The modern system (iOS 16+)

NavigationStack

This is the current, correct way.


NavigationStack {
    Text("Home")
}

Think of NavigationStack as:

A stack of data, not views

SwiftUI derives views from that data.

4️⃣ Push navigation (simple)


NavigationStack {
    NavigationLink("Go to Details", value: "details")
        .navigationDestination(for: String.self) { value in
            DetailView(id: value)
        }
}

What’s happening:

  • value is pushed onto the stack
  • SwiftUI decides which screen to show
  • Navigation is type-safe

5️⃣ Programmatic navigation (the big win)

Using a path


@State private var path = NavigationPath()

NavigationStack(path: $path) {
    Button("Go to Details") {
        path.append("details")
    }
    .navigationDestination(for: String.self) { value in
        DetailView(id: value)
    }
}

Now you can:

  • Push
  • Pop
  • Reset navigation

path.removeLast()     // pop
path = NavigationPath() // go back to root

This is how real apps navigate.

6️⃣ Navigation with models (best practice)

Instead of strings:


struct User: Hashable {
    let id: Int
}                
            

path.append(User(id: 42))
            

.navigationDestination(for: User.self) { user in
    UserProfileView(user: user)
}

Why this is powerful:

  • Compile-time safety
  • Deep linking friendly
  • Serializable navigation state
  • Perfect for Redux / MVVM / TCA

7️⃣ iPad & Mac: NavigationSplitView

For sidebar layouts:


NavigationSplitView {
    SidebarView()
} detail: {
    DetailView()
}

This replaces the mess that NavigationView used to create.

8️⃣ Toolbar & title (modern way)


.navigationTitle("Home")
.toolbar {
    ToolbarItem(placement: .navigationBarTrailing) {
        Button("Add") {}
    }
}

No more navigationBarItems.

9️⃣ Common mistakes (and how to avoid them)

❌ Navigating inside onAppear

❌ Using Bool flags for every screen

❌ Embedding NavigationStack inside another

❌ Thinking imperatively (“push now”)

✅ Single root NavigationStack

✅ Navigate via data

✅ Centralize navigation state

✅ Let SwiftUI react to changes

🔟 Mental model that clicks

UIKit

ViewController pushes ViewController

SwiftUI

State changes → navigation updates → views appear

If navigation feels hard, it’s usually because:

You’re thinking like UIKit in a SwiftUI world

Alright, let’s really unpack SwiftUI navigation, from the old world to the modern way, and explain what’s actually happening under the hood—not just “use this API” but why it works. 🧭

I’ll go bottom → top, because that’s how SwiftUI itself thinks.

1. What “navigation” means in SwiftUI (conceptually)

Navigation in SwiftUI is state-driven UI composition.

You don’t “push” or “pop” views.

You change state, and SwiftUI recomputes the view tree.

UIKit:


navigationController.pushViewController(vc, animated: true)

SwiftUI:


path.append(User(id: 1))

2. The view tree & identity (the real foundation)

SwiftUI builds a view tree every time state changes.

Key ideas:

  • Views are value types (struct)
  • Navigation state = data
  • Destination views are derived from that data

If SwiftUI sees:

  • same view type
  • same identity

    ➡️ it keeps the view alive

    Otherwise ➡️ it destroys and recreates it

This is why Hashable & Identifiable matter so much in navigation.

3. Old navigation (iOS 13–15): NavigationView

Basic example


NavigationView {
    List {
        NavigationLink("Go to Detail") {
            DetailView()
        }
    }
}

How it works

  • NavigationView wraps UIKit’s UINavigationController
  • NavigationLink is both:
    • a button
    • a navigation instruction

Problems:

  • Hard to programmatically navigate
  • iPad split-view bugs
  • State-driven navigation was clunky
  • No real stack control

🚨 Apple now considers this legacy

4. Bridge era (iOS 14–15): isActive & tag/selection

Example:


NavigationLink(
    destination: DetailView(),
    isActive: $isActive
) {
    Text("Go")
}

This kind of introduced state-based navigation, but:

  • You had to hide links
  • Logic leaked into UI
  • Multiple destinations = nightmare

This pain led directly to…

5. Modern SwiftUI (iOS 16+): NavigationStack

This is the real SwiftUI navigation system.


NavigationStack {
    List(users) { user in
        NavigationLink(value: user) {
            Text(user.name)
        }
    }
    .navigationDestination(for: User.self) { user in
        UserDetailView(user: user)
    }
}

What changed?

Navigation is now:

  • data-driven
  • type-safe
  • stack-based

6. NavigationStack under the hood

The stack is just data


@State private var path: [User] = []

Each element in path:

  • Must conform to Hashable
  • Represents one screen

SwiftUI does:


[path] → destination resolver → view

No view pushing.

Just state → UI projection.

7. NavigationPath (type-erased stacks)

When you need multiple destination types:


@State private var path = NavigationPath()
            

path.append(User(id: 1))
path.append(Settings())

And destinations:


.navigationDestination(for: User.self) { user in
    UserDetailView(user: user)
}
.navigationDestination(for: Settings.self) { _ in
    SettingsView()
}

Why NavigationPath exists

  • Allows heterogeneous navigation
  • Runtime flexibility
  • Serializable (for deep linking & state restoration)

8. Programmatic navigation (the clean way)

Push


path.append(user)

Pop one


path.removeLast()

Pop to root


path.removeLast(path.count)

Replace stack


path = [home, profile]

This is exactly how deep linking works.

9. Navigation ≠ presentation

Navigation Presentation
NavigationStack sheet
Push/pop Modal
Back button Swipe-down
Stack-based Overlay-based

SwiftUI keeps these strictly separate.

This separation allows:

  • Predictable UI
  • Clear mental model
  • Less UIKit-style chaos

10. iPad & macOS: NavigationSplitView


NavigationSplitView {
    Sidebar()
} detail: {
    DetailView()
}

This replaces:

  • NavigationView + size classes
  • UIKit split view controllers

SwiftUI automatically adapts:

  • iPhone → stack
  • iPad → sidebar + detail
  • macOS → multi-column

11. Navigation bar customization


.navigationTitle("Users")
.navigationBarTitleDisplayMode(.inline)
.toolbar {
    ToolbarItem(placement: .navigationBarTrailing) {
        Button("Add") {}
    }
}

Important rule:

Navigation modifiers apply to the current level in the stack

12. Deep linking (real-world example)


func handle(url: URL) {
    path.removeLast(path.count)
    path.append(User(id: url.userID))
}

This works because:

  • Navigation is just data
  • Views are reconstructed automatically

No manual controller juggling 🎉

13. Common mistakes (and why they break)

❌ Navigating with booleans everywhere


@State var showDetail = false

➡️ Breaks with multiple destinations

❌ Putting logic in NavigationLink

NavigationLink should be dumb.

❌ Non-stable Hashable

If hashValue changes → SwiftUI thinks it’s a new screen.

14. Mental model you should keep

NavigationStack is a pure function

Screens = f(Path)

Change the data → SwiftUI handles the rest.

If you think like UIKit, SwiftUI navigation feels painful.

If you think like state → UI, it becomes elegant.

Perfect choice. Let’s build a real SwiftUI app flow the modern way:

Auth → Home → Detail

✅ state-driven

✅ deep-link ready

✅ no hidden NavigationLinks

✅ scales to real apps

I’ll explain architecture first, then code, then why this works.

1. App flow as data (the key idea)

We define what screen the app is on as state, not UI actions.


enum AppRoute: Hashable {
    case login
    case home
    case detail(Item)
}

This is your entire navigation model.

2. App entry point (root coordinator)

This replaces:

  • AppDelegate routing
  • UIKit coordinators
  • “if user is logged in” hacks

@main
struct MyApp: App {

    @State private var path = NavigationPath()
    @State private var isAuthenticated = false

    var body: some Scene {
        WindowGroup {
            NavigationStack(path: $path) {
                Group {
                    if isAuthenticated {
                        HomeView(path: $path)
                    } else {
                        LoginView(isAuthenticated: $isAuthenticated)
                    }
                }
            }
        }
    }
}

What’s happening

  • Auth state decides the root
  • NavigationStack stays alive
  • No navigation glitches when login status changes

3. Login screen (auth → home)


struct LoginView: View {

    @Binding var isAuthenticated: Bool

    var body: some View {
        VStack(spacing: 20) {
            Text("Welcome")
                .font(.largeTitle)

            Button("Login") {
                // simulate auth success
                isAuthenticated = true
            }
        }
    }
}

🔑 No navigation here

➡️ Auth success simply changes state

4. Home screen (list → detail)


struct HomeView: View {

    @Binding var path: NavigationPath

    let items: [Item] = [
        Item(id: 1, name: "Apple"),
        Item(id: 2, name: "Banana")
    ]

    var body: some View {
        List(items) { item in
            NavigationLink(value: item) {
                Text(item.name)
            }
        }
        .navigationTitle("Home")
        .navigationDestination(for: Item.self) { item in
            DetailView(item: item, path: $path)
        }
    }
}

Why this is clean

  • NavigationLink only describes intent
  • Actual destination is declared once
  • Easy to add more routes later

5. Detail screen (pop / logout / deep logic)


struct DetailView: View {

    let item: Item
    @Binding var path: NavigationPath

    var body: some View {
        VStack(spacing: 16) {
            Text(item.name)
                .font(.title)

            Button("Go Back") {
                path.removeLast()
            }

            Button("Pop to Root") {
                path.removeLast(path.count)
            }
        }
        .navigationTitle("Detail")
    }
}

6. Model (Hashable + Identifiable = stable navigation)


struct Item: Identifiable, Hashable {
    let id: Int
    let name: String
}

🔐 Why both?

  • Identifiable → List diffing
  • Hashable → Navigation identity

7. Logout flow (real-world important)

Add logout anywhere:


Button("Logout") {
    path.removeLast(path.count)
    isAuthenticated = false
}

This:

  1. Clears navigation stack
  2. Returns to login
  3. Avoids ghost screens

8. Deep linking ready (zero extra UI work)


func openItem(_ item: Item) {
    path.removeLast(path.count)
    path.append(item)
}

This works because:

  • Navigation = data
  • SwiftUI rebuilds screens automatically
