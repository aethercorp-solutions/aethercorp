Smooth App Review approval

If your goal is smooth App Review approval, the setup for Firebase + Apple Sign-In isn’t just about making it work—it’s about meeting Apple Inc.’s strict review rules (especially around login + account access).

Here’s the battle-tested setup (2026) used by teams to avoid rejection 👇

🍎🔥 Best Setup: Firebase + Apple Sign-In (Approval-Safe)

🧩 1. Use Firebase Auth as the single source of truth

Use Firebase Authentication to unify all providers:

  • Apple Sign-In
  • Google Sign-In
  • Email/password (optional)

👉 Result:

  • One UID per user
  • Easier account linking
  • Cleaner backend (important for review + compliance)

🍏 2. Proper Apple Sign-In Flow (Critical)

Use:

  • @invertase/react-native-apple-authentication

Flow:

  1. Request Apple login
  2. Get:
    • identityToken
    • authorizationCode
  3. Pass token to Firebase:

const appleCredential = auth.AppleAuthProvider.credential(identityToken, nonce);
await auth().signInWithCredential(appleCredential);

⚠️ 3. Handle Apple Private Email (VERY IMPORTANT)

Apple may return:


example@privaterelay.appleid.com

Required:

  • Treat it as a real email
  • Do NOT force user to enter another email
  • Store it in your DB

👉 Common rejection:

“App requires additional information beyond Apple Sign-In”

🔗 4. Account Linking Strategy (Avoid Duplicate Accounts)

If user signs in with:

  • Apple first
  • Then Google (same email)

👉 Link accounts:


await currentUser.linkWithCredential(googleCredential);

Why this matters:

  • Prevents duplicate users
  • Apple reviewers test this

🧪 5. App Review Requirement: Demo/Test Account

This is where most devs get rejected.

✅ BEST PRACTICE (recommended)

Provide BOTH:

Option A — Apple Sign-In works in review

  • Ensure Apple login works without:
    • OTP
    • External device
  • No SMS/email verification required

Option B — Demo account (backup)

Provide in App Store Connect:


Username: reviewer@test.com
Password: Test1234!

❌ Avoid:

  • OTP-required login
  • Geo-restricted login
  • Paid login access

🔐 6. If You Have OTP (Firebase Phone Auth)

Apple WILL reject if:

  • Reviewer cannot receive OTP

Fix:

  • Disable OTP for review account
  • OR whitelist Apple reviewers

👉 Best:

Provide a non-OTP login path

🧾 7. Required UI/UX for Apple Compliance

MUST HAVE:

  • “Sign in with Apple” button:
    • Same prominence as Google login
  • Use official Apple button style

🗑️ 8. Account Deletion (MANDATORY 2026 RULE)

Apple now requires:

Users must be able to delete their account inside the app

You must:

  • Delete Firebase user:

await auth().currentUser.delete();
  • Delete backend data (MongoDB, etc.)

🔒 9. Backend Verification (Recommended)

In your AWS Lambda:

  • Verify Apple token (optional but strong security)
  • Or trust Firebase ID token:

const decoded = await admin.auth().verifyIdToken(idToken);

⚠️ 10. Common Apple Rejections (REAL CASES)

❌ 1. Apple Sign-In not working

  • Sandbox misconfigured
  • Missing capability in Xcode

❌ 2. OTP required

  • Reviewer blocked

❌ 3. Asking email after Apple login

  • Instant rejection

❌ 4. No account deletion

  • Automatic rejection

❌ 5. Button not visible enough

  • UI violation

🏗️ Recommended Architecture (Your Stack)

Since you're using:

  • React Native
  • AWS Lambda
  • MongoDB

Ideal flow:


React Native
   ↓
Apple Sign-In / Google
   ↓
Firebase Auth (central identity)
   ↓
Firebase ID Token
   ↓
AWS Lambda (verify token)
   ↓
MongoDB user record

🚀 Pro Setup (Used by Production Apps)

  • Firebase Auth = Identity
  • AWS = Business logic
  • MongoDB = user profile
  • Apple Sign-In = compliance
  • Google Sign-In = convenience

✅ Final Approval Checklist

  • Apple Sign-In works without OTP
  • No extra info required after login
  • Demo account provided
  • Account deletion implemented
  • Apple button UI compliant
  • Backend verifies Firebase token
