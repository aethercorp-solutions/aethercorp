Type Erasure

Type erasure in Swift is a technique used to hide the concrete type of a value that conforms to a protocol, while still exposing the protocol's interface. It's commonly used when you want to store heterogeneous types that conform to the same protocol in a uniform way (like in an array or as return types).

Why use Type Erasure?

Swift protocols with associated types or Self requirements (e.g., IteratorProtocol, View, etc.) can't be used as existential types (e.g., you can't say let a: SomeProtocol = someValue if the protocol has associated types). Type erasure gets around this by wrapping the conforming types in a box that hides the underlying type.

Example Use Case: AnySequence

The standard library provides a type-erased sequence wrapper called AnySequence. You can use it to store any sequence without knowing the underlying type.


func makeSequence() -> AnySequence {
    let array = [1, 2, 3]
    return AnySequence(array)
}

Here, Array is type-erased to AnySequence.

How to Implement Type Erasure Manually

Let’s say you have a protocol:


protocol Animal {
    func speak() -> String
}

You want to store multiple types (e.g., Dog, Cat) that conform to Animal in one array:


struct Dog: Animal {
    func speak() -> String { "Woof" }
}

struct Cat: Animal {
    func speak() -> String { "Meow" }
}

let animals: [Animal] = [Dog(), Cat()] // ❌ if `Animal` has associated type or Self


let animals: [Animal] = [Dog(), Cat()] // ❌ if `Animal` has associated type or Self

But with type erasure:

Step 1: Create the type-erased wrapper


class AnyAnimal: Animal {
    private let _speak: () -> String

    init<A: Animal>(_ animal: A) {
        self._speak = animal.speak
    }

    func speak() -> String {
        return _speak()
    }
}

Step 2: Use the wrapper


let animals: [AnyAnimal] = [AnyAnimal(Dog()), AnyAnimal(Cat())]

for animal in animals {
    print(animal.speak())  // Woof, Meow
}

Use With Protocols That Have Associated Types

Say you have a protocol like:


protocol DataSource {
    associatedtype Data
    func getData() -> Data
}

You can't use [DataSource], but you can do:


class AnyDataSource<Output>: DataSource {
    private let _getData: () -> Output

    init<DS: DataSource>(_ dataSource: DS) where DS.Data == Output {
        self._getData = dataSource.getData
    }

    func getData() -> Output {
        _getData()
    }
}

Summary

  • Use type erasure when working with protocols with associated types or Self requirements.
  • Swift’s standard library already provides type-erased wrappers like AnySequence, AnyIterator, and AnyHashable.
  • You can implement your own wrappers using closures that forward protocol method calls.

Type Erasure in SwiftUI

Great — let's walk through type erasure in SwiftUI, specifically using AnyView, which is one of the most common real-world examples.

Problem in SwiftUI

SwiftUI views conform to the View protocol:


struct MyTextView: View {
    var body: some View {
        Text("Hello")
    }
}

But View has an associated type called Body, which means you can't do this:


func makeView(condition: Bool) -> View { // ❌ Error
    if condition {
        return Text("Yes")
    } else {
        return Image(systemName: "xmark")
    }
}

Swift doesn't know what concrete type to expect for the return.

Solution: Use AnyView (Type Erasure)

AnyView erases the underlying type and wraps any view:


func makeView(condition: Bool) -> AnyView {
    if condition {
        return AnyView(Text("Yes"))
    } else {
        return AnyView(Image(systemName: "xmark"))
    }
}

Now Swift is happy because the function always returns one concrete type: AnyView, regardless of what's inside.

How AnyView Works Internally (Simplified)

Here’s a conceptual implementation of AnyView:


struct AnyView: View {
    private let _view: () -> Any
    
    init<V: View>(_ view: V) {
        self._view = { view }
    }

    var body: some View {
        _view() as! some View // simplified for demo; real implementation is more complex
    }
}

Of course, in reality, AnyView is implemented in a highly optimized and opaque way in SwiftUI's internals, but conceptually it’s just wrapping a view in a box that hides the specific type.

Use Case: Arrays of Views

You can’t do this:


let views: [View] = [Text("A"), Image(systemName: "star")] // ❌

But you can:


let views: [AnyView] = [
    AnyView(Text("A")),
    AnyView(Image(systemName: "star"))
]

Then use them like:


VStack {
    ForEach(0..<views.count) { index in
        views[index]
    }
}

When Not to Use AnyView

  • Type erasure adds runtime overhead and removes compile-time layout optimization. Avoid using AnyView unless you really need to:
  • Store different views in a collection
  • Abstract view generation behind a function or protocol

Alternatives Before Reaching for AnyView

  • Use Group or TupleView if all branches return the same type.
  • Use view builders with @ViewBuilder which helps Swift infer types.

@ViewBuilder
func makeView(condition: Bool) -> some View {
    if condition {
        Text("Yes")
    } else {
        Image(systemName: "xmark")
    }
}

This avoids type erasure entirely.

Summary

Use Case Should You Use AnyView?
Return different views from a function Yes
Store heterogeneous views in an array Yes
Uniform layout, simple conditions Prefer @ViewBuilder
Performance-critical layout Avoid AnyView
