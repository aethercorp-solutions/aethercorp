Deep linking in SwiftUI

Deep linking in SwiftUI is how your app opens to a specific screen or state when it’s launched from outside the app—like a URL, email, push notification, QR code, or another app.

Think: “Don’t just open the app — open this exact page inside it.” 🔗

What a deep link looks like

Examples:

  • myapp://profile/42
  • https://trycastanet.com/invite/abc123
  • Push notification → “Open message #99”

All of those should land the user exactly where they expect, not just the home screen.

How deep linking works in SwiftUI (high level)

There are three layers:

  1. System – iOS hands your app a URL
  2. App entry – SwiftUI receives the URL
  3. Navigation – your app routes to the right screen

SwiftUI gives you .onOpenURL and .scenePhase to catch the link.

Basic deep link handling in SwiftUI

1️⃣ Register a URL scheme (Info.plist)


<key>CFBundleURLTypes</key>
<array>
    <dict>
    <key>CFBundleURLSchemes</key>
    <array>
        <string>myapp</string>
    </array>
    </dict>
</array>

Now myapp://something opens your app.

Nice question — this is one of those iOS things that looks tiny but is actually doing a lot under the hood. Let’s peel it down to bare metal 🧠⚙️

What this snippet actually is


<key>CFBundleURLTypes</key>
<array>
  <dict>
    <key>CFBundleURLSchemes</key>
    <array>
      <string>myapp</string>
    </array>
  </dict>
</array>

This lives in Info.plist and tells iOS:

“This app claims ownership of the URL scheme myapp://”

Once installed, your app becomes a URL handler at the OS level.

Think in OS terms (not app terms)

Step 1: App installation

When your app is installed:

  1. iOS reads Info.plist
  2. Extracts CFBundleURLTypes
  3. Registers the scheme myapp in a system-wide URL dispatch table

Conceptually (not literally):


URL Scheme Registry:
- myapp → com.yourcompany.yourapp

This registry lives in SpringBoard / LaunchServices, not your app.

Step 2: Someone opens a URL

Somewhere in the system:


myapp://profile/123?tab=posts

This can come from:

  • Safari
  • Messages
  • Mail
  • Another app
  • Terminal (simulator)
  • Universal link fallback
  • OAuth redirect (Google / Apple / Firebase)

Step 3: URL resolution (kernel → system services)

  1. The system parses the URL 
    
scheme = myapp
host   = profile
path   = /123
query  = tab=posts
  2. iOS asks:

    “Who owns the myapp scheme?”

    
com.yourcompany.yourapp
  3. LaunchServices responds:

Step 4: App launch mechanics

If the app is NOT running

iOS:

  • Forks a new process for your app
  • Loads the Mach-O binary
  • Initializes Objective-C / Swift runtime
  • Creates UIApplication
  • Calls app lifecycle hooks

If the app IS running

  • App stays alive
  • URL is delivered directly

Step 5: URL delivery to your code

UIKit (AppDelegate)


func application(
  _ app: UIApplication,
  open url: URL,
  options: [UIApplication.OpenURLOptionsKey : Any] = [:]
) -> Bool {
    // You get the URL here
    return true
}

Scene-based apps (modern iOS)


func scene(
  _ scene: UIScene,
  openURLContexts URLContexts: Set<UIOpenURLContext>
) {
    let url = URLContexts.first!.url
}

This is the first moment your code touches the URL.

What the system guarantees

By the time your code runs:

  • The app identity is verified
  • The scheme matches your Info.plist
  • The URL is syntactically valid
  • The caller is sandboxed (no shared memory)
  • Data is passed by value, not reference

No app can “fake” being your app unless it owns the same bundle ID + signature.

Security model (important)

Custom URL schemes are:

❌ NOT exclusive

❌ NOT cryptographically verified

❌ NOT secure for secrets

Any app can also declare:


<string>myapp</string>

If two apps declare the same scheme:

  • iOS picks one (undefined order)
  • You get scheme hijacking

That’s why OAuth prefers:

  • Universal Links
  • ASWebAuthenticationSession

Why OAuth uses this anyway

Example Google Sign-In flow:

  1. App opens browser: 
    
https://accounts.google.com/...
  2. User logs in
  3. Google redirects: 
    
myapp://oauth?code=abc123
  4. iOS:
    • Resolves myapp
    • Reopens your app
    • Passes the auth code
  5. App exchanges code → token securely over HTTPS

    The URL itself carries no secrets, only short-lived codes.

Bare-metal mental model

Think of it like:


Interrupt vector table (but for URLs)

URL scheme → App process

The OS:

  • Owns the table
  • Dispatches events
  • Starts processes
  • Delivers structured data

Your Info.plist entry is literally saying:

“Register an interrupt handler for scheme myapp”

Comparison to Universal Links

Feature Custom Scheme Universal Link
OS-level routing ✅ ✅
Domain ownership ❌ ✅ (SSL + apple-app-site-association)
App exclusivity ❌ ✅
OAuth recommended ❌ ✅
Simpler setup ✅ ❌

TL;DR “Too Long; Didn’t Read.” (but still technical)

  • CFBundleURLTypes registers a URL scheme handler at install time
  • iOS keeps a global scheme → app map
  • When myapp:// is opened, iOS:
    • Resolves the app
    • Launches or resumes it
    • Delivers the URL to lifecycle callbacks
  • This is OS-level dispatch, not app-to-app messaging
  • Powerful, simple, but not secure by itself

2️⃣ Catch the URL in SwiftUI


@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .onOpenURL { url in
                    print("Opened with URL:", url)
                }
        }
    }
}

At this point, you have the link, but you’re not navigating yet.

Real deep linking = navigation

SwiftUI deep linking usually works with NavigationStack + NavigationPath.

Central router pattern (recommended)


final class AppRouter: ObservableObject {
    @Published var path = NavigationPath()

    func handle(url: URL) {
        guard url.scheme == "myapp" else { return }

        let components = url.pathComponents

        if components.contains("profile"),
            let id = components.last {
            path.append(Route.profile(id: id))
        }
    }
}

Define routes:


enum Route: Hashable {
    case profile(id: String)
}

Hook it into the app


@main
struct MyApp: App {
    @StateObject private var router = AppRouter()

    var body: some Scene {
        WindowGroup {
            NavigationStack(path: $router.path) {
                HomeView()
                    .navigationDestination(for: Route.self) { route in
                        switch route {
                        case .profile(let id):
                            ProfileView(userID: id)
                        }
                    }
            }
            .onOpenURL { url in
                router.handle(url: url)
            }
        }
    }
}

Now:

  • myapp://profile/42 → opens ProfileView(userID: "42")

That’s real deep linking.

Universal Links (production-grade)

Instead of myapp://, you can use https links:


https://trycastanet.com/invite/abc123

Benefits:

  • Works even if the app is not installed
  • Opens website → App Store → app → correct screen
  • Required for many Apple Review flows

You’ll need:

  • Associated Domains capability
  • apple-app-site-association file on your server

SwiftUI handling is the same — still .onOpenURL.

Common deep link use cases

  • ✅ Email verification
  • ✅ Magic login links
  • ✅ Invite links
  • ✅ Push notification routing
  • ✅ OAuth callbacks (Google / Apple / Azure AD 👀)

(You’ve already brushed against this with Google Sign-In + Apple review.)

SwiftUI gotchas (important)

⚠️ App state matters

  • Cold start vs background vs foreground
  • Auth not ready yet → store the deep link and replay after login

⚠️ NavigationPath must be ready

  • Don’t append before NavigationStack exists

⚠️ Parsing URLs cleanly

  • Prefer URLComponents over string parsing

Mental model

Deep linking = URL → Intent → Route → View

SwiftUI just gives you the hook — you decide how smart the routing is.
