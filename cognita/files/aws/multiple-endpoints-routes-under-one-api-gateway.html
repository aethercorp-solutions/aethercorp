multiple endpoints (routes) under one API Gateway

You create multiple endpoints (routes) under one API Gateway by defining multiple routes/resources that all live inside the same gateway and can point to the same or different backends (Lambda, HTTP service, etc.).

Below is the practical, no-confusion explanation, with both HTTP API and REST API patterns.

🧠 Core Concept

One API Gateway

→ many routes (endpoints)

→ each route maps to a backend

Example:


https://api.example.com/users
https://api.example.com/users/{id}
https://api.example.com/orders
https://api.example.com/orders/{id}

All of these live under ONE API Gateway.

✅ OPTION 1: HTTP API (Recommended)

1️⃣ Create Multiple Routes (Endpoints)

In API Gateway → Your HTTP API → Routes

Add routes like:

Method Path
GET /users
POST /users
GET /users/{id}
GET /orders
POST /orders

Each route can point to:

  • Same Lambda (monolith-style)
  • Different Lambdas (microservices)

2️⃣ Single Lambda Handling Multiple Endpoints (Most Common)

Best for small–medium APIs


export const handler = async (event) => {
    const { routeKey, pathParameters, body } = event;
    
    switch (routeKey) {
        case "GET /users":
        return ok(await getUsers());
    
        case "POST /users":
        return ok(await createUser(JSON.parse(body)));
    
        case "GET /users/{id}":
        return ok(await getUser(pathParameters.id));
    
        default:
        return notFound();
    }
};
    
const ok = (data) => ({
    statusCode: 200,
    body: JSON.stringify(data),
});
    
const notFound = () => ({
    statusCode: 404,
    body: "Not Found",
});

📌 routeKey is the key for HTTP APIs

(METHOD + path)

3️⃣ Different Lambda per Endpoint (Microservice Style)

Example:


GET  /users     → users-get-lambda
POST /users     → users-post-lambda
GET  /orders    → orders-get-lambda

✅ Cleaner separation

❌ More Lambdas to manage

✅ OPTION 2: REST API (Older but Powerful)

Resource Structure


/
├── users
│   ├── GET
│   ├── POST
│   └── {id}
│       └── GET
└── orders
    └── POST

Each method is a separate endpoint.

🔁 Path Parameters & Query Strings

Path Parameter


GET /users/123
            

event.pathParameters.id

Query String


GET /users?limit=10
            

event.queryStringParameters.limit

🔐 Securing Multiple Endpoints

You can apply different auth per route:

HTTP API → Routes → Authorization

Route Auth
GET /public None
POST /users JWT
GET /admin IAM

📁 Recommended Folder Structure (Lambda)


src/
 ├── handlers/
 │   ├── users.js
 │   ├── orders.js
 │   └── auth.js
 └── index.js
            

import * as users from "./handlers/users.js";
import * as orders from "./handlers/orders.js";

export const handler = async (event) => {
  switch (event.routeKey) {
    case "GET /users":
      return users.getAll();
    case "POST /orders":
      return orders.create();
  }
};

⚡ Best Practices

✅ Use one API Gateway per app

✅ Use HTTP API unless you need REST features

✅ Group endpoints logically

✅ Use one Lambda first, split later if needed

✅ Version your API (/v1/users)

✔ One API Gateway can have unlimited endpoints

✔ Add multiple routes (method + path)

✔ Map routes to one or many Lambdas

✔ Handle routing inside Lambda or via API Gateway
