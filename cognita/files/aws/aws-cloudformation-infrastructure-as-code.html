CloudFormation

CloudFormation is AWS’s “infrastructure as code” service.

In simple terms:

👉 You describe your cloud resources in a file, and AWS builds them for you automatically.

What it does

Instead of clicking around the AWS console to create things like:

  • EC2 instances
  • Lambda functions
  • API Gateway
  • S3 buckets
  • IAM roles

You write a template (YAML or JSON) that says:

“I want this resource, with these settings, connected to those resources.”

AWS CloudFormation then:

  • Creates everything in the right order
  • Updates resources safely
  • Deletes them cleanly if needed

Why people use it

  • Repeatable – same setup every time (no “works on my account” issues)
  • Version-controlled – templates live in Git
  • Automated – great for CI/CD
  • Rollback support – if something fails, AWS can undo changes

Key concepts

  • Template – the YAML/JSON file you write
  • Stack – a running instance of that template
  • Change set – preview of what will change before applying
  • Resources – the AWS services you define
  • Parameters & Outputs – inputs and exported values

Tiny example (YAML)


Resources:
  MyBucket:
    Type: AWS::S3::Bucket

That’s it — deploy this, and AWS creates an S3 bucket.

How it compares (you’ll appreciate this)

  • CloudFormation → native AWS, very powerful, verbose
  • SAM → CloudFormation + shortcuts for serverless (Lambda/API)
  • Serverless Framework → abstraction over CloudFormation
  • Terraform → multi-cloud alternative

Show how Serverless Framework maps to CloudFormation

Perfect question — this is where the magic curtain gets pulled back 🪄

Since you’re already using Serverless Framework, here’s exactly how it maps to CloudFormation.

Big picture

Serverless Framework does NOT replace CloudFormation.

It GENERATES CloudFormation for you, then deploys it.

Think of it like this:


serverless.yml
      ↓
Serverless Framework
      ↓
CloudFormation template (auto-generated)
      ↓
AWS creates/updates resources

When you run:


serverless deploy

Serverless:

  1. Compiles serverless.yml
  2. Generates a CloudFormation template
  3. Creates or updates a CloudFormation stack
  4. Uploads artifacts to S3
  5. Calls CloudFormation to deploy

Where the CloudFormation lives

After deploy, check:


.serverless/cloudformation-template-update-stack.json

That file is the CloudFormation template AWS actually used.

1️⃣ Functions → Lambda + IAM + Logs

serverless.yml


functions:
  hello:
    handler: handler.hello

CloudFormation generated


HelloLambdaFunction:
  Type: AWS::Lambda::Function
  Properties:
    Handler: handler.hello
    Runtime: nodejs18.x
    Role:
      Fn::GetAtt:
        - IamRoleLambdaExecution
        - Arn

Also auto-created:

  • CloudWatch Log Group
  • Execution IAM Role
  • Permissions policies

👉 One function = multiple CloudFormation resources

2️⃣ Events → Event Source Mappings

HTTP API example


functions:
  hello:
    handler: handler.hello
    events:
      - http:
          path: hello
          method: get

CloudFormation generated

  • AWS::ApiGateway::RestApi
  • AWS::ApiGateway::Method
  • AWS::ApiGateway::Resource
  • AWS::Lambda::Permission

👉 Serverless wires API Gateway ↔ Lambda automatically.

3️⃣ Environment variables → Lambda properties

serverless.yml


provider:
  environment:
    TABLE_NAME: Users

CloudFormation generated


Environment:
  Variables:
    TABLE_NAME: Users

4️⃣ IAM statements → IAM Role policies

serverless.yml


provider:
  iam:
    role:
      statements:
        - Effect: Allow
          Action: dynamodb:GetItem
          Resource: "*"

CloudFormation generated


IamRoleLambdaExecution:
  Type: AWS::IAM::Role
  Properties:
    Policies:
      - PolicyDocument:
          Statement:
            - Effect: Allow
              Action: dynamodb:GetItem
              Resource: "*"

5️⃣ Custom resources → Raw CloudFormation

This is where you drop down to pure CloudFormation.

serverless.yml


resources:
  Resources:
    MyTable:
      Type: AWS::DynamoDB::Table
      Properties:
        TableName: Users

Serverless:

  • Injects this directly into the CloudFormation template
  • No abstraction — 1:1 mapping

👉 This is how you handle anything Serverless doesn’t support

6️⃣ Stages → CloudFormation stacks


serverless deploy --stage prod

Creates a stack like:


my-service-prod

Each stage = separate CloudFormation stack

  • Separate resources
  • Separate env vars
  • Separate IAM roles

7️⃣ Rollbacks & failures

Because CloudFormation is underneath:

  • Failed deploy → automatic rollback
  • Partial resources cleaned up
  • Stack stays consistent

Serverless just reports CloudFormation’s status.

Mental model (important)

Serverless CloudFormation
serverless.yml Template
Service Stack
Function Lambda + Role + Logs
Event API / Trigger resources
Stage Stack name suffix
resources: Raw CF

When to think “CloudFormation” instead of “Serverless”

You should drop to CF when:

  • You need fine-grained IAM
  • You need VPC, custom networking
  • You need unsupported AWS services
  • You want cross-stack exports

Serverless is convenience — CloudFormation is the engine.

Pro tip (advanced, your level)

You can reference Serverless-generated resources inside CloudFormation:


Resources:
  MyPolicy:
    Type: AWS::IAM::Policy
    Properties:
      Roles:
        - Ref: IamRoleLambdaExecution

That role is created by Serverless.
