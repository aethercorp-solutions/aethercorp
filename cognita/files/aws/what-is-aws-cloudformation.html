AWS CloudFormation

AWS CloudFormation is AWS’s Infrastructure as Code (IaC) service. It lets you define, provision, and manage AWS resources using code instead of clicking around the AWS Console.

Think of it as “deploy your entire AWS infrastructure from a template.”

Core Idea (Simple Explanation)

Instead of manually creating:

  • EC2 instances
  • Lambda functions
  • API Gateway
  • VPCs
  • IAM roles
  • Databases

You write a CloudFormation template (YAML or JSON), and AWS:

  1. Reads the template
  2. Creates all resources in the correct order
  3. Manages updates, rollbacks, and deletions safely

Key Concepts

1. Template

A template is a file (YAML or JSON) that describes:

  • What resources to create
  • How they are configured
  • How they relate to each other

Example (very simple):


Resources:
  MyBucket:
    Type: AWS::S3::Bucket

This creates an S3 bucket.

2. Stack

A stack is a running instance of a template.

  • Template = blueprint
  • Stack = actual deployed infrastructure

You can have:

  • dev-stack
  • staging-stack
  • prod-stack

All from the same template.

3. Resources

Resources are AWS services you define:

  • AWS::Lambda::Function
  • AWS::EC2::Instance
  • AWS::DynamoDB::Table
  • AWS::IAM::Role

CloudFormation figures out dependencies automatically.

4. Parameters

Parameters let you customize stacks without changing the template.


Parameters:
  Environment:
    Type: String
    Default: dev

Useful for:

  • Environment names
  • Instance sizes
  • Region-specific values

5. Outputs

Outputs expose values after deployment.


Outputs:
  ApiUrl:
    Value: !GetAtt ApiGateway.RestApiId

Useful for:

  • Sharing values with other stacks
  • Showing important info after deploy

What CloudFormation Does for You

✅ Automatic Dependency Handling

If a Lambda needs an IAM role:

  • CloudFormation creates the role first
  • Then attaches it to the Lambda

No manual ordering required.

✅ Safe Updates (Change Sets)

Before updating:

  • CloudFormation shows what will change
  • You can review before applying

If something fails:

  • Automatic rollback to last working state

✅ Repeatable & Consistent Deployments

You get:

  • Same infrastructure every time
  • No “works on my account” problems

Perfect for teams and CI/CD.

✅ Stack Deletion

Delete the stack → AWS deletes all resources created by it (no forgotten EC2s or leaked resources).

CloudFormation vs AWS Serverless Framework (Important for You)

Since you work heavily with AWS Lambda + API Gateway:

Feature CloudFormation Serverless Framework
Level Low-level High-level
Verbosity Very verbose Very concise
Control Full AWS control Opinionated
Uses CF under the hood ❌ ✅
Best for Core infra App-focused serverless

👉 Serverless Framework actually generates CloudFormation templates behind the scenes.

When to Use CloudFormation

Use CloudFormation when:

  • You need full control over AWS resources
  • You manage VPCs, IAM, networking
  • You build shared infra (base stacks)
  • You want AWS-native IaC (no third-party tools)

When NOT to Use CloudFormation Directly

You might prefer:

  • Serverless Framework → serverless apps
  • AWS CDK → write infra using TypeScript/Python
  • Terraform → multi-cloud setups

(These all eventually generate CloudFormation or API calls.)

Simple Mental Model

CloudFormation = AWS’s native way to describe infrastructure as code, deploy it safely, and manage its lifecycle.
